Chapter 00

Setup & Tooling

Before writing any Go code, you need to install the Go toolchain. Think of this like setting up a workshop before building furniture — you need the right tools in place first. The great news is that Go ships as a single, all-in-one toolkit. When you install Go, you get everything: the compiler (which turns your code into a runnable program), a build tool, a test runner, a dependency manager, a code formatter, and a documentation generator. There is no separate tool to install for each job. One command — go — does it all.

This is a genuinely unusual design decision. Most languages require you to separately install a build tool, a test framework, a package manager, and a formatter. Go bundles all of these together because its creators believe that tooling consistency across all projects is more important than letting each project choose its own tools. The result: every Go project you will ever encounter uses the exact same commands.

Installation

Installing Go is straightforward on every operating system. On macOS, Homebrew is the quickest path. On Linux, you download the official tarball from go.dev. On Windows, use the MSI installer. After installation, run go version to confirm everything worked — you want Go 1.22 or later for the newest features.

Think of it like installing a Swiss Army knife. Once it is on your system, you have access to every blade (compiler, tester, formatter) without hunting for separate tools. Let’s get it installed.

# ── macOS: install Go using Homebrew (a popular macOS package manager) ──
brew install go

# ── Linux: download the official tarball from go.dev ──
# wget fetches the file; tar extracts it into /usr/local
wget https://go.dev/dl/go1.22.5.linux-amd64.tar.gz
sudo tar -C /usr/local -xzf go1.22.5.linux-amd64.tar.gz

# ── Verify the installation succeeded ──
# This prints the installed Go version (you want 1.22 or higher)
go version

# ── Check important environment variables ──
# GOPATH: where Go stores downloaded packages and compiled binaries
# Think of it as Go's "downloads folder" for libraries
go env GOPATH

# GOROOT: where Go itself is installed on your machine
# You rarely need to change this; it's set automatically
go env GOROOT

The GOPATH is where Go stores downloaded libraries and compiled binaries. When you download someone else’s code to use in your project, it goes here. Go fetches code directly from version control hosts like GitHub — there is no separate “package registry” you need to sign up for. If code is public on GitHub, anyone can use it.

Project Initialization

Every Go project begins with a single command: go mod init. This creates a file called go.mod, which is the “identity card” of your project. It declares two things: your project’s unique name (called the module path) and the list of external libraries your project depends on.

Think of go.mod like a recipe card at the top of your cookbook. It says “This is the Agent System cookbook, and we need flour (an HTTP library), sugar (a JSON parser), and eggs (a database driver) to make it work.” Anyone who picks up your project can run one command and Go will fetch all the ingredients automatically.

The module path is typically a URL-like string (like github.com/yourname/project). This is not just a name — it is how other people import your code. If you share your project on GitHub, others can reference it by this exact path.

# ── Create a new project directory and enter it ──
# mkdir creates the folder; cd moves into it
mkdir my-agent-system && cd my-agent-system

# ── Initialize a new Go module ──
# This creates go.mod, which tracks your project's identity and dependencies
# The path after "init" is your module's unique name
go mod init github.com/yourname/my-agent-system

# ── What go.mod looks like after initialization ──
# module github.com/yourname/my-agent-system
# go 1.22
# (dependencies will appear here as you add them)

When you add external libraries, Go automatically updates go.mod and creates a companion file called go.sum. The go.sum file is a “trust ledger” — it records a cryptographic fingerprint of every dependency so that Go can verify no one tampered with the code you downloaded. Run go mod tidy periodically to clean up any libraries you stopped using.

IDE Setup

You have two excellent editor options. VS Code with the official Go extension provides autocompletion, inline error detection, debugging, test running, and code navigation. It is free, lightweight, and very popular. GoLand by JetBrains is a full-featured IDE with deeper refactoring tools, an integrated debugger, and advanced code analysis. It costs money but is extremely polished.

Both editors automatically run a tool called gofmt when you save a file. This tool reformats your code to match the one true Go style. This is not optional — it is a core part of Go culture. Every Go project in the world looks the same: tabs for indentation, opening braces on the same line as the statement, imports sorted alphabetically. There are zero debates about code style in Go because the formatter decides for you. This might feel restrictive, but it means you never waste time arguing about formatting in code reviews.

Why one style matters
Imagine if every restaurant wrote their menus in a different language and layout. You’d waste time decoding each menu before you could order. Go’s single format means every Go project “reads the same way.” You can open any Go project on GitHub and immediately understand its structure because the formatting is always identical.

Essential Go Commands

Here are the commands you will use every day when working with Go. Each one is fast — typically finishing in under a few seconds even for large projects. This speed changes how you work: instead of running tests only before you commit your code, you run them constantly as you write, catching mistakes in seconds rather than minutes.

Think of these commands like kitchen tools. go run is your taste-test spoon — quick feedback on what you just wrote. go build is the oven — it produces the final dish (a compiled program). go test is the food safety inspector — it checks that everything meets standards.

# ── go run: compile and execute your program in one step ──
# Great for quick testing during development
go run main.go

# ── go build: compile your code into a standalone binary (executable file) ──
# The -o flag names the output file "myapp"
# The "." means "build the current directory"
go build -o myapp .

# ── go test: run all tests in all sub-packages ──
# The ./... pattern means "this directory and everything inside it"
go test ./...

# ── go test -race: run tests with Go's race condition detector ──
# This finds bugs where two parts of your code unsafely share data
go test -race ./...

# ── go mod tidy: clean up your dependency list ──
# Removes libraries you stopped using, adds any missing ones
go mod tidy

# ── go vet: run static analysis to find common bugs ──
# Catches things the compiler misses, like unreachable code
go vet ./...

# ── gofmt: auto-format all Go files to the standard style ──
# The -w flag writes changes back to the files (instead of just printing)
gofmt -w .

# ── go doc: read documentation for any function or package ──
# Useful for quickly looking up how a standard library function works
go doc fmt.Println

Cross-Compilation

One of Go’s most powerful features is cross-compilation: the ability to build a program for a different operating system right from your own machine. Imagine you are cooking in a kitchen in New York, but the meal needs to be served in a kitchen in Tokyo that uses different plates and utensils. With Go, you can “plate the meal for Tokyo” right from your New York kitchen.

This is incredibly useful in practice. Most developers write code on macOS or Windows, but their programs run on Linux servers in the cloud. With Go, you set two environment variables (GOOS for the target operating system and GOARCH for the target processor architecture) and build. The result is a single, self-contained binary file that runs on the target machine with zero dependencies — no runtime to install, no framework to configure, nothing. You just copy the file and run it.

# ── Build a program that runs on Linux, from your Mac ──
# GOOS=linux means "target operating system is Linux"
# GOARCH=amd64 means "target processor is 64-bit Intel/AMD"
GOOS=linux GOARCH=amd64 go build -o myapp .

# ── Build for ARM processors (used in AWS Graviton, Raspberry Pi, etc.) ──
# arm64 is the processor type used in newer, more power-efficient servers
GOOS=linux GOARCH=arm64 go build -o myapp .

# ── Create a minimal Docker container ──
# "scratch" means an empty base image with literally nothing in it
# Your Go binary is the ONLY file in the entire container
# FROM scratch
# COPY myapp /myapp
# CMD ["/myapp"]
# Result: a ~15MB Docker image (most language runtimes are 200-500MB)
Why small binaries matter
A typical Go program compiled into a Docker container is 10-20MB. Programs in most other languages need a runtime environment that brings the image to 200-500MB. Smaller images mean faster deployments, less storage cost, fewer security vulnerabilities (less code = fewer things that can be exploited), and faster scaling when your system needs to spin up new copies under heavy load.
Chapter 01

Why Go?

Go was created at Google in 2007 by three legendary computer scientists: Robert Griesemer, Rob Pike, and Ken Thompson (who also co-created Unix and the C programming language). They built Go because they were frustrated. Google’s massive codebases were slow to compile, hard to read, and buried under layers of abstraction. They wanted a language that was simple to read, fast to compile, and great at handling many things at once (concurrency).

Think of it this way: imagine you run a restaurant. Some kitchens are designed with every possible gadget — a dozen specialized tools, complex equipment with instruction manuals thicker than cookbooks, wiring that only a certified electrician understands. Go is the kitchen designed by chefs who have cooked in those over-engineered kitchens for twenty years and said: “Give me sharp knives, a good stove, and clean counter space. I can cook anything with those.”

Go deliberately leaves out features that its designers considered sources of accidental complexity — problems that come from the tools themselves rather than the problem you are trying to solve. There is no class inheritance (where types form deep family trees). There is no exception handling (where errors jump across the program in hard-to-follow ways). There is no method overloading (where the same name does different things based on inputs). What you see in the code is what runs — there is no hidden magic.

This will feel limiting at first if you come from a language with many features. Then it will feel liberating. A Go codebase reads top-to-bottom like a story. You can open any file and understand what it does without tracing through layers of abstraction, hidden configuration, or “magic” behavior that happens behind the scenes.

Why Go for Agentic Systems Specifically

Agentic AI systems — where autonomous software agents make decisions, call tools, and collaborate with each other — have very specific technical requirements. Let’s use our restaurant analogy to understand why Go excels here.

An agentic system is like a busy restaurant kitchen during dinner rush. You have multiple chefs (agents) working simultaneously. Each chef needs to call out orders to the prep station (LLM API calls), check the pantry (database lookups), and coordinate with other chefs (inter-agent communication). Some tasks take a long time (waiting for a slow oven), but the kitchen cannot stop while waiting — other chefs need to keep cooking.

Lightweight Concurrency

Go can run hundreds of thousands of goroutines (lightweight tasks) on a single machine. Each one uses only ~2KB of memory. This means your orchestrator can manage dozens of agents, each making multiple LLM calls, all at the same time — without running out of resources.

Fast Startup & Low Memory

A Go program starts in milliseconds and uses a fraction of the memory that other runtimes need. When your system needs to scale up under load (more agents, more requests), new instances spin up nearly instantly.

Simple Deployment

One binary file. No runtime. No framework. Copy it to a server and run it. This dramatically simplifies deploying agent systems, especially in containerized environments like Kubernetes.

Explicit Error Handling

Agent systems deal with constant failures: LLM timeouts, rate limits, invalid tool outputs. Go’s explicit error handling forces you to think about what happens when things go wrong at every step, making your agents more resilient.

The Go Philosophy: Simplicity Is a Feature

Go’s designers made bold decisions about what to leave out. Understanding these choices helps you think “the Go way” from the start:

  • No inheritance — Go uses composition (combining small pieces) instead of inheritance (building deep family trees). Think of it like LEGO bricks vs. Russian nesting dolls: bricks can be combined in infinite ways, while nesting dolls only fit inside each other.
  • No exceptions — Errors are returned as ordinary values alongside results. This makes error paths visible in the code instead of hidden in a separate control flow.
  • No generics (until recently) — Go added generics in version 1.18 (2022), deliberately waiting 13 years until the design was right. They are simpler than generics in most languages.
  • No implicit behavior — If something happens, you can see it in the code. No annotations that secretly modify behavior, no dependency injection frameworks that wire things together behind the scenes.
  • No unused variables or imports — If you declare a variable and do not use it, Go refuses to compile. This keeps codebases clean.
The “boring technology” philosophy
Go embraces being “boring.” It is not exciting or cutting-edge in its features. It does not have the most elegant syntax or the most powerful type system. But boring is exactly what you want when building systems that must run reliably in production 24/7. Airlines do not fly experimental aircraft on commercial routes. Go is the Boeing 737 of programming languages: well-understood, proven, and reliable.

Core Syntax & Types

Hello World & the Package System

Every Go program is organized into packages. A package is simply a folder containing one or more .go files that all declare the same package name at the top. Think of packages like departments in a company: the “accounting” department has its own files and responsibilities, the “engineering” department has its own, and they communicate through well-defined interfaces.

There is one special package name: main. When Go sees a package called main with a function called main(), it knows this is the entry point of the program — the place where execution starts, like the front door of a building. Every executable Go program must have exactly one main package with one main() function.

The import statement brings in other packages so you can use their code. Go’s standard library is extensive — fmt (short for “format”) handles printing and formatting text, net/http provides a full HTTP web server, encoding/json handles converting data to and from JSON format. One strict rule: if you import a package and do not use it, Go refuses to compile your code. This sounds harsh, but it keeps every project’s code clean — no dead imports cluttering the top of files.

// Every Go file starts with a package declaration.
// "package main" tells Go: "This is an executable program."
package main

// Import the "fmt" package from Go's standard library.
// "fmt" stands for "format" — it provides functions for printing text.
import "fmt"

// The main() function is where your program starts executing.
// When you run "go run main.go", Go calls this function first.
func main() {
    // fmt.Println prints a line of text to the terminal and adds a newline.
    // This is Go's version of "print" found in every language.
    fmt.Println("Hello from Go!")
}
Analogy: packages are like departments
A Go package is like a department in a company. The fmt package is the “communications department” — it knows how to format and print messages. The net/http package is the “receptionist” — it handles incoming web requests. Your main package is the “CEO’s office” — it is where the program starts and decides what to do first.

Variable Declaration: var vs :=

Variables are named containers that hold values — like labeled boxes in a warehouse. Go gives you two ways to create variables, and understanding when to use each is important.

The first way is the var keyword. This is the explicit, formal way to declare a variable. You can specify the type (what kind of data the box holds) and optionally give it an initial value. If you do not give it a value, Go fills it with a sensible default (more on this in “Zero Values” below).

The second way is the short declaration operator :=. This is the quick, informal way. Go looks at the value on the right side and figures out the type automatically. You will use := for the vast majority of variables inside functions. The key restriction: := can only be used inside functions, not at the file level. At the file level (outside any function), you must use var.

Think of var as filling out a formal shipping label (“Contents: Books, Weight: 5kg, Destination: Library”) and := as tossing something in a box and writing on it with a marker (“Books → Library”). Both get the job done; the short form is what you use day-to-day.

// Every Go file starts with a package declaration.
package main

// Import the fmt package for printing.
import "fmt"

// ── Package-level variable: must use "var" (cannot use := outside a function) ──
// This variable is accessible by any function in this file.
var appName string = "Agent System"

func main() {
    // ── Explicit declaration with type ──
    // "var vendorCount int = 42" means: create a box labeled "vendorCount",
    // specify it holds integers, and put 42 inside.
    var vendorCount int = 42

    // ── Type inferred from value ──
    // Go sees 80.5 and automatically knows this is a float64 (decimal number).
    var riskThreshold = 80.5

    // ── Short declaration with := (most common form inside functions) ──
    // Go infers the type from the right-hand side automatically.
    agentName := "risk-assessor"   // Go infers: this is a string
    isActive := true               // Go infers: this is a bool (true/false)
    maxRetries := 3                // Go infers: this is an int (whole number)

    // fmt.Printf prints formatted text. %s = string, %d = integer, %.1f = decimal with 1 digit.
    // \n means "start a new line".
    fmt.Printf("App: %s, Vendors: %d, Threshold: %.1f\n",
        appName, vendorCount, riskThreshold)

    // %v means "print the value in its default format" — works for any type.
    fmt.Printf("Agent: %s, Active: %v, Retries: %d\n",
        agentName, isActive, maxRetries)
}

Zero Values — Go’s Safety Net

Here is one of Go’s most thoughtful design decisions. In many languages, if you create a variable without giving it a value, you get either an error or unpredictable garbage data. Go takes a different approach: every type has a “zero value” — a sensible, safe default that the variable is automatically set to if you do not initialize it.

Think of it like a hotel room. When you check in, the room is not empty and it is not full of the previous guest’s belongings. It is in a known, clean state: bed made, lights off, thermostat at a default temperature. Go does the same thing for every variable. An uninitialized number is 0. An uninitialized string is "" (an empty string, not missing or undefined). An uninitialized boolean is false. An uninitialized struct has all its fields set to their zero values.

This design eliminates an entire category of bugs that plague other languages. You never access an uninitialized variable and get random memory contents. The only type where the zero value can cause a crash is a pointer (which defaults to nil, meaning “points to nothing”), and even then you must explicitly try to follow the pointer to crash.

func main() {
    // ── Declaring variables WITHOUT assigning values ──
    // Go automatically fills each one with its zero value.

    var i int        // zero value: 0 (not garbage, not undefined — exactly 0)
    var f float64    // zero value: 0.0
    var s string     // zero value: "" (an empty string, NOT missing/null)
    var b bool       // zero value: false
    var p *int       // zero value: nil (a pointer that points to nothing)
    var sl []int     // zero value: nil slice (empty, but len() and append() still work safely)
    var m map[string]int  // zero value: nil map (reading returns zero values; writing panics)

    // Print them all — no crashes, no surprises.
    fmt.Println(i, f, s, b, p)  // Output: 0 0  false <nil>

    // Even nil slices support len() — it returns 0.
    fmt.Println(len(sl))         // Output: 0

    // You can safely append to a nil slice. Go allocates memory automatically.
    sl = append(sl, 1, 2, 3)   // sl is now [1, 2, 3]

    // ── Struct zero values ──
    // Every field gets its own zero value. The struct is immediately usable.
    var v Vendor
    fmt.Println(v.Name)       // "" (empty string, not nil, not an error)
    fmt.Println(v.RiskScore)  // 0
    fmt.Println(v.Active)     // false
}
Design around zero values
The best Go code is written so that the zero value of a struct is already useful. For example, Go’s built-in sync.Mutex (a lock for shared data) works correctly as soon as you declare it — no initialization needed. When designing your own types, ask: “Is the zero value of this struct a valid, useful starting state?” If yes, your users never need a special constructor function.

Constants

Constants are values that never change once you define them. Think of them as permanent labels etched in metal, as opposed to variables which are labels written in pencil. You use constants for values that are fixed throughout the life of your program: the maximum number of retries, an API version string, a mathematical constant like pi.

Go constants are computed at compile time, meaning the Go compiler calculates them before your program ever runs. This makes them extremely efficient and completely safe from accidental modification.

Go also has a unique feature called iota — an auto-incrementing counter for generating sequences of related constants. Think of it like a ticket machine at a deli counter: the first customer gets ticket 0, the next gets 1, then 2, and so on. This is Go’s lightweight way of creating related constants (like risk levels: low=0, medium=1, high=2, critical=3) without manually typing each number.

// ── Simple constants: values that never change ──
// "const" means this value is permanent and cannot be modified.
const maxRetries = 3           // max number of times to retry a failed operation
const apiVersion = "v1"        // the version of our API (a fixed string)

// ── Related constants with iota (auto-incrementing) ──
// First, we define a custom type "RiskLevel" based on int.
// This makes our code self-documenting: a function that takes a RiskLevel
// is clearer than one that takes a plain int.
type RiskLevel int

// Inside a const block, "iota" starts at 0 and increments by 1 for each line.
// Think of it like a deli ticket machine: 0, 1, 2, 3...
const (
    RiskLow      RiskLevel = iota  // 0 — iota starts at 0 on the first line
    RiskMedium                       // 1 — iota automatically increments
    RiskHigh                         // 2
    RiskCritical                     // 3
)

// ── iota with expressions: create bit flags (powers of 2) ──
// "1 << iota" means "shift 1 left by iota positions" — producing 1, 2, 4, 8.
// Bit flags let you combine permissions: PermRead | PermWrite = 3 (both read and write).
const (
    PermRead    = 1 << iota  // 1 (binary: 0001)
    PermWrite                  // 2 (binary: 0010)
    PermExecute                // 4 (binary: 0100)
    PermAdmin                  // 8 (binary: 1000)
)

Basic Types

Go has a small, well-defined set of built-in data types. Unlike some languages that have dozens of numeric types and complex hierarchies, Go keeps things simple. There are integers (whole numbers), floats (decimal numbers), strings (text), booleans (true/false), and a few special types.

One important rule in Go: type conversions are always explicit. If you have an integer and need a decimal number, you must write the conversion yourself. Go never silently converts between types because silent conversions can hide precision-loss bugs. This might feel tedious at first, but it means you always know exactly what type your data is.

Think of it like currency exchange. If you have US dollars and need Euros, you do not expect the cashier to silently convert without telling you the exchange rate. Go makes you write the “exchange” explicitly so there are no surprises.

// ── Integer types ──
var age int       = 42         // "int" is platform-sized (64-bit on modern computers)
var score int64   = 9999999   // explicitly 64-bit integer (for when you need to be specific)
var b byte        = 255       // "byte" is an alias for uint8 (0-255, used for raw data)

// ── Floating-point (decimal numbers) ──
var temp float64  = 98.6      // 64-bit decimal (Go's default when you write a decimal literal)

// ── Rune: a single Unicode character ──
// "rune" is an alias for int32. It holds one Unicode code point.
// Unicode is the standard that defines every character in every language.
var r rune        = 'A'       // Single quotes for a single character (rune)

// ── Strings: immutable (unchangeable) sequences of UTF-8 bytes ──
// Double quotes for regular strings (with escape sequences like \n for newline).
name := "Acme Corp"

// Backticks for "raw" strings — no escape sequences processed.
// Everything between backticks is literal text, including newlines.
raw := `This is a raw string literal.
It can span multiple lines.
Backslash-n is literal \n, not a newline character.`

// ── Boolean ──
isCompliant := true  // Only two values: true or false

// ── Explicit type conversions (Go NEVER converts silently) ──
var x int = 42
var y float64 = float64(x)  // Must explicitly convert int to float64
var z int64 = int64(x)      // Even int to int64 requires explicit conversion
No silent type conversions
Writing var f float64 = someInt is a compile error in Go. You must write var f float64 = float64(someInt). This explicitness prevents subtle bugs where precision is silently lost (e.g., converting a large integer to a smaller type that cannot hold the value).

Strings and Runes: Text in Go

Strings in Go are immutable sequences of bytes, encoded in UTF-8. This is important to understand because it means the “length” of a string is the number of bytes, not the number of characters. The English letter “A” takes 1 byte, but the Chinese character “世” takes 3 bytes. So the string "Hello, 世界" has 9 visible characters but is 13 bytes long.

Think of a string like a necklace of beads. Each bead is a byte. Some characters (like English letters) are represented by a single bead, while others (like Chinese, Japanese, or emoji characters) need 2, 3, or even 4 beads. When Go says a string has len(s) == 13, it is counting beads, not pearls.

When you need to work with individual characters (called runes in Go, which is an alias for int32), you use range to iterate over the string. The range loop is smart enough to decode the UTF-8 bytes and give you one character at a time. Direct indexing like s[0] gives you a raw byte, not a character — so use range for character work.

// A string with both ASCII and Unicode characters.
greeting := "Hello, 世界"

// len() counts BYTES, not characters!
// "Hello, " = 7 bytes, "世" = 3 bytes, "界" = 3 bytes → total 13 bytes.
fmt.Println(len(greeting))          // Output: 13

// ── range iterates over RUNES (characters), not bytes ──
// Each iteration gives you the byte index and the rune (character) at that position.
for i, r := range greeting {
    // %d = integer, %c = character, utf8.RuneLen(r) = how many bytes this rune takes
    fmt.Printf("byte_index=%d character=%c bytes_used=%d\n", i, r, utf8.RuneLen(r))
}
// byte_index=0  character=H bytes_used=1
// byte_index=7  character=世 bytes_used=3
// byte_index=10 character=界 bytes_used=3

// ── Convert string to a slice of runes for character-level operations ──
// Now len() counts characters, not bytes.
runes := []rune(greeting)
fmt.Println(len(runes))              // Output: 9 (the actual number of characters)

// ── Building strings efficiently with strings.Builder ──
// Strings are immutable in Go, so concatenating with + creates new strings each time.
// strings.Builder is like a notepad — you write to it, then get the final string once.
var builder strings.Builder
for _, vendor := range vendors {
    builder.WriteString(vendor.Name)  // Append the vendor's name
    builder.WriteString(", ")          // Append a comma separator
}
result := builder.String()             // Get the final combined string

Collections: Slices

Go has two main collection types built into the language: slices (ordered lists that can grow and shrink) and maps (key-value lookups, like a dictionary). Unlike some languages that have extensive collection hierarchies with linked lists, tree maps, hash sets, etc., Go keeps it simple. Slices and maps handle the vast majority of real-world needs.

A slice is a dynamically-sized, ordered collection of elements of the same type. Think of it like a row of chairs in a theater. You can add more chairs to the end (append), count how many chairs there are (len), and look at individual chairs by their position number (indexing). Under the hood, a slice is backed by an array (a fixed block of memory), and Go automatically allocates a bigger array when you run out of space.

Slices have three properties: a pointer (where the underlying data lives in memory), a length (how many elements it currently holds), and a capacity (how many elements it can hold before needing to allocate more space). When you append beyond the capacity, Go allocates a new, larger array, copies the existing data over, and updates the slice to point to the new array. This happens automatically — you just call append and Go handles the rest.

// ── Creating slices ──

// Slice literal: declare and fill at the same time.
// The []string{} syntax creates a slice of strings with initial values.
names := []string{"Alice", "Bob", "Charlie"}

// make() creates a slice with a specified length and capacity.
// This pre-allocates space for 10 ints, but starts with 0 elements.
// Pre-allocating is a performance optimization when you know the approximate size.
scores := make([]int, 0, 10)   // length=0, capacity=10

// ── Appending elements ──
// append() returns a NEW slice header (because the underlying array might move).
// ALWAYS reassign: scores = append(scores, ...) — never just append(scores, ...).
scores = append(scores, 85, 92, 78)  // scores is now [85, 92, 78]

// ── Slicing a slice (extracting a portion) ──
// Syntax: slice[start:end] — includes start, excludes end.
// Think of it like cutting a section out of a row of seats.
firstTwo := names[:2]   // ["Alice", "Bob"] — from beginning to index 2 (exclusive)
lastTwo := names[1:]    // ["Bob", "Charlie"] — from index 1 to end
middle := names[1:2]   // ["Bob"] — from index 1 to index 2 (exclusive)

// ── Length and capacity ──
fmt.Println(len(names))  // 3 — how many elements it currently holds
fmt.Println(cap(names))  // 3 — how many elements it can hold before growing
Slice gotcha: shared memory
When you slice a slice (e.g., b := a[1:3]), both a and b point to the same underlying memory. Changing b[0] also changes a[1] because they share the same data. If you need an independent copy, use the built-in copy() function: b := make([]int, 2); copy(b, a[1:3]). This is a common source of bugs for beginners.

Collections: Maps

A map is an unordered collection of key-value pairs — like a real-world dictionary where you look up a word (the key) to find its definition (the value). Maps are incredibly useful for lookups: given a vendor’s name, find their risk score. Given an agent’s ID, find their configuration.

Maps must be initialized before you can write to them. A map created with var alone is nil (empty and uninitialized), and writing to a nil map causes a crash. Always create maps using a map literal (map[KeyType]ValueType{...}) or make(map[KeyType]ValueType).

Go maps have a powerful feature called the “comma-ok” idiom. When you look up a key, you can ask for two values: the value itself and a boolean that tells you whether the key existed. This is important because looking up a missing key does not crash — it silently returns the zero value of the value type. Without the comma-ok check, you cannot distinguish between “key exists with value 0” and “key does not exist at all.”

// ── Creating a map with initial values ──
// map[KeyType]ValueType{...} creates a map and populates it.
// Keys are strings (vendor names), values are ints (risk scores).
vendorScores := map[string]int{
    "acme":    85,   // "acme" vendor has risk score 85
    "globex":  72,   // "globex" vendor has risk score 72
    "initech": 91,   // "initech" vendor has risk score 91
}

// ── The "comma-ok" idiom: safely check if a key exists ──
// Two return values: the value and a boolean (true if key was found).
score, exists := vendorScores["acme"]
if exists {
    // Only use the score if we confirmed the key exists.
    fmt.Println("Acme score:", score)  // Output: Acme score: 85
}

// ── Looking up a missing key returns the zero value (no crash) ──
// For int, the zero value is 0. This does NOT mean the vendor has score 0.
missing := vendorScores["unknown"]  // Returns 0, no error, no crash

// ── Delete a key-value pair ──
// The built-in delete() function removes a key from the map.
// If the key doesn't exist, delete() does nothing (no crash).
delete(vendorScores, "globex")

// ── Iterate over all key-value pairs with range ──
// IMPORTANT: Go deliberately randomizes map iteration order.
// Do NOT rely on maps being in any particular order.
for name, score := range vendorScores {
    fmt.Printf("%s: %d\n", name, score)  // Prints each vendor and their score
}

// ── Creating an empty map with make() ──
// Use make() when you want an empty map to fill later.
agentConfigs := make(map[string]string)  // Empty map, ready for writing
agentConfigs["model"] = "gpt-4"          // Add a key-value pair
agentConfigs["temperature"] = "0.7"     // Add another pair

Control Flow: The for Loop (Go’s Only Loop)

Go has exactly one loop construct: for. There is no while loop and no do-while loop. But do not worry — Go’s for is flexible enough to cover every looping pattern you will ever need. It has three forms, each suited to a different situation.

Think of it like a Swiss Army knife for repetition. One tool, multiple configurations. The classic three-part form (for i := 0; i < 10; i++) is for counted loops. The condition-only form (for condition {}) is for “keep going until something happens” loops. The bare form (for {}) is for infinite loops that you break out of manually. And the range form (for i, v := range collection) is for iterating over collections.

// ── Form 1: Classic three-part for loop ──
// init; condition; post — just like for loops in C, JavaScript, etc.
// i starts at 0, the loop runs while i < 10, and i increases by 1 each time.
for i := 0; i < 10; i++ {
    fmt.Println(i)  // Prints 0, 1, 2, ..., 9
}

// ── Form 2: Condition-only (works like "while" in other languages) ──
// The loop runs as long as the condition is true.
attempts := 0
for attempts < maxRetries {
    err := tryConnect()       // Try to connect to a service
    if err == nil {
        break                 // Success! Exit the loop immediately.
    }
    attempts++                // Failed — increment the counter and try again
}

// ── Form 3: Infinite loop (bare "for" with no condition) ──
// Runs forever until you explicitly break out of it.
// Useful for servers, event loops, and agents that continuously process work.
for {
    msg := receiveMessage()   // Wait for the next incoming message
    if msg == nil {
        break                 // No more messages — exit the loop
    }
    process(msg)              // Handle the message
}

// ── Form 4: range loop — iterate over collections ──
// range gives you the index AND the value for each element.
vendors := []string{"Acme", "Globex", "Initech"}

// With both index and value:
for i, name := range vendors {
    fmt.Printf("%d: %s\n", i, name)  // "0: Acme", "1: Globex", "2: Initech"
}

// Value only — use _ (blank identifier) to discard the index.
// Go requires every variable to be used; _ explicitly says "I don't need this."
for _, name := range vendors {
    fmt.Println(name)  // "Acme", "Globex", "Initech"
}

// Index only — when you just need the position, not the value.
for i := range vendors {
    fmt.Println(i)  // 0, 1, 2
}

The Blank Identifier: _

Go’s compiler refuses to compile code that has unused variables. This is not a warning that you can ignore — it is a hard error. This strict rule keeps codebases clean, but sometimes you genuinely do not need a value. That is what the blank identifier _ is for. It tells Go: “I know this value exists; I am intentionally discarding it.”

Think of _ like a trash can at a post office. When you open a package, sometimes it comes with packing peanuts you do not need. You do not leave them on the counter (compiler error: unused variable). You throw them in the trash (_) because you only care about the item inside.

// ── Discard the index when iterating (you only need the value) ──
for _, vendor := range vendors {
    process(vendor)  // We only care about the vendor, not its index position
}

// ── Discard one return value from a function ──
// doSomething() returns two values, but we only care about the error.
_, err := doSomething()

// ── Import a package only for its side effects ──
// Some packages run initialization code when imported (registering drivers, etc.).
// The _ before the path means "import this, but I won't call any of its functions directly."
import _ "github.com/lib/pq"  // Registers the PostgreSQL database driver

Control Flow: if, switch, and Type Switch

Go’s if statement has a unique feature that you will use constantly: an optional initialization statement before the condition. This lets you declare a variable, use it in the condition, and have it automatically disappear when the if/else block ends. This is especially useful for error handling, where you want the err variable to exist only for as long as you are checking it.

Go’s switch statement is cleaner than in most languages because each case automatically stops at the end — there is no accidental “fall-through” to the next case. You also get type switches, which let you check the actual type of a value and act accordingly. This is extremely useful in agent systems where messages can be different types (text, tool calls, handoffs) and you need to handle each one differently.

// ── if with init statement ──
// The "err := validateVendor(v)" part runs BEFORE the condition is checked.
// The err variable ONLY exists inside this if/else block — it vanishes after.
// This keeps the variable scope tight and your code clean.
if err := validateVendor(v); err != nil {
    return fmt.Errorf("validation failed: %w", err)  // Return an error if validation fails
}
// err is NOT accessible here — it went out of scope with the if block.

// ── switch: no break needed (each case stops automatically) ──
// This is like a routing table: match the risk level and take the right action.
switch riskLevel {
case "critical":
    escalateImmediately()       // Critical risk: alert someone right away
case "high", "medium":         // Multiple values can share one case
    scheduleReview()            // High or medium: schedule a review
default:
    logAndAccept()              // Everything else: log it and move on
}

// ── Type switch: check the actual type of a value ──
// rawMsg is an interface (could be any type). The type switch inspects it.
// Inside each case, "msg" is automatically the correct concrete type.
switch msg := rawMsg.(type) {
case *TextMessage:
    handleText(msg)       // msg is *TextMessage here — you can access its fields
case *ToolCallMessage:
    executeTool(msg)      // msg is *ToolCallMessage here
case *HandoffMessage:
    routeToAgent(msg)     // msg is *HandoffMessage here
default:
    // %T prints the type of the value — useful for debugging unknown types.
    log.Printf("unknown message type: %T", msg)
}

Functions & Multiple Return Values

Functions are the building blocks of any Go program. Think of a function like a recipe: it has a name, takes ingredients (parameters), follows steps (the function body), and produces a dish (the return value). In Go, functions can return multiple values at once — this is not a workaround or a library feature; it is built into the language itself.

The most important pattern in all of Go is the (result, error) return. Almost every function that can fail returns both the result and an error value. If the error is nil (Go’s word for “nothing”), the operation succeeded. If the error is not nil, something went wrong and you should not use the result. This pattern makes error handling explicit and impossible to accidentally ignore.

Go also supports variadic functions (functions that accept any number of arguments), named return values (where you name the return variables in the function signature), and first-class functions (functions can be stored in variables, passed as arguments, and returned from other functions). First-class functions enable powerful patterns like closures and callbacks.

// ── A function that returns two values: a result and an error ──
// This is THE fundamental Go pattern. Almost every function uses it.
// Parameters: vendorID is a string input.
// Returns: (int, error) means "an integer result AND an error value."
func assessVendorRisk(vendorID string) (int, error) {
    // Call lookupVendor, which also returns (result, error).
    // We capture both values and immediately check the error.
    vendor, err := lookupVendor(vendorID)
    if err != nil {
        // Something went wrong. Return 0 for the score and a wrapped error.
        // %w wraps the original error so callers can inspect it later.
        return 0, fmt.Errorf("lookup failed: %w", err)
    }

    // Calculate the risk score from the vendor data.
    score := calculateRiskScore(vendor)

    // Validate the score is in a reasonable range.
    if score < 0 || score > 100 {
        return 0, fmt.Errorf("invalid score %d for vendor %s", score, vendorID)
    }

    // Success! Return the score and nil (no error).
    return score, nil
}

// ── Named return values: the return variables are named in the signature ──
// "result" and "err" are pre-declared as zero-valued variables.
// A bare "return" (with no values) returns their current values.
func divideScores(a, b float64) (result float64, err error) {
    if b == 0 {
        err = fmt.Errorf("division by zero")  // Set the named error variable
        return  // Returns result=0.0 (zero value) and err=the error we just set
    }
    result = a / b  // Set the named result variable
    return           // Returns result=a/b and err=nil (zero value of error)
}

// ── Variadic function: accepts any number of arguments ──
// The "..." before the type means "zero or more values of this type."
// Inside the function, "scores" is a slice ([]int).
func averageScores(scores ...int) float64 {
    if len(scores) == 0 {
        return 0  // Avoid division by zero if no scores were passed
    }
    total := 0
    for _, s := range scores {
        total += s  // Add each score to the running total
    }
    // Convert both to float64 before dividing to get a decimal result.
    return float64(total) / float64(len(scores))
}

// ── First-class functions: functions can be stored in variables ──
// makeScoreFilter returns a FUNCTION as its result.
// The returned function "remembers" the minScore value (this is a closure).
func makeScoreFilter(minScore int) func(Vendor) bool {
    // This anonymous function captures minScore from the outer scope.
    // It can access minScore even after makeScoreFilter has returned.
    return func(v Vendor) bool {
        return v.RiskScore >= minScore  // Compare the vendor's score to the threshold
    }
}

// ── Using the closure ──
// highRiskFilter is a function that returns true for vendors with score >= 80.
highRiskFilter := makeScoreFilter(80)
for _, v := range vendors {
    if highRiskFilter(v) {     // Call the filter function for each vendor
        escalate(v)             // Only escalate vendors above the threshold
    }
}

The := Shadowing Trap

One of the most common bugs for Go newcomers involves variable shadowing with :=. When you use := inside an inner scope (like inside an if block), it creates a brand new variable that happens to have the same name as one in the outer scope. Changes to the inner variable do not affect the outer one. This is especially dangerous with err variables in nested error checks.

Think of it like two people named “Alex” in different rooms. If someone in Room B (inner scope) changes Alex’s shirt, Alex in Room A (outer scope) still has the same shirt. They are different people who share a name.

func shadowingTrap() error {
    var result string  // Outer "result" — starts as "" (zero value)

    if true {
        // BUG: := creates a BRAND NEW "result" variable inside this if block!
        // The outer "result" is completely untouched.
        result, err := fetchData()  // This "result" only lives inside this if block
        if err != nil {
            return err
        }
        fmt.Println(result)  // Prints the fetched data (inner result)
    }
    fmt.Println(result)  // Prints "" — the OUTER result was never changed!

    // ── THE FIX: declare err separately, then use = (not :=) ──
    var err error  // Declare err in the outer scope
    if true {
        // = assigns to the EXISTING outer variables (result and err)
        result, err = fetchData()  // No colon! This modifies the outer variables.
        if err != nil {
            return err
        }
    }
    fmt.Println(result)  // Now this correctly prints the fetched data
    return nil
}
Detecting shadowing
Run go vet -shadow ./... to detect variable shadowing in your codebase. Make it part of your development habit. Shadowing bugs are subtle because the code compiles and runs without errors — it just does the wrong thing silently.

Type Assertions and Type Switches

Sometimes you have a value whose exact type you do not know at compile time. This happens when working with interfaces (which we cover in detail in Chapter 03) — especially the any type, which can hold a value of any type. To get the concrete value out, you use a type assertion.

Think of it like a package at the post office. The box (interface) can contain anything. A type assertion is you opening the box and checking: “Is this a book?” The safe form (val, ok := box.(Book)) checks without crashing. The unsafe form (val := box.(Book)) crashes if the box does not contain a book.

// ── Setup: a variable of type "any" (can hold anything) ──
var val any = "hello"  // val holds a string, but its type is "any"

// ── Safe type assertion with comma-ok ──
// Asks: "Is val a string?" If yes, str gets the string and ok is true.
// If no, str gets the zero value ("") and ok is false.
str, ok := val.(string)
if ok {
    fmt.Println("got string:", str)  // Output: got string: hello
}

// ── Unsafe type assertion (PANICS if the type is wrong!) ──
// Only use this when you are 100% certain of the type.
str = val.(string)  // Works here because val IS a string
// num := val.(int)  // This would PANIC because val is not an int!

// ── Type switch: handle multiple types cleanly ──
// This is incredibly useful for agent systems where different message types
// (text, numbers, JSON objects) flow through the same pipeline.
func formatToolResult(result any) string {
    // switch v := result.(type) checks the type and assigns the typed value to v.
    switch v := result.(type) {
    case string:
        return v                        // v is already a string — return it directly
    case int:
        return fmt.Sprintf("%d", v)      // Convert integer to string
    case float64:
        return fmt.Sprintf("%.2f", v)    // Convert float to string with 2 decimal places
    case []byte:
        return string(v)                // Convert raw bytes to a string
    case nil:
        return "<nil>"                  // Handle the nil case explicitly
    default:
        // For any other type, convert to JSON (a common fallback).
        data, _ := json.Marshal(v)      // json.Marshal converts any value to JSON bytes
        return string(data)             // Convert JSON bytes to string
    }
}

Structs & Interfaces

Structs: Blueprints for Data

Go does not have classes. This is one of its most distinctive and deliberate design choices. Instead of classes, Go has structs — simple data structures that group related fields together. Think of a struct as a blueprint for a form. A vendor registration form has fields for “Name,” “Risk Score,” “Categories,” and “Active Status.” The struct defines what those fields are; each filled-out form is an instance of that struct.

In Go, data (fields) and behavior (methods) are separated. A struct defines the data, and methods are standalone functions that are “attached” to the struct through a special syntax called a receiver. This is different from languages where methods live inside the class body. The separation might feel strange at first, but it has a powerful benefit: you can add methods to types defined anywhere, and the data layout is always clear at a glance.

Structs in Go also use struct tags — metadata strings in backticks after each field. Libraries read these tags at runtime to control behavior like JSON serialization. For example, `json:"risk_score"` tells the JSON library to use “risk_score” as the field name in JSON output instead of the Go field name “RiskScore.”

The visibility rule is simple and consistent: fields that start with an uppercase letter (like Name) are exported (visible to other packages). Fields that start with a lowercase letter (like active) are unexported (only visible within the same package). There are no access modifier keywords — the capitalization is the access modifier.

// ── Define a struct: a blueprint for vendor data ──
// "type" introduces a new type name. "struct" means it's a data structure with fields.
type Vendor struct {
    // Each field has a name, a type, and an optional struct tag (in backticks).
    // Struct tags are metadata that libraries use — here, the json library uses them
    // to know what name to use when converting this struct to/from JSON.
    ID         string    `json:"id"`            // Exported (uppercase): visible outside this package
    Name       string    `json:"name"`          // Exported: the vendor's human-readable name
    RiskScore  int       `json:"risk_score"`    // Exported: numeric risk assessment (0-100)
    Categories []string  `json:"categories"`    // Exported: slice of category labels
    Active     bool      `json:"active"`        // Exported: whether the vendor is currently active
}

// ── Create instances of the struct ──
func main() {
    // Method 1: Named fields (recommended — clear and order-independent)
    vendor := Vendor{
        ID:        "v-001",                      // Assign by field name
        Name:      "Acme Corp",                   // Order doesn't matter with named fields
        RiskScore: 85,                             // Unmentioned fields get their zero values
        Active:    true,                            // Categories is omitted, so it's nil ([]string zero value)
    }

    // Access fields with dot notation.
    fmt.Println(vendor.Name)       // Output: Acme Corp
    fmt.Println(vendor.RiskScore)  // Output: 85
}

Methods & Receivers: Value vs Pointer

A method in Go is a function with a special first parameter called a receiver. The receiver ties the function to a specific type, allowing you to call it with dot syntax (like vendor.IsHighRisk()). Think of the receiver as the person performing the action: “this vendor checks if it is high risk.”

There are two kinds of receivers, and understanding the difference is critical:

A value receiver (written v Vendor) gets a copy of the struct. It is like photocopying a form and writing on the photocopy — the original form is unchanged. Use value receivers for methods that only read data.

A pointer receiver (written v *Vendor) gets the memory address of the original struct. It is like holding the actual form and writing directly on it — the original is modified. Use pointer receivers when you need to change the struct’s data.

A good rule of thumb: if any method on a struct needs a pointer receiver, use pointer receivers for all methods on that struct. This keeps things consistent and avoids confusing behavior.

// ── Value receiver: works on a COPY of the struct ──
// "v Vendor" means this method receives a copy of a Vendor.
// Any changes inside this method do NOT affect the original.
// Use for read-only methods that just compute something and return it.
func (v Vendor) IsHighRisk() bool {
    return v.RiskScore > 80  // Just reads RiskScore, doesn't change anything
}

// ── Pointer receiver: works on the ORIGINAL struct ──
// "v *Vendor" means this method receives the memory address of the original Vendor.
// Changes inside this method DO affect the original.
// Use when you need to modify the struct's data.
func (v *Vendor) UpdateScore(newScore int) {
    v.RiskScore = newScore  // Modifies the ORIGINAL vendor's score
}

// ── Common mistake: trying to mutate with a value receiver ──
// This method gets a COPY, so changes vanish when the method returns.
func (v Vendor) BrokenUpdate(newScore int) {
    v.RiskScore = newScore  // Changes the COPY — the original is untouched!
}

func main() {
    // Create a vendor with an initial risk score of 50.
    vendor := Vendor{Name: "Acme", RiskScore: 50}

    // Call the broken method (value receiver) — the score does NOT change.
    vendor.BrokenUpdate(99)
    fmt.Println(vendor.RiskScore)  // Still 50! The copy was modified, not the original.

    // Call the correct method (pointer receiver) — the score DOES change.
    vendor.UpdateScore(99)
    fmt.Println(vendor.RiskScore)  // 99 — the original was modified through the pointer.
}
Analogy: photocopy vs. original document
A value receiver is like handing someone a photocopy of a document. They can scribble all over it, but your original is safe. A pointer receiver is like handing someone the original document with a pen. Whatever they write on it is permanent. Choose carefully based on whether you want the method to be able to change the data.

Constructor Pattern

Go has no constructors — there is no special new ClassName() syntax. Instead, the convention is to write a regular function named NewTypeName that creates and returns a pointer to your struct. This function serves as the constructor: it sets up default values, validates inputs, and ensures the struct starts in a valid state.

Think of a constructor function like a factory that produces filled-out forms. Instead of giving someone a blank form and hoping they fill in all the required fields correctly, the factory pre-fills the obvious defaults (like setting Active to true for a new vendor) and only asks for the information that the caller must provide.

// ── Constructor function: creates a new Vendor with sensible defaults ──
// By convention, constructor functions are named "New" + the type name.
// Returns a pointer (*Vendor) because the caller usually wants to modify it later.
func NewVendor(id, name string) *Vendor {
    // &Vendor{...} creates a Vendor AND returns its memory address (a pointer).
    // The & symbol means "give me the address of this thing."
    return &Vendor{
        ID:     id,              // Caller provides the ID
        Name:   name,            // Caller provides the name
        Active: true,            // Default: new vendors start as active
        // RiskScore defaults to 0 (zero value) — not yet assessed
        // Categories defaults to nil (zero value) — no categories assigned yet
    }
}

// ── Usage ──
vendor := NewVendor("v-001", "Acme Corp")  // vendor is a *Vendor (pointer)
fmt.Println(vendor.Name)                     // Output: Acme Corp
fmt.Println(vendor.Active)                   // Output: true (set by the constructor)

Composition via Embedding

Go does not have inheritance. There is no extends keyword, no class hierarchies, no “is-a” relationships between types. Instead, Go uses embedding — a form of composition where one struct includes another as a field. The embedded struct’s fields and methods are “promoted” to the outer struct, meaning you can access them directly without going through the embedded field name.

Think of it like building with LEGO bricks. Inheritance is like Russian nesting dolls — each doll must fit inside a specific larger doll, and the hierarchy is rigid. Embedding is like LEGO — you can snap any brick onto any other brick in any combination. A RiskAgent is not “a child of BaseAgent”; it contains a BaseAgent as one of its building blocks.

This distinction is important. With inheritance, the child class “knows” about the parent. With embedding, the embedded struct has no idea it is being embedded. There is no super keyword, no virtual method dispatch, no fragile base class problem. You get code reuse without the coupling and complexity that deep inheritance hierarchies bring.

// ── Base struct: contains common fields and methods for all agents ──
type BaseAgent struct {
    ID    string  // Unique identifier for this agent
    Name  string  // Human-readable name
    Model string  // Which LLM model this agent uses (e.g., "gpt-4")
}

// LogAction is a method on BaseAgent. It logs what the agent is doing.
func (a BaseAgent) LogAction(action string) {
    // log.Printf formats and prints a log message with a timestamp.
    log.Printf("[%s] %s", a.Name, action)
}

// ModelName returns the LLM model this agent uses.
func (a BaseAgent) ModelName() string {
    return a.Model
}

// ── RiskAgent EMBEDS BaseAgent (composition, not inheritance) ──
// "BaseAgent" without a field name means it is embedded.
// Its fields and methods are "promoted" — accessible directly on RiskAgent.
type RiskAgent struct {
    BaseAgent              // Embedded (anonymous) field — promotes all BaseAgent fields/methods

Interfaces: Contracts Without Declarations

Interfaces in Go are profoundly different from interfaces in most other languages, and understanding them is the key to writing idiomatic Go. An interface defines a set of method signatures (a contract). Any type that has all the methods listed in the interface automatically satisfies it — without declaring that it does.

This is called implicit satisfaction, and it is one of Go’s most powerful features. There is no implements keyword. The type does not need to know the interface exists. It just needs to have the right methods with the right signatures.

Think of interfaces like a job posting. The posting says: “We need someone who can cook pasta and bake bread.” Anyone who can do both those things qualifies for the job — they do not need a certificate saying “I am a certified Pasta-and-Bread specialist.” They just need the skills. Similarly, any struct that has the methods an interface requires automatically satisfies that interface — no special declaration needed.

This design enables an extraordinary level of decoupling. You can define an interface in one package and satisfy it in a completely separate package that does not even import the first one. This means you can swap implementations, create test mocks, and extend your system without modifying existing code.

// ── Define an interface: a contract describing behavior ──
// "Agent" describes what any agent must be able to do.
// Any type with both Run() and Name() methods automatically satisfies this interface.
type Agent interface {
    // Run executes the agent with a context and messages, returning a response or error.
    Run(ctx context.Context, msgs []Message) (Response, error)
    // Name returns the human-readable name of this agent.
    Name() string
}

// ── RiskAssessor satisfies the Agent interface ──
// There is NO "implements Agent" declaration anywhere. It just has the right methods.
type RiskAssessor struct {
    model  string      // lowercase = unexported (private to this package)
    tools  []Tool      // tools this agent can call
    client LLMClient   // the LLM API client
}

// Run method: satisfies the first method of the Agent interface.
// The pointer receiver (*RiskAssessor) means this method can modify the struct.
func (r *RiskAssessor) Run(ctx context.Context, msgs []Message) (Response, error) {
    // ... implementation would go here ...
    return Response{}, nil  // Return an empty response and no error (placeholder)
}

// Name method: satisfies the second method of the Agent interface.
func (r *RiskAssessor) Name() string {
    return "risk-assessor"  // Return a descriptive name for this agent
}

// ── Compile-time verification: prove that RiskAssessor satisfies Agent ──
// This line assigns a nil *RiskAssessor to an Agent variable.
// If *RiskAssessor is missing any Agent methods, this line fails to compile.
// The _ discards the variable — we only care about the type check.
var _ Agent = (*RiskAssessor)(nil)
The compile-time interface check
The line var _ Agent = (*RiskAssessor)(nil) is a common Go idiom. It creates a nil pointer of type *RiskAssessor and tries to assign it to a variable of type Agent. If *RiskAssessor does not have all the methods required by Agent, the compiler rejects this line with a clear error message telling you which methods are missing. This is your safety net during development.

Interface Embedding

Just as structs can embed other structs, interfaces can embed other interfaces. This lets you compose small, focused interfaces into larger ones. The Go standard library does this extensively: io.Reader has one method (Read), io.Writer has one method (Write), and io.ReadWriter embeds both to require both methods.

The best practice in Go is to keep interfaces small — one to three methods. A huge interface with ten methods is hard to satisfy and hard to mock in tests. Small interfaces are more flexible because types can satisfy many small interfaces without being designed specifically for any of them.

// ── Small, focused interfaces (one method each) ──
type Reader interface {
    // Read fills the byte slice p with data and returns how many bytes were read.
    Read(p []byte) (n int, err error)
}

type Writer interface {
    // Write sends the byte slice p somewhere and returns how many bytes were written.
    Write(p []byte) (n int, err error)
}

// ── Composed interface: embeds both Reader and Writer ──
// Any type that satisfies ReadWriter must have BOTH Read() and Write() methods.
type ReadWriter interface {
    Reader  // Embed the Reader interface (requires Read method)
    Writer  // Embed the Writer interface (requires Write method)
}

// ── The empty interface: "any" ──
// "any" (or the older syntax "interface{}") is an interface with zero methods.
// EVERY type satisfies it, because every type has at least zero methods.
// Use it when you need a function that accepts literally any type of value.
func logAnything(val any) {
    // %T prints the type, %v prints the value in its default format.
    fmt.Printf("type=%T, value=%v\n", val, val)
}
Accept interfaces, return structs
This is one of the most important Go design rules. Function parameters should be interfaces so callers can pass any type that satisfies the contract. Return values should be concrete types so callers know exactly what they are getting. Keep interfaces small — the standard library’s io.Reader (one method) and io.Writer (one method) are the gold standard of interface design.

Error Handling

Error handling in Go is fundamentally different from what you may have seen in other languages. There are no try/catch blocks. There are no exceptions. There is no throws clause. Instead, errors in Go are ordinary values — just like strings or integers — that are returned alongside the result of a function call.

The philosophy behind this design is profound: errors are a normal, expected part of program execution, not “exceptional events” that require special control flow. A network request might fail. A file might not exist. An LLM might return garbage. These are not exceptional — they are everyday occurrences that your code must handle gracefully.

Think of it like a restaurant order. In some restaurants (exception-based languages), the waiter takes your order and walks away. If the kitchen is out of an ingredient, an alarm blares and the manager runs out to handle the crisis. In Go’s restaurant, the waiter comes back and says: “Here is your pasta. Oh, and we were out of parmesan, so I brought pecorino instead. Is that okay?” The error information comes back alongside the result, and you (the caller) decide what to do about it.

The canonical Go pattern is: result, err := doSomething(). You get both the result and an error. If err is nil, the operation succeeded and you can use result. If err is not nil, something went wrong and you must handle it before touching result.

The if err != nil Pattern

You will write if err != nil hundreds, possibly thousands of times in a Go project. This is not a flaw — it is the intended design. Each if err != nil check is a conscious decision point: “Something could go wrong here. What do I do about it?” The repetition makes error paths visible in code review. You can scan down a function and immediately see every place where an error is checked and handled.

When handling an error, you have three choices: (1) handle it (take corrective action), (2) wrap it with context and propagate it (add information about what you were doing when the error occurred, then pass it up to the caller), or (3) discard it (which you should almost never do). The most common choice is wrapping and propagating — each layer of your code adds context, creating a trail that makes debugging easy.

// ── A function that chains multiple operations, each of which can fail ──
// Notice how each step checks for errors before proceeding to the next.
func processVendorAssessment(ctx context.Context, vendorID string) (*Report, error) {
    // Step 1: Look up the vendor. This could fail (vendor not found, DB error, etc.).
    vendor, err := lookupVendor(ctx, vendorID)
    if err != nil {
        // Wrap the error with context: WHAT we were doing when it failed.
        // %w wraps the original error so callers can inspect it later.
        return nil, fmt.Errorf("lookup vendor %s: %w", vendorID, err)
    }

    // Step 2: Fetch documents. Only runs if Step 1 succeeded.
    docs, err := fetchDocuments(ctx, vendor)
    if err != nil {
        return nil, fmt.Errorf("fetch docs for %s: %w", vendorID, err)
    }

    // Step 3: Calculate risk. Only runs if Steps 1 and 2 succeeded.
    score, err := calculateRisk(ctx, vendor, docs)
    if err != nil {
        return nil, fmt.Errorf("risk calculation for %s: %w", vendorID, err)
    }

    // All steps succeeded. Return the complete report with nil error.
    // &Report{...} creates the report and returns a pointer to it.
    return &Report{Vendor: vendor, Score: score, Documents: docs}, nil
}

Error Wrapping with %w

When you propagate an error, you should wrap it with context using fmt.Errorf("context: %w", err). The %w verb (introduced in Go 1.13) is special: it creates a new error that contains the original error inside it, forming a chain. This chain is like a paper trail — each layer adds information about what was happening when the error occurred.

Think of error wrapping like a customer complaint in a large company. The cashier says “the credit card was declined.” The store manager wraps it: “Customer purchase at Store #42 failed: the credit card was declined.” The regional manager wraps it again: “Revenue loss at Northeast region: customer purchase at Store #42 failed: the credit card was declined.” Each layer adds context without losing the original cause. You can “unwrap” the chain at any point to find the root cause.

// ── Layer 1: The database returns a raw error ──
// sql.ErrNoRows is a predefined error meaning "no matching rows found."
err := db.QueryRow("SELECT ...").Scan(&vendor)
// err = sql.ErrNoRows

// ── Layer 2: The repository wraps it with "what we were trying to do" ──
// %w preserves the original error inside the new one.
return nil, fmt.Errorf("vendor repo: find by id %s: %w", id, err)
// err = "vendor repo: find by id v-123: sql: no rows in result set"

// ── Layer 3: The service wraps it again with even more context ──
return nil, fmt.Errorf("assessment service: %w", err)
// err = "assessment service: vendor repo: find by id v-123: sql: no rows"

// ── At ANY layer, you can check if the original error is inside the chain ──
// errors.Is() looks through the entire chain for a matching error.
if errors.Is(err, sql.ErrNoRows) {
    // Even though the error was wrapped twice, we can still detect the root cause.
    // This lets the HTTP handler return a proper 404 response.
}

Sentinel Errors vs Custom Error Types

Go provides two main patterns for creating your own errors, and understanding when to use each is important for building robust systems.

Sentinel errors are predefined error values — like constants. They represent specific, well-known failure conditions. Think of them like warning signs on a road: “Bridge Out,” “Road Closed,” “Speed Limit.” Each sign has a fixed meaning that everyone understands. You create sentinel errors with errors.New() and check for them with errors.Is(). Use them when the identity of the error is all you need — “was this a not-found error?” “was this a timeout?”

Custom error types are structs that implement the error interface (which requires just one method: Error() string). They carry additional structured data beyond a message. Think of them like incident reports: they include not just “something went wrong” but also which agent failed, at what step, with what input, and what the recovery options are. You check for custom error types with errors.As(). Use them when callers need structured information from the error, not just its identity.

// ── Sentinel errors: predefined, well-known error values ──
// Declare them as package-level variables using errors.New().
// Convention: start with "Err" prefix (e.g., ErrNotFound, ErrTimeout).
var (
    ErrVendorNotFound  = errors.New("vendor not found")        // The vendor doesn't exist
    ErrScoreOutOfRange = errors.New("score must be 0-100")    // Invalid risk score
    ErrAgentTimeout    = errors.New("agent execution timed out")  // Agent took too long
    ErrRateLimited     = errors.New("API rate limit exceeded")    // Too many API calls
)

// ── Custom error type: carries structured data about what went wrong ──
// This struct holds detailed information about an agent failure.
type AgentError struct {
    AgentName string  // Which agent failed
    Step      string  // At what step in the pipeline
    Cause     error   // The underlying error that caused the failure
}

// Error() satisfies the "error" interface. This is the only method required.
// It returns a human-readable description of the error.
func (e *AgentError) Error() string {
    return fmt.Sprintf("agent %s failed at step %s: %v", e.AgentName, e.Step, e.Cause)
}

// Unwrap() enables error chain traversal. It returns the underlying error.
// This allows errors.Is() and errors.As() to look inside this error.
func (e *AgentError) Unwrap() error {
    return e.Cause
}

// ── Checking for sentinel errors with errors.Is() ──
// errors.Is() looks through the ENTIRE error chain for a match.
// Even if the error was wrapped multiple times, Is() can find the original.
if errors.Is(err, ErrVendorNotFound) {
    // The root cause was "vendor not found" — return a 404 HTTP response.
    http.Error(w, "vendor not found", http.StatusNotFound)
}

// ── Extracting a custom error type with errors.As() ──
// errors.As() looks through the chain for a specific error TYPE
// and fills the target variable with the matching error.
var agentErr *AgentError                // Declare a variable to receive the typed error
if errors.As(err, &agentErr) {          // Pass a pointer to the variable
    // We found an AgentError in the chain — now we can access its fields.
    slog.Error("agent failed",            // Log a structured error message
        "agent", agentErr.AgentName,    // Which agent failed
        "step", agentErr.Step,          // At what step
    )
}

Defer: Guaranteed Cleanup

The defer statement schedules a function call to run when the enclosing function exits, no matter how it exits — whether by a normal return, an error return, or even a panic (crash). Deferred calls execute in LIFO order (last-in, first-out): the last defer runs first.

Think of defer like a “cleanup checklist” you write as you go. When you open a file, you immediately write on your checklist: “close this file when I leave.” When you acquire a lock, you write: “release this lock when I leave.” No matter what happens next — even if an emergency interrupts you — Go guarantees that your cleanup checklist will be executed. This is incredibly valuable for resource management: files get closed, locks get released, database connections get returned to the pool.

The most important pattern is: defer the cleanup immediately after acquiring the resource. Open a file? The very next line should be defer file.Close(). Lock a mutex? The very next line should be defer mutex.Unlock(). This pairing ensures you never forget to clean up, regardless of how complex the function becomes.

// ── File cleanup with defer ──
// This function opens a file, reads its contents, and ALWAYS closes it.
func readConfig(path string) ([]byte, error) {
    // os.Open opens a file for reading. Returns the file handle and an error.
    f, err := os.Open(path)
    if err != nil {
        return nil, err  // File couldn't be opened — return the error
    }
    // Schedule f.Close() to run when this function exits.
    // This runs REGARDLESS of whether the function returns normally or with an error.
    defer f.Close()

    // Read the entire file contents. Even if this fails, f.Close() still runs.
    return io.ReadAll(f)
}

// ── Mutex unlock with defer ──
// SafeStore uses a read-write lock to protect concurrent access to a map.
func (s *SafeStore) Get(key string) (string, bool) {
    s.mu.RLock()              // Acquire a read lock (multiple readers can hold this simultaneously)
    defer s.mu.RUnlock()      // ALWAYS release the lock when this function exits

    val, ok := s.data[key]    // Look up the key in the map
    return val, ok             // Return the value and whether it was found
}

// ── LIFO order: defers run in reverse order of declaration ──
func deferOrder() {
    defer fmt.Println("first deferred — runs LAST")   // Deferred #1
    defer fmt.Println("second deferred — runs FIRST")  // Deferred #2
    fmt.Println("normal execution")                    // Runs immediately
    // Output:
    // normal execution
    // second deferred — runs FIRST
    // first deferred — runs LAST
}

// ── Panic recovery with defer ──
// recover() catches a panic (crash) and converts it into a normal error.
// This is essential in agent systems: one agent crashing should NOT
// bring down the entire orchestrator.
func safeAgentRun(agent Agent, input []Message) (resp Response, err error) {
    // This deferred function runs when safeAgentRun exits.
    // If a panic occurred, recover() returns the panic value; otherwise nil.
    defer func() {
        if r := recover(); r != nil {
            // A panic happened! Convert it into a regular error.
            // The named return value "err" is set here, and returned to the caller.
            err = fmt.Errorf("agent panicked: %v", r)
        }
    }()
    // Run the agent. If it panics, the deferred function above catches it.
    return agent.Run(context.Background(), input)
}
Panic vs Error: know the difference
Use panic only for truly unrecoverable programmer bugs: array index out of bounds, nil pointer dereference, impossible states that indicate a logic error in your code. For everything else — network failures, invalid user input, LLM timeouts, missing files, rate limiting — return an error. In agent systems, robustness is paramount: a single agent crashing should not bring down the entire orchestrator.

Error Handling Strategies for Agent Systems

Agent systems operate in an inherently unreliable environment. LLMs sometimes return nonsense. APIs have rate limits and outages. Network connections drop. Tools return unexpected output formats. A well-designed agent system anticipates these failures at every level.

Think of a well-run hospital emergency room. The triage nurse (orchestrator) assigns patients to doctors (agents). If one doctor is unavailable (agent error), the nurse routes the patient to another doctor. If the lab (tool) cannot process a test, the doctor makes a judgment with available information rather than refusing to treat the patient. The ER continues functioning even when individual components fail.

Here are the error handling strategies you should use at each layer:

  • Tool level: Return descriptive errors with context. Include what was attempted and why it failed. Wrap underlying errors with %w.
  • Agent level: Decide whether to retry, use a fallback, or note the gap and continue. A risk agent that fails to call the breach database should continue the assessment with available data, noting the gap in the report.
  • Orchestrator level: Decide whether partial results are acceptable. If three of four parallel agents succeed, can you still produce a useful report? Usually yes.
// ── Error classification for agent systems ──
// Categorizing errors tells the system HOW to respond to each failure type.
type ErrorKind int

const (
    // ErrKindTransient: the error is temporary — retrying may succeed.
    // Examples: network timeout, HTTP 503, rate limiting.
    ErrKindTransient ErrorKind = iota

    // ErrKindPermanent: retrying will NOT help — the request is fundamentally broken.
    // Examples: invalid input, authentication failure, vendor not found.
    ErrKindPermanent

    // ErrKindPartial: the operation partially succeeded — some data is available.
    // Examples: 3 of 4 data sources responded, some fields are missing.
    ErrKindPartial
)

// classifyError examines an error and determines its kind.
// This classification drives the retry/fallback logic in the orchestrator.
func classifyError(err error) ErrorKind {
    // Check if the error is a deadline exceeded (timeout) or rate limiting.
    // These are transient — retrying after a delay may succeed.
    if errors.Is(err, context.DeadlineExceeded) || errors.Is(err, ErrRateLimited) {
        return ErrKindTransient
    }

    // Check if the error is an HTTP error with a 5xx status code.
    // Server errors are usually transient (the server is temporarily overwhelmed).
    var httpErr *HTTPError
    if errors.As(err, &httpErr) && httpErr.StatusCode >= 500 {
        return ErrKindTransient
    }

    // Everything else is assumed permanent — don't waste resources retrying.
    return ErrKindPermanent
}
Analogy: the restaurant kitchen during a rush
Error classification in an agent system works like a busy restaurant kitchen. If the oven is temporarily broken (transient error), the chef waits and retries. If the kitchen is out of an ingredient completely (permanent error), the chef adapts the recipe or informs the customer. If the salad is ready but the steak is delayed (partial success), the waiter might serve the salad first and bring the steak when ready. Your agent system should have the same intelligence: retry what is fixable, accept what is not, and deliver partial results when full results are unavailable.

Packages & Modules

Think of a Go project as a building. The building itself is the module — it has a street address (the module path), a blueprint (the go.mod file), and a security manifest listing every supplier that contributed materials (go.sum). Inside the building, each room is a package. Every room has a specific purpose: the kitchen handles food prep, the office handles paperwork, the storage room holds inventory. You wouldn't mix kitchen equipment into the office, and you wouldn't scatter paperwork across the kitchen. That's exactly how Go packages work — each directory is a room with a single, clear purpose, and all the .go files in that directory share the same package name.

This building analogy extends further. When you need supplies from outside — say, a specialized security scanner or an AI toolkit — you order them from a supplier (an external module). Go fetches these directly from where the supplier hosts their code (usually GitHub or GitLab), records the exact version in go.mod, and stamps a cryptographic fingerprint in go.sum so nobody can swap the delivery with a tampered package. It's like having a chain-of-custody form for every single dependency.

Unlike many languages where you can import individual files, Go's unit of code organization is always the directory. Every .go file in a directory must declare the same package name. You import entire packages, not individual files or functions. This keeps things simple: when you see import "github.com/yourname/my-agents/internal/agent", you know you're bringing in everything that the agent package exports, from one specific directory.

Let's start from the very beginning: creating a module and understanding each piece of the system.

Creating a Module: The go.mod File

Every Go project begins with go mod init. This creates a go.mod file, which is your project's identity card. It declares two things: who you are (the module path) and what Go version you need. The module path is typically a URL-like string that matches where your code is hosted, because this is also how other people import your packages. When someone writes import "github.com/yourname/my-agents/internal/scanner", Go uses that module path to find and download your code.

Here's what the process looks like from scratch. Imagine you're building a multi-agent system — software that helps organizations evaluate risks using coordinated AI agents.

// Step 1: Create a directory for your project
// This is like breaking ground on a new building
mkdir my-agents          // create the project folder
cd my-agents             // move into it

// Step 2: Initialize the module
// This creates go.mod — your project's identity card
// The module path is how others import your code

After running that command, you'll have a go.mod file. Let's look at what a more mature go.mod looks like once you've added some dependencies:

// This file declares your module identity and all external dependencies

// each line is: module-path version

    github.com/sashabaranov/go-openai v1.26.0  // OpenAI API client for LLM calls

)

The go.sum File: Your Dependency Fingerprints

When you add a dependency, Go also creates or updates go.sum. This file contains cryptographic hashes (SHA-256 checksums) of every dependency. Think of it as a tamper-proof receipt: if someone modifies a dependency after you downloaded it (a supply chain attack), the hash won't match and Go will refuse to build. You should always commit both go.mod and go.sum to version control.

Visibility: The Restaurant Analogy

Go has the simplest visibility system of any major language. There are no public, private, or protected keywords. Instead, Go uses a single rule based on the first letter of a name:

  • Uppercase first letter = Exported (visible to other packages)
  • Lowercase first letter = Unexported (only visible within the same package)

Think of it like a restaurant. The menu (exported names) is what customers see — dish names, prices, descriptions. These are the public interface. The kitchen recipes (unexported names) are internal secrets — the exact spice blends, cooking temperatures, supplier names. Customers don't need to know the recipe; they just order from the menu. Similarly, other packages don't need to see your internal helper functions; they just call your exported API.

This convention applies to everything: functions, types, struct fields, methods, constants, and variables. If it starts with a capital letter, anyone can use it. If it starts with a lowercase letter, only code in the same package can see it.

package vendor  // this file belongs to the "vendor" package

// ── Exported (the Menu — visible to other packages) ──

// Vendor is an exported type — other packages can create and use Vendor values
// The uppercase 'V' makes it part of the public menu
type Vendor struct {
    Name      string  // exported field — anyone can read/write this
    RiskScore int     // exported field — part of the public data
    active    bool    // unexported field — only this package can touch it
    notes     string  // unexported field — internal kitchen secret
}

// NewVendor is an exported function — this is a "constructor"
// Other packages call vendor.NewVendor("Acme") to create a Vendor
// The uppercase 'N' puts it on the public menu
func NewVendor(name string) *Vendor {
    return &Vendor{       // create a new Vendor and return a pointer to it
        Name:   name,      // set the public name
        active: true,      // set the private field — only we can do this
    }
}

// IsActive is an exported method — other packages can check if a vendor is active
// This is how you provide controlled access to unexported fields
func (v *Vendor) IsActive() bool {
    return v.active  // we can read the unexported field because we're in the same package
}

// ── Unexported (the Kitchen — hidden from other packages) ──

// calculateRisk is unexported — the lowercase 'c' keeps it in the kitchen
// Only code inside the "vendor" package can call this function
// This is our secret recipe for risk scoring
func calculateRisk(v *Vendor) int {
    score := 0           // start with a base score of zero
    if !v.active {        // check the private field
        score += 50       // inactive vendors are riskier
    }
    return score           // return the calculated risk
}

// defaultTimeout is an unexported constant — internal configuration
// Other packages don't need to know our default timeout value
const defaultTimeout = 30  // seconds — only used inside this package

Now let's see what happens when another package tries to use these:

package main  // this is a different package from "vendor"

import (
    "fmt"                                          // standard library formatting
    "github.com/yourname/my-agents/internal/vendor" // import the vendor package
)

func main() {
    // This works — NewVendor is exported (uppercase N)
    v := vendor.NewVendor("Acme Corp")  // call the public constructor

    // This works — Name is exported (uppercase N)
    fmt.Println(v.Name)  // prints "Acme Corp"

    // This works — IsActive is exported (uppercase I)
    fmt.Println(v.IsActive())  // prints "true"

    // COMPILE ERROR — active is unexported (lowercase a)
    // fmt.Println(v.active)  // cannot access — it's a kitchen secret!

    // COMPILE ERROR — calculateRisk is unexported (lowercase c)
    // vendor.calculateRisk(v)  // cannot call — it's a kitchen recipe!
}
Why this design is brilliant
The uppercase/lowercase convention means you can tell at a glance whether something is public or private, without looking at any declaration. When you read vendor.NewVendor(), the capital N instantly tells you it's an exported function. When you see v.active inside the vendor package, the lowercase a tells you it's private. No keywords needed — the information is baked into the name itself.

The internal/ Directory: A Stronger Lock

Unexported names protect things within a single package. But what about when you have multiple packages in your project that should share code with each other, but not with outside users? That's what the internal/ directory is for. Go enforces a special rule: packages inside internal/ can only be imported by code in the parent directory tree. Think of it as a staff-only area in the restaurant — employees from different departments (packages) can enter, but customers (external code) cannot.

// Project structure showing internal/ usage
// Think of this as the building floor plan

// github.com/yourname/my-agents/

The init() Function: Automatic Setup

Go lets you define a special function called init() in any file. This function runs automatically when the package is first loaded, before main() executes. Think of it as the setup crew that arrives before the restaurant opens — turning on ovens, stocking ingredients, preparing the space. You don't call init() yourself; Go calls it for you.

Use init() sparingly. It's best for truly one-time setup tasks like registering database drivers. Overusing init() makes your code harder to test and reason about, because things happen "magically" before your code runs. Prefer explicit initialization in main() when possible.

package database  // this package handles database connections

import (
    "database/sql"          // Go's standard database interface
    "log"                   // logging package for startup messages
    _ "github.com/lib/pq"  // the underscore _ means "import for side effects only"
                            // this runs pq's init() which registers the PostgreSQL driver
                            // we don't directly call any pq functions — we just need it registered
)

// init runs automatically when this package is first imported
// it runs BEFORE main() — think of it as the pre-opening setup crew
func init() {
    log.Println("database package loaded — PostgreSQL driver registered")  // log that we're ready
}
Use init() sparingly
init() functions run before test setup, cannot accept arguments, and cannot return errors. This makes them hard to test and debug. In production code, prefer explicit initialization in main() where you can pass configuration and handle errors gracefully. Reserve init() for truly package-level setup like registering database drivers or encoding types.

Package Design Guidelines

Good package design is one of the most impactful architectural decisions in a Go project. Think of it as designing the rooms in your building — each room should have a clear purpose, and you should be able to describe what it does in one sentence. Here are the key principles:

  • Name packages by what they provide, not what they contain. Use agent (good) not agents, agentutils, or agenthelpers (bad). Avoid generic names like utils, common, or helpers — they become dumping grounds.
  • Avoid circular imports. Go strictly forbids circular dependencies between packages. If package A imports package B, then B cannot import A. This forces clean, one-way dependency graphs and is one reason Go codebases are easier to navigate.
  • Keep the exported API small. Only export what other packages genuinely need. Every exported name is a commitment — once external code depends on it, you can't easily change it. Think of exports as menu items: a focused menu is better than a 200-item one.
  • One package per directory. All .go files in a directory must have the same package declaration. There's no nesting — internal/agent/ and internal/agent/tools/ are two separate packages.

Chapter 06

Pointers & Memory

What Even Is Memory?

Before we talk about pointers, let's understand what's actually happening inside your computer. When your program runs, the operating system gives it a block of memory — think of it as a gigantic numbered mailbox wall at a post office. Each mailbox has a unique number (its address, like 0xc0000b4000) and can hold a piece of data. When you create a variable like name := "Acme Corp", Go finds an empty mailbox, writes "Acme Corp" into it, and remembers that when you say name, you mean "whatever is in mailbox #4832."

Normally, you never think about mailbox numbers. You just use the variable name, and Go handles the mapping. But sometimes, you don't want to put a copy of the data into a new mailbox — you want to tell someone else which mailbox to look in. That's what a pointer is.

The House Address Analogy

Imagine your friend Alice lives in a house. The house itself is the data (a big, physical thing). The house's street address (123 Main Street) is the pointer. If someone asks you about Alice's house, you have two choices:

  • Copy the whole house (pass by value): Build an exact replica of Alice's house at a new location and give them that. Now there are two independent houses. If they repaint their copy, Alice's original house is unaffected.
  • Give them the address (pass by pointer): Write "123 Main Street" on a piece of paper and hand it to them. Now they can go to Alice's actual house. If they repaint it, they're repainting Alice's real house.

Pointers are the "piece of paper with the address." They're tiny (just 8 bytes on a 64-bit machine, regardless of how big the house is), and they let you share and modify the original data without copying it. Go makes pointers safe: unlike languages like C, you can't do pointer arithmetic (no "go two houses to the left"), and Go's garbage collector automatically cleans up houses that nobody has the address for anymore.

Let's see this in actual Go code, starting from the very basics:

Your First Pointer: & and *

Go gives you two operators for working with pointers. The & operator (read as "address of") gives you the address of a variable. The * operator (read as "value at") follows an address to get the data stored there. These are inverse operations: & goes from value to address, * goes from address back to value.

package main  // entry point of our program

import "fmt"  // standard library for formatted output

func main() {
    // Step 1: Create a regular variable
    // This is like building a house — the data lives directly in the variable
    // Memory: [mailbox #100] contains "Acme Corp"
    vendorName := "Acme Corp"  // vendorName holds the string "Acme Corp" directly

    // Step 2: Create a pointer using & (address-of operator)
    // This is like writing down the house's street address on a piece of paper
    // Memory: [mailbox #200] contains the ADDRESS of mailbox #100
    ptr := &vendorName  // ptr is of type *string — "a pointer to a string"
                         // ptr doesn't hold "Acme Corp" — it holds the ADDRESS where "Acme Corp" lives

    // Step 3: Print the pointer itself — you'll see a memory address
    fmt.Println(ptr)   // Output: 0xc0000b4000 (or similar hex address)
                        // This is the "street address" — just a number

    // Step 4: Dereference the pointer using * (value-at operator)
    // This is like going to the street address and looking inside the house
    fmt.Println(*ptr)  // Output: "Acme Corp" — we followed the address to find the data

    // Step 5: Modify the original data through the pointer
    // This is like going to Alice's house (via her address) and repainting it
    *ptr = "Globex Inc"  // follow the address, change what's stored there

    // Step 6: The original variable is changed!
    // Because the pointer pointed at the SAME mailbox, not a copy
    fmt.Println(vendorName)  // Output: "Globex Inc" — the original was modified!

    // ┌─────────────────────┐     ┌────────────────────────────┐
    // │ ptr                 │     │ vendorName                 │
    // │ [address: 0xc000..] │────▶│ "Globex Inc"               │
    // │ type: *string       │     │ type: string               │
    // └─────────────────────┘     └────────────────────────────┘
    //   The pointer                  The actual data
    //   (piece of paper              (the house)
    //    with address)
}

Value vs Pointer: The Photocopy Machine

Here's the critical concept that trips up every beginner. When you pass a variable to a function in Go, Go always makes a copy. Always. No exceptions. The question is: what are you copying? If you pass the value directly, Go copies the entire house (all the data). If you pass a pointer, Go copies just the address (a tiny 8-byte piece of paper). Both copies are independent, but when two pointers hold the same address, they both point to the same house.

Think of it like a photocopy machine. If you photocopy a 500-page document, the copy is completely separate from the original — editing the copy doesn't change the original. But if you photocopy a sticky note that says "the document is in drawer #7," both the original sticky note and the copy point to the same drawer. Changing what's in drawer #7 affects everyone who has the address.

package main  // entry point

import "fmt"  // for printing

// Vendor represents a vendor in our system
type Vendor struct {
    Name      string  // vendor's company name
    RiskScore int     // their risk rating (0 = safe, 100 = dangerous)
}

// tryToUpdateByValue receives a COPY of the Vendor
// Analogy: photocopying the entire house blueprint, then modifying the copy
// The original house is completely unaffected
func tryToUpdateByValue(v Vendor) {
    v.RiskScore = 99  // modifying the COPY — the original is untouched
    fmt.Println("Inside function (value):", v.RiskScore)  // prints 99
}

// updateByPointer receives the ADDRESS of the Vendor
// Analogy: receiving the house address, going there, and actually repainting
// The original house IS affected because we're modifying it directly
func updateByPointer(v *Vendor) {
    v.RiskScore = 99  // modifying the ORIGINAL through the pointer
                       // Go automatically dereferences struct pointers — no need to write (*v).RiskScore
    fmt.Println("Inside function (pointer):", v.RiskScore)  // prints 99
}

func main() {
    acme := Vendor{Name: "Acme Corp", RiskScore: 42}  // create a vendor with risk score 42

    // Pass by VALUE — sends a copy
    tryToUpdateByValue(acme)                          // the function gets its own copy
    fmt.Println("After value call:", acme.RiskScore)  // prints 42 — original unchanged!

    // Pass by POINTER — sends the address
    updateByPointer(&acme)                             // &acme = "here's the address of acme"
    fmt.Println("After pointer call:", acme.RiskScore)  // prints 99 — original was changed!
}

Why Go Uses Pointers: Three Real Reasons

Now that you understand the mechanics, let's talk about when and why you'd use pointers. There are exactly three scenarios where pointers are the right tool.

Reason 1: Mutation — When You Need to Change the Original

If a function or method needs to modify its receiver (the struct it's called on), it must use a pointer receiver. Without a pointer, the method operates on a copy, and the original struct never changes. This is the most common reason to use pointers.

// Deactivate needs to change the vendor's status
// It uses a pointer receiver (*Vendor) so it modifies the ORIGINAL vendor
// If we used (v Vendor) instead, it would deactivate a copy and the original stays active
func (v *Vendor) Deactivate() {
    v.Active = false  // changes the original vendor — pointer lets us reach the real data
    v.DeactivatedAt = time.Now()  // record when we deactivated
}

// Usage:
acme := &Vendor{Name: "Acme", Active: true}  // create a pointer to a new Vendor
acme.Deactivate()                               // this changes acme.Active to false
fmt.Println(acme.Active)                         // false — the original was mutated

Reason 2: Avoiding Large Copies — Efficiency

Imagine a struct with 50 fields, some of which contain large strings or slices. Every time you pass it by value, Go copies all 50 fields. Passing a pointer instead copies only 8 bytes (the address), regardless of how enormous the struct is. For small structs (under ~64 bytes), this doesn't matter — copying is fast. For large structs, pointers save real time and memory.

// VendorProfile is a large struct — imagine it has megabytes of data
type VendorProfile struct {
    Name          string    // vendor name
    Description   string    // long description text
    Certifications []string // list of security certifications
    AuditHistory  []Audit    // years of audit records
    Documents     []Document // uploaded security documents
    // ... imagine 30 more fields ...
}

// analyzeProfile takes a POINTER to avoid copying the entire huge struct
// Only 8 bytes (the address) are passed, not megabytes of profile data
func analyzeProfile(profile *VendorProfile) *Analysis {
    // We can read all fields through the pointer
    score := calculateScore(profile.Certifications)  // access fields normally
    return &Analysis{Score: score}                    // return a pointer to the result too
}

Reason 3: Representing "Not Set" — The nil Pointer

Every type in Go has a "zero value" — the default when you don't assign anything. For int it's 0, for string it's "", for bool it's false. But what if zero is a valid value? How do you distinguish "the temperature is 0 degrees" from "the temperature was never measured"? A pointer solves this: nil means "not set," and any non-nil pointer means "set to this value."

// AgentConfig holds settings for an AI agent
// Some settings are optional — nil means "use the default"
type AgentConfig struct {
    MaxRetries *int             // nil = "use default (3)" vs &0 = "explicitly zero retries"
    Timeout    *time.Duration    // nil = "no timeout" vs &(5 * time.Second) = "5 second timeout"
    Model      string            // "" works fine here — empty string means "use default model"
    Verbose    bool              // false is a fine default — no pointer needed
}

// intPtr is a helper function that returns a pointer to an int value
// You can't write &5 directly in Go — you need a variable or a helper like this
func intPtr(v int) *int {
    return &v  // v is a local variable; &v returns a pointer to it
                // Go is smart enough to keep v alive even after this function returns
}

func main() {
    // Config with explicit values: "I want exactly 5 retries"
    cfg1 := AgentConfig{
        MaxRetries: intPtr(5),  // set to 5 — this is not nil, it's a pointer to the value 5
    }

    // Config using defaults: "I don't care about retries, use whatever"
    cfg2 := AgentConfig{
        MaxRetries: nil,  // explicitly nil — "not set, use the default"
    }

    // When reading the config, check for nil before using the value
    if cfg1.MaxRetries != nil {              // check: was this value set?
        fmt.Println(*cfg1.MaxRetries)        // dereference to get the actual number: 5
    }

    if cfg2.MaxRetries == nil {              // check: is this unset?
        fmt.Println("using default retries") // yes — apply the default behavior
    }
}
Nil pointer panic — the most common Go crash
If you dereference a nil pointer (try to follow an address that doesn't exist), your program panics and crashes. This is like trying to drive to a house address that's blank — there's no house to visit. Always check if ptr != nil before using *ptr. In practice, this mostly comes up with optional struct fields and function return values.

Stack vs Heap: Where Does Data Live?

Your program has two main areas of memory: the stack and the heap. The stack is fast but temporary — like a whiteboard that gets erased when you leave the room (when the function returns). The heap is slower but persistent — like a filing cabinet that lasts until someone cleans it out (when the garbage collector runs).

Go's compiler performs escape analysis to decide where to put each variable. If a variable never leaves the function (no pointer to it escapes), it goes on the fast stack. If a pointer to the variable is returned or stored elsewhere, the variable "escapes" to the heap. You don't control this directly — Go decides for you. But understanding it helps you write faster code when performance matters.

// Stack-allocated: v never escapes this function
// When processLocal returns, v is instantly freed — no garbage collector needed
// This is like writing on a whiteboard that gets erased when you leave the room
func processLocal() {
    v := Vendor{Name: "Acme"}  // v lives on the stack — fast allocation
    fmt.Println(v.Name)         // use v, but don't share its address with anyone
    // v is freed automatically when this function returns
}

// Heap-allocated: v escapes because we return a pointer to it
// The caller needs v to survive after this function returns
// So Go puts v on the heap, where the garbage collector will eventually clean it up
// This is like putting a document in the filing cabinet instead of the whiteboard
func newVendor(name string) *Vendor {
    v := Vendor{Name: name}  // v is allocated on the heap because it escapes (returned as pointer)
    return &v                 // return the address of v — v must outlive this function
}

// You can see Go's escape analysis decisions with this command:
// go build -gcflags="-m" ./...
// It will tell you exactly which variables escape to the heap
When to use pointers vs values
Use pointers when: (1) You need to modify the original data, (2) The struct is large (more than ~64 bytes), (3) You need to represent "not set" with nil. Use values when: (1) The struct is small (like time.Time, which is a value type), (2) You want to guarantee immutability (the callee gets its own copy), (3) The data is a primitive type (int, string, bool). When in doubt, start with values. Switch to pointers only when you have a concrete reason.
Chapter 07

Generics (Go 1.18+)

The Vending Machine Analogy

Imagine you're designing a vending machine. Without generics, you'd need to build a separate machine for every product type: one for sodas, one for candy bars, one for chips. Each machine has the exact same mechanism (accept coin, turn crank, dispense item), but the code for each is completely duplicated. If you want to add a "buy two get one free" feature, you have to change three machines.

Generics let you build one vending machine that works with any product type. You define the mechanism once — "accept payment of type T, dispense item of type T" — and then plug in the specific type when you use it: VendingMachine[Soda], VendingMachine[Candy], VendingMachine[Chips]. The machine's logic is written once, but it's fully type-safe: a soda machine only accepts and dispenses sodas.

Go added generics in version 1.18 (March 2022). The Go team waited over a decade to add this feature, because they wanted a design that was simple enough to fit Go's philosophy of clarity and readability. The result uses square brackets [T any] to declare type parameters. Let's build up from simple to complex.

Generic Functions: Write Once, Use With Any Type

A generic function declares one or more type parameters in square brackets after the function name. Inside the function, you use these type parameters like regular types. When you call the function, Go usually figures out the concrete type automatically from the arguments you pass — this is called type inference.

package main  // entry point of our program

import "fmt"  // standard library for formatted output

// Filter is a generic function that works with ANY type T
// [T any] means: "T is a type parameter that can be any type"
// Think: a vending machine that can filter any product by a condition
// It takes a slice of T values and a test function, returns only the items that pass
func Filter[T any](items []T, test func(T) bool) []T {
    var result []T                   // create an empty slice to hold matching items
    for _, item := range items {     // loop through every item in the input slice
        if test(item) {              // call the test function — does this item pass?
            result = append(result, item)  // yes — add it to our results
        }
    }
    return result  // return all items that passed the test
}

// Vendor represents a vendor in our system
type Vendor struct {
    Name      string  // vendor's name
    RiskScore int     // risk rating: 0 (safe) to 100 (dangerous)
}

func main() {
    // Create a list of vendors with different risk scores
    vendors := []Vendor{
        {Name: "Acme Corp", RiskScore: 25},     // low risk
        {Name: "Globex Inc", RiskScore: 85},    // high risk
        {Name: "Initech", RiskScore: 60},       // medium risk
        {Name: "Umbrella Co", RiskScore: 95},   // very high risk
    }

    // Use Filter with Vendor type — Go infers T = Vendor from the arguments
    // We pass a function that returns true for vendors with risk score above 70
    highRisk := Filter(vendors, func(v Vendor) bool {
        return v.RiskScore > 70  // keep only vendors with risk score above 70
    })

    // highRisk now contains only Globex and Umbrella
    fmt.Println(highRisk)  // [{Globex Inc 85} {Umbrella Co 95}]

    // The same Filter function works with strings too!
    // T is now inferred as string — same machine, different product
    names := []string{"Alice", "Bob", "Charlie", "Dave"}  // a list of strings
    longNames := Filter(names, func(s string) bool {
        return len(s) > 4  // keep only names longer than 4 characters
    })
    fmt.Println(longNames)  // [Alice Charlie]

    // And with integers!
    numbers := []int{1, 2, 3, 4, 5, 6, 7, 8}  // a list of integers
    evens := Filter(numbers, func(n int) bool {
        return n%2 == 0  // keep only even numbers (divisible by 2)
    })
    fmt.Println(evens)  // [2 4 6 8]
}

Type Constraints: Limiting What Goes in the Machine

The any keyword means "this type parameter accepts literally any type." But sometimes you need to restrict what types are allowed. Going back to our vending machine analogy: if the machine needs to sort the products by weight, you need products that have a weight — not just "any" product. In Go, you restrict type parameters using constraints, which are defined as interfaces.

For example, if you want to write a Sum function that adds numbers together, you can't use [T any] because not every type supports the + operator (you can't add two structs together). You need a constraint that says "T must be a numeric type."

// Numeric is a custom type constraint
// It defines which types are allowed as type parameters
// Only types listed here (or types based on them) can be used
type Numeric interface {
    ~int | ~int8 | ~int16 | ~int32 | ~int64 |  // all integer types
    ~float32 | ~float64                              // all floating-point types
    // the | means "or" — T can be any of these types
    // the ~ means "or any named type based on these" (explained below)
}

// Sum adds up all values in a slice
// [T Numeric] means: T must satisfy the Numeric constraint
// You can't call Sum with strings or structs — only numeric types
func Sum[T Numeric](values []T) T {
    var total T                       // start with the zero value of T (0 for numbers)
    for _, v := range values {        // loop through every value
        total += v                    // add each value to the running total
                                      // this works because Numeric guarantees + is available
    }
    return total                      // return the sum
}

// Usage with different numeric types
intSum := Sum([]int{10, 20, 30})            // T = int, result: 60
floatSum := Sum([]float64{1.5, 2.7, 3.3})  // T = float64, result: 7.5

// Sum([]string{"a", "b"})  // COMPILE ERROR: string doesn't satisfy Numeric

The ~ Tilde: Including Named Types

Go lets you create named types based on built-in types. For example, type RiskScore int creates a new type called RiskScore whose underlying type is int. Without the ~ (tilde) in your constraint, only the exact built-in type matches. With ~, any named type based on that type also matches. Think of ~int as "int and all of int's children."

// RiskScore is a named type — its underlying type is int
// It's still a number, but Go considers it a different type from plain int
// This is useful for type safety: you can't accidentally mix risk scores with ages
type RiskScore int

// Without ~: only accepts the exact type 'int', NOT 'RiskScore'
type StrictNumeric interface {
    int | float64  // ONLY int and float64 — nothing else
}

// With ~: accepts int, RiskScore, and ANY type whose underlying type is int
type FlexNumeric interface {
    ~int | ~float64  // int and its children, float64 and its children
}

// This demonstrates the difference:
scores := []RiskScore{85, 92, 78}  // a slice of RiskScore values

// Sum[FlexNumeric] works because ~int matches RiskScore
total := Sum(scores)  // Go infers T = RiskScore, which satisfies ~int
fmt.Println(total)    // 255

// If Sum used StrictNumeric (without ~), this would be a compile error
// because RiskScore is not literally 'int' — it's a named type based on int

Generic Structs: Type-Safe Containers

Generics aren't just for functions — you can create generic structs too. This is incredibly useful for building type-safe containers, results, and wrappers. Think of a generic struct as a labeled box: Result[Vendor] is a box that holds vendor data, Result[Report] is a box that holds report data. The box design is the same, but the contents are type-checked.

// Result is a generic struct that wraps any operation's output
// T is the type of the data this result contains
// Think of it as a shipping box: same box design, different contents
type Result[T any] struct {
    Data    T              // the actual result data — its type depends on T
    Error   error          // any error that occurred
    Elapsed time.Duration  // how long the operation took
}

// NewResult creates a successful Result with the given data
// [T any] makes this function generic too — it matches the struct's type parameter
func NewResult[T any](data T, elapsed time.Duration) Result[T] {
    return Result[T]{       // create a Result with the concrete type T
        Data:    data,      // store the data
        Elapsed: elapsed,   // store how long it took
    }
}

// NewErrorResult creates a failed Result
func NewErrorResult[T any](err error, elapsed time.Duration) Result[T] {
    return Result[T]{       // create a Result that represents failure
        Error:   err,       // store the error
        Elapsed: elapsed,   // store how long we tried before failing
    }
}

// Usage: each Result is fully type-safe
scoreResult := NewResult[int](85, 200*time.Millisecond)          // Result[int]
vendorResult := NewResult[Vendor](acme, 2*time.Second)            // Result[Vendor]
fmt.Println(scoreResult.Data)   // 85 — Data is of type int
fmt.Println(vendorResult.Data)  // {Acme Corp 25} — Data is of type Vendor

Constraints with Methods: The Validatable Pattern

You can also create constraints that require types to have specific methods. This combines the power of interfaces with generics. If your constraint says "T must have a Validate() method," then inside your generic function, you can call .Validate() on any value of type T.

// Validatable is a constraint requiring types to have a Validate method
// Any type that has Validate() error can be used where Validatable is required
type Validatable interface {
    Validate() error  // the type must have this method
}

// ValidateAll checks every item in a slice by calling its Validate method
// [T Validatable] means T must have a Validate() error method
func ValidateAll[T Validatable](items []T) error {
    for i, item := range items {    // loop through each item with its index
        if err := item.Validate(); err != nil {  // call Validate — the constraint guarantees this method exists
            return fmt.Errorf("item %d invalid: %w", i, err)  // report which item failed
        }
    }
    return nil  // all items passed validation
}

// Vendor satisfies the Validatable constraint because it has Validate() error
func (v Vendor) Validate() error {
    if v.Name == "" {  // a vendor must have a name
        return fmt.Errorf("vendor name cannot be empty")  // return an error
    }
    return nil  // validation passed
}

// Now ValidateAll works with []Vendor because Vendor has Validate()
err := ValidateAll(vendors)  // checks every vendor in the slice
When to use generics
Use generics for data structure operations (filter, map, sort), type-safe containers (Result[T], Pool[T]), and algorithm implementations that need to work with multiple types. Don't use generics when a concrete type or a plain interface would be clearer. The Go community values simplicity — write the straightforward version first and only reach for generics when you find yourself duplicating the same logic for different types.

Concurrency vs Parallelism: The Restaurant Kitchen

Before we write any concurrent code, let's make sure we understand two terms that are often confused. Imagine a restaurant kitchen:

Concurrency is about structure. It means your program is designed to handle multiple tasks. A single chef who chops vegetables, then checks the oven, then stirs a sauce, then goes back to chopping — that's concurrency. The chef is only doing one thing at any instant, but the kitchen is structured to handle multiple dishes. The chef interleaves tasks, switching between them efficiently.

Parallelism is about execution. It means multiple tasks are literally happening at the same instant. Three chefs each working on a different dish simultaneously — that's parallelism. You need multiple "workers" (CPU cores, chefs) for true parallelism.

Go makes concurrency easy with goroutines (lightweight concurrent tasks) and lets the runtime handle parallelism automatically based on how many CPU cores you have. You write concurrent code; Go decides how much of it actually runs in parallel. This is a key distinction: you structure your program around concurrent tasks, and Go's scheduler handles the rest.

CONCURRENCY (one chef, many dishes): Chef: [chop veg] → [check oven] → [stir sauce] → [plate dish] → [chop veg] → ... ─────────────────────────── time ──────────────────────────────▶ Only ONE task at any instant, but progress on ALL dishes PARALLELISM (many chefs, many dishes): Chef 1: [chop vegetables ─────────────────────] Chef 2: [prepare sauce ───────────────────────] Chef 3: [bake bread ─────────────────────────] ─────────────────────────── time ──────────────────────────────▶ Multiple tasks happening at the SAME instant

What Is a Goroutine?

A goroutine is Go's version of a lightweight thread. Think of it as hiring a new chef in your kitchen. Each chef works independently, can handle their own tasks, and communicates with other chefs through a specific system (channels, which we'll cover next). But unlike real chefs (or operating system threads), goroutines are incredibly cheap:

  • A goroutine starts with only ~2 KB of memory (an OS thread uses ~1 MB — that's 500x more)
  • You can create millions of goroutines on a typical machine
  • Starting a goroutine takes microseconds, not milliseconds
  • Go's runtime scheduler automatically manages which goroutines run on which CPU cores

Starting a goroutine is the simplest thing in Go: just put the keyword go before any function call. That's it. No thread pools to configure, no executor services to set up, no callback chains to wire together. One keyword.

package main  // entry point of our program

import (
    "fmt"   // for printing output
    "time"  // for time.Sleep to see goroutines in action
)

// assessVendor simulates running a risk assessment on a vendor
// In a real system, this might call an LLM API or analyze documents
func assessVendor(name string) {
    fmt.Printf("Starting assessment for %s\n", name)  // announce we're starting
    time.Sleep(2 * time.Second)                         // simulate work taking 2 seconds
    fmt.Printf("Finished assessment for %s\n", name)   // announce we're done
}

func main() {
    // WITHOUT goroutines: sequential execution (one chef doing everything)
    // Each assessment waits for the previous one to finish
    // Total time: 2 + 2 + 2 = 6 seconds

    // WITH goroutines: concurrent execution (multiple chefs)
    // The "go" keyword launches each assessment as a separate goroutine
    go assessVendor("Acme Corp")      // hire chef #1 — starts immediately, runs concurrently
    go assessVendor("Globex Inc")     // hire chef #2 — also starts immediately
    go assessVendor("Umbrella Co")    // hire chef #3 — also starts immediately
    // All three are now running concurrently!
    // Total time: ~2 seconds (they all work in parallel)

    // BUT WAIT — there's a critical trap here!
    // main() doesn't wait for goroutines to finish.
    // If main() returns, ALL goroutines are killed immediately.
    // It's like closing the restaurant while the chefs are still cooking.

    // This Sleep is a TEMPORARY hack to keep main alive.
    // In real code, you'd use channels or sync.WaitGroup (covered below)
    time.Sleep(3 * time.Second)  // wait 3 seconds so goroutines can finish
    fmt.Println("All assessments complete")  // this prints after the sleep
}
The "main exits, goroutines die" trap
This is the #1 mistake every Go beginner makes. When the main() function returns, the entire program exits immediately, killing all running goroutines. It doesn't matter if a goroutine is halfway through writing to a database or waiting for an API response — it gets killed without ceremony. You must use channels, sync.WaitGroup, or another synchronization mechanism to wait for goroutines to finish. Never use time.Sleep in production code for this — it's a race condition waiting to happen.

Goroutines with Anonymous Functions

You'll often launch goroutines with anonymous functions (functions without a name, defined inline). This is handy for short tasks or when you need to capture variables from the surrounding scope. But there's a subtlety with loop variables that you need to understand.

package main  // entry point

import (
    "fmt"   // for printing
    "sync"  // for WaitGroup — proper goroutine synchronization
)

func main() {
    // A list of vendor names to assess
    vendors := []string{"Acme", "Globex", "Initech", "Umbrella"}

    // sync.WaitGroup lets us wait for all goroutines to finish
    // Think of it as a checklist: we check off each task as it completes
    var wg sync.WaitGroup  // create a WaitGroup (counter starts at 0)

    for _, name := range vendors {  // loop through each vendor name
        wg.Add(1)  // tell the WaitGroup: "one more task is starting" (counter: +1)

        // Launch a goroutine with an anonymous function
        // We pass 'name' as a parameter to avoid a common closure bug
        go func(vendorName string) {  // anonymous function that takes a string parameter
            defer wg.Done()  // when this function finishes, mark one task as done (counter: -1)
                              // defer ensures Done() runs even if the function panics
            fmt.Printf("Assessing vendor: %s\n", vendorName)  // print the vendor name
        }(name)  // pass the current value of 'name' to the goroutine
                  // this is crucial — it captures the VALUE, not the variable
    }

    // Wait for ALL goroutines to call Done()
    // This blocks until the counter reaches 0
    wg.Wait()  // like waiting at the restaurant until every dish is served

    fmt.Println("All vendors assessed!")  // this only prints after ALL goroutines finish
}

Channels: The Conveyor Belt Between Chefs

Goroutines are great for doing work concurrently, but they need a way to communicate. How does one chef tell another that the sauce is ready? How does a worker goroutine send its results back to the main function? The answer is channels.

Think of a channel as a conveyor belt between two chefs. One chef places an item on the belt (sends), and another chef picks it up on the other end (receives). The belt is type-safe: if it's a "string belt," you can only put strings on it. And it's synchronized: if the receiving chef isn't ready to pick up the item, the sending chef waits with their hand on the belt until someone grabs it.

Channels are Go's primary tool for safe communication between goroutines. The Go proverb says it best: "Don't communicate by sharing memory; share memory by communicating." Instead of having goroutines read and write shared variables (dangerous, requires locks), you send values through channels (safe, coordinated).

Let's start with the basics and build up our understanding piece by piece.

Creating and Using a Channel

package main  // entry point

import "fmt"  // for printing

func main() {
    // Create a channel that transports string values
    // make(chan TYPE) creates an UNBUFFERED channel (we'll explain this next)
    // Think: installing a conveyor belt that carries strings
    ch := make(chan string)  // ch is of type chan string — a channel of strings

    // Launch a goroutine that SENDS a value into the channel
    // Think: Chef #1 places a finished dish on the conveyor belt
    go func() {
        // The <- operator sends a value INTO a channel when channel is on the LEFT
        ch <- "vendor assessment complete"  // SEND: put this string into the channel
        // This line blocks until someone reads from ch
        // The sending chef stands at the belt, waiting for the receiver to grab the dish
    }()

    // RECEIVE a value FROM the channel
    // Think: Chef #2 picks up the dish from the conveyor belt
    // The <- operator receives FROM a channel when channel is on the RIGHT
    msg := <-ch  // RECEIVE: wait for a value and store it in msg
                  // This line blocks until someone sends to ch

    fmt.Println(msg)  // Output: "vendor assessment complete"
}

Unbuffered Channels: The Direct Handoff

An unbuffered channel (created with just make(chan T)) has no storage capacity. It's like handing a plate directly from one chef to another — both chefs must be present at the same time. The sender blocks until a receiver is ready, and the receiver blocks until a sender has data. This creates a synchronization point — you know that when the send completes, the receiver has the value.

package main  // entry point

import (
    "fmt"   // for printing
    "time"  // for simulating work with Sleep
)

func main() {
    // Unbuffered channel — no internal storage, direct handoff only
    // Analogy: a doorway between two rooms — only one plate can pass at a time
    // and both people must be at the doorway simultaneously
    ch := make(chan string)  // no second argument means unbuffered (capacity = 0)

    // Chef goroutine: prepares a dish and hands it through the doorway
    go func() {
        fmt.Println("Chef: Starting to cook...")  // chef starts working
        time.Sleep(2 * time.Second)                // simulate 2 seconds of cooking
        fmt.Println("Chef: Dish ready! Waiting for waiter...")  // food is done
        ch <- "grilled salmon"                    // SEND — blocks until receiver is ready
        fmt.Println("Chef: Waiter took the dish!")  // this prints AFTER someone receives
    }()

    // Waiter: waits at the doorway to pick up the dish
    fmt.Println("Waiter: Waiting for the chef...")  // waiter is ready
    dish := <-ch                                       // RECEIVE — blocks until chef sends
    fmt.Println("Waiter: Got", dish)                    // prints "Waiter: Got grilled salmon"

    // Output order:
    // Waiter: Waiting for the chef...
    // Chef: Starting to cook...
    // Chef: Dish ready! Waiting for waiter...
    // Chef: Waiter took the dish!
    // Waiter: Got grilled salmon
}

Buffered Channels: The Counter Where Plates Can Wait

A buffered channel has internal storage for a fixed number of values. Think of it as a kitchen counter that can hold several plates. Chefs can place dishes on the counter without waiting for a waiter, as long as there's space. The waiter picks up dishes whenever they're ready. Sends only block when the counter is full; receives only block when the counter is empty.

You create a buffered channel by specifying the buffer size: make(chan T, size).

package main  // entry point

import "fmt"  // for printing

func main() {
    // Buffered channel with capacity 3
    // Analogy: a kitchen counter with 3 plate slots
    // Chefs can place up to 3 dishes without waiting for a waiter
    ch := make(chan string, 3)  // buffered channel — can hold 3 values

    // These sends DON'T block because the buffer has room
    ch <- "dish 1"  // place on counter — buffer: [dish 1, _, _] — instant, no waiting
    ch <- "dish 2"  // place on counter — buffer: [dish 1, dish 2, _] — still room
    ch <- "dish 3"  // place on counter — buffer: [dish 1, dish 2, dish 3] — full now!

    // If we tried ch <- "dish 4" here, it would BLOCK (deadlock!)
    // because the buffer is full and no one is receiving

    // Receive values — first in, first out (FIFO order)
    fmt.Println(<-ch)  // "dish 1" — first dish placed is first dish picked up
    fmt.Println(<-ch)  // "dish 2" — second dish
    fmt.Println(<-ch)  // "dish 3" — third dish

    // Practical example: collecting results from concurrent assessments
    // Buffer size = number of vendors we're assessing
    results := make(chan string, 3)  // room for 3 results — one per vendor

    // Launch 3 concurrent assessments — each sends its result to the channel
    go func() { results <- "Acme: low risk" }()      // goroutine 1 sends result
    go func() { results <- "Globex: high risk" }()    // goroutine 2 sends result
    go func() { results <- "Initech: medium risk" }()  // goroutine 3 sends result

    // Collect all 3 results — we know exactly how many to expect
    for i := 0; i < 3; i++ {  // loop 3 times to receive 3 results
        fmt.Println(<-results)  // receive one result each iteration
                                 // order may vary — goroutines run concurrently
    }
}
Unbuffered vs Buffered — when to use which
Unbuffered (make(chan T)): Use when you want a guaranteed handoff — the sender knows the receiver got the value. Good for signaling and synchronization. Think: direct hand-to-hand pass.

Buffered (make(chan T, n)): Use when the sender and receiver work at different speeds, or when you want to decouple their timing. Good for collecting results from multiple goroutines. Think: kitchen counter with limited space.

When in doubt, start with unbuffered. Add a buffer only when you have a specific reason (usually to avoid blocking a producer while the consumer is temporarily slow).

Channel Direction: One-Way Streets

You can restrict a channel to be send-only or receive-only in function parameters. This is a compile-time safety net: it prevents a function from accidentally doing the wrong operation on a channel. A producer function that should only send values can't accidentally read from its output channel.

// producer only SENDS to the channel — declared as chan<- (arrow points INTO channel)
// Think: this chef can only PUT dishes on the belt, not take them off
func producer(out chan<- string) {
    out <- "vendor report ready"  // SEND — allowed, this is a send-only channel
    // msg := <-out  // COMPILE ERROR! Cannot receive from a send-only channel
}

// consumer only RECEIVES from the channel — declared as <-chan (arrow points OUT of channel)
// Think: this chef can only PICK UP dishes from the belt, not put them on
func consumer(in <-chan string) {
    msg := <-in  // RECEIVE — allowed, this is a receive-only channel
    fmt.Println("Received:", msg)  // print what we got
    // in <- "reply"  // COMPILE ERROR! Cannot send to a receive-only channel
}

func main() {
    // Create a regular bidirectional channel
    ch := make(chan string)  // both send and receive are allowed on this channel

    // Go automatically converts bidirectional channels to directional ones
    // when you pass them to functions with directional parameters
    go producer(ch)  // ch is automatically narrowed to chan<- string (send-only)
    consumer(ch)     // ch is automatically narrowed to <-chan string (receive-only)
}

Range Over Channel and close()

You can iterate over a channel with for range, which keeps receiving values until the channel is closed. The sender calls close(ch) to signal "I'm done, no more values are coming." This is the idiomatic way to build producer-consumer patterns.

Think of it like a bakery with a "now serving" window. The baker puts bread on the counter (sends to channel), and when the last loaf is done, flips the "CLOSED" sign (calls close()). Customers keep taking bread (ranging over the channel) until they see the closed sign.

package main  // entry point

import "fmt"  // for printing

// generateVendorIDs creates a channel and sends vendor IDs into it
// It returns a receive-only channel — the caller can only read from it
func generateVendorIDs(ids []string) <-chan string {
    ch := make(chan string)  // create an unbuffered channel

    // Launch a goroutine that sends all IDs and then closes the channel
    go func() {
        defer close(ch)  // IMPORTANT: close the channel when we're done sending
                          // defer ensures close runs even if something panics
                          // this tells the receiver "no more data is coming"

        for _, id := range ids {  // loop through each vendor ID
            ch <- id              // send each ID into the channel
        }
    }()

    return ch  // return the channel immediately — data will flow through it as the goroutine runs
}

func main() {
    // Create some vendor IDs to process
    ids := []string{"V-001", "V-002", "V-003", "V-004"}

    // Range over the channel — this receives values until the channel is closed
    // It's like standing at the bakery window, taking bread until the "CLOSED" sign appears
    for id := range generateVendorIDs(ids) {  // receives one ID each iteration
        fmt.Println("Processing vendor:", id)    // process each vendor ID
    }
    // The loop exits automatically when the channel is closed
    // If we forgot to close(ch), this loop would block forever — deadlock!

    fmt.Println("All vendors processed")  // we get here after all IDs are consumed
}

Checking if a Channel Is Closed: The Two-Value Receive

When you receive from a channel, you can use a second return value to check whether the channel is still open or has been closed:

// Two-value receive: value + ok flag
// ok is true if the value came from a send, false if the channel is closed
value, ok := <-ch  // receive with closed-check

if ok {
    fmt.Println("Received:", value)  // channel is open, we got real data
} else {
    fmt.Println("Channel closed")    // channel was closed, value is the zero value
}

Common Deadlock Scenarios

A deadlock happens when all goroutines are stuck waiting and none can proceed. It's like two chefs standing at a doorway, each waiting for the other to go first — nobody moves. Go's runtime detects deadlocks and panics with: "fatal error: all goroutines are asleep - deadlock!". Here are the three most common traps:

// ══════════════════════════════════════
// DEADLOCK #1: Sending with no receiver
// ══════════════════════════════════════
func deadlock1() {
    ch := make(chan int)  // create an unbuffered channel
    ch <- 42              // DEADLOCK! This blocks forever because no goroutine is receiving
    // Both the sender (main) and... there IS no receiver. Everyone is stuck.
    // Fix: launch a goroutine to receive, or make the channel buffered
}

// ══════════════════════════════════════════════
// DEADLOCK #2: Forgetting to close a ranged channel
// ══════════════════════════════════════════════
func deadlock2() {
    ch := make(chan int)  // create a channel
    go func() {
        ch <- 1           // send first value
        ch <- 2           // send second value
        // OOPS — forgot to close(ch)!
    }()
    for v := range ch {  // range waits for close() that never comes
        fmt.Println(v)    // prints 1, then 2, then hangs forever
    }
    // Fix: add 'defer close(ch)' in the goroutine
}

// ══════════════════════════════════════════════
// DEADLOCK #3: Circular dependency between channels
// ══════════════════════════════════════════════
func deadlock3() {
    a := make(chan int)  // channel a
    b := make(chan int)  // channel b
    go func() {
        a <- <-b  // waits to receive from b, then sends to a
    }()
    b <- <-a      // waits to receive from a, then sends to b
    // goroutine waits for b, main waits for a — circular wait = deadlock!
    // Fix: break the circular dependency — one side must not depend on the other
}
Deadlock prevention rules
  1. Always have a receiver for every unbuffered send, running in a separate goroutine.
  2. Always close channels when the sender is done, especially if a range loop is consuming.
  3. Use select with timeouts to avoid blocking forever (covered in Chapter 09).
  4. Run with -race flag: go test -race -timeout 10s catches many concurrency bugs.

Putting It All Together: A Complete Concurrent Example

Let's build a practical example that uses everything we've learned: goroutines, channels, buffered channels, closing, and range. We'll create a system that assesses multiple vendors concurrently and collects their results. This is a simplified version of what a real multi-agent system might do.

package main  // entry point of our program

import (
    "fmt"    // for formatted output
    "math/rand"  // for generating random risk scores
    "sync"   // for WaitGroup to coordinate goroutines
    "time"   // for simulating work with Sleep
)

// AssessmentResult holds the outcome of a vendor risk assessment
type AssessmentResult struct {
    VendorName string  // which vendor was assessed
    RiskScore  int     // their calculated risk score (0-100)
    Duration   time.Duration  // how long the assessment took
}

// assessVendor simulates running a risk assessment on a single vendor
// In a real system, this might call an LLM API, scrape public data, etc.
func assessVendor(name string) AssessmentResult {
    start := time.Now()                // record when we started
    sleepTime := time.Duration(rand.Intn(3)+1) * time.Second  // random 1-3 second delay
    time.Sleep(sleepTime)              // simulate the work taking time
    score := rand.Intn(100)            // generate a random risk score 0-99
    return AssessmentResult{           // return the result
        VendorName: name,               // which vendor
        RiskScore:  score,              // their risk score
        Duration:   time.Since(start),  // how long it took
    }
}

func main() {
    // List of vendors to assess
    vendors := []string{"Acme Corp", "Globex Inc", "Initech", "Umbrella Co", "Stark Industries"}

    // Create a buffered channel to collect results
    // Buffer size = number of vendors, so no goroutine blocks on send
    results := make(chan AssessmentResult, len(vendors))  // buffered: room for all results

    // WaitGroup to know when all assessments are done
    var wg sync.WaitGroup  // counter starts at 0

    // Launch concurrent assessments — one goroutine per vendor
    // Like hiring 5 chefs, each responsible for one dish
    for _, name := range vendors {  // loop through each vendor
        wg.Add(1)  // increment the counter: "one more task starting"
        go func(vendorName string) {  // launch a goroutine for this vendor
            defer wg.Done()  // when done, decrement counter: "one task finished"
            result := assessVendor(vendorName)  // run the assessment (takes 1-3 seconds)
            results <- result                   // send the result into the channel
        }(name)  // pass current vendor name to avoid closure bug
    }

    // Close the results channel once ALL goroutines finish
    // We do this in a separate goroutine so it doesn't block main
    go func() {
        wg.Wait()       // wait for all goroutines to call Done()
        close(results)  // signal: no more results will be sent
    }()

    // Collect and display all results
    // Range over the channel — this exits when close(results) is called
    fmt.Println("=== Vendor Risk Assessment Report ===")  // header
    for r := range results {  // receive results one by one until channel is closed
        riskLevel := "LOW"           // default risk level
        if r.RiskScore > 70 {        // check if high risk
            riskLevel = "HIGH"       // mark as high risk
        } else if r.RiskScore > 40 { // check if medium risk
            riskLevel = "MEDIUM"     // mark as medium risk
        }
        // Print formatted result for this vendor
        fmt.Printf("  %-20s Score: %3d  Risk: %-6s  Time: %v\n",
            r.VendorName,  // vendor name, left-aligned in 20 chars
            r.RiskScore,   // risk score, right-aligned in 3 digits
            riskLevel,     // risk level text
            r.Duration.Round(time.Millisecond),  // duration, rounded for readability
        )
    }
    fmt.Println("=== Assessment Complete ===")  // footer
}
Chapter 09

Select Statement

The Food Court Analogy

Imagine you're at a food court with three restaurants. You've ordered from all three, and now you're waiting. You don't know which restaurant will finish first — it depends on what you ordered and how busy they are. You want to grab the first dish that's ready, rather than waiting for a specific restaurant.

That's exactly what Go's select statement does. It waits on multiple channels simultaneously and proceeds with whichever one is ready first. If multiple channels are ready at the same instant, Go picks one at random (preventing starvation — no channel is systematically favored). If no channel is ready and there's a default case, select returns immediately. If no channel is ready and there's no default, select blocks until at least one becomes ready.

The select statement looks like a switch, but instead of comparing values, each case is a channel operation (send or receive). This is one of Go's most powerful features for building responsive concurrent systems, because it lets you react to multiple events — results arriving, timeouts expiring, cancellation signals — all in one clean construct.

Let's build up from simple to complex with detailed examples.

Basic Select: Waiting on Multiple Channels

package main  // entry point

import (
    "fmt"   // for printing
    "time"  // for simulating delays
)

// simulateAssessment pretends to assess a vendor and sends the result to a channel
func simulateAssessment(name string, delay time.Duration, ch chan<- string) {
    time.Sleep(delay)                                     // simulate work taking some time
    ch <- fmt.Sprintf("%s assessment complete", name)  // send the result when done
}

func main() {
    // Create two channels — one for each vendor assessment
    riskCh := make(chan string)        // channel for risk assessment results
    complianceCh := make(chan string)  // channel for compliance check results

    // Launch both assessments concurrently — like ordering from two restaurants
    go simulateAssessment("Risk", 2*time.Second, riskCh)            // risk takes 2 seconds
    go simulateAssessment("Compliance", 1*time.Second, complianceCh)  // compliance takes 1 second

    // select waits on BOTH channels — whichever finishes first wins
    // Like standing in the food court waiting for whichever restaurant finishes first
    select {
    case result := <-riskCh:        // if risk assessment finishes first...
        fmt.Println("First result:", result)  // handle it
    case result := <-complianceCh:  // if compliance check finishes first...
        fmt.Println("First result:", result)  // handle it
    }
    // Output: "First result: Compliance assessment complete"
    // because compliance (1s) finishes before risk (2s)
    // select only handles ONE case, then continues past the select block
}

Timeout Pattern: Don't Wait Forever

One of the most practical uses of select is implementing timeouts. If you're calling an external API (like an LLM) and it doesn't respond in time, you want to give up and move on rather than hanging forever. The time.After() function returns a channel that receives a value after the specified duration, which you can use as a timeout case in select.

package main  // entry point

import (
    "fmt"   // for printing
    "time"  // for timeout duration and time.After
)

// callLLM simulates calling an LLM API that might be slow
func callLLM(prompt string) chan string {
    ch := make(chan string, 1)  // buffered so the goroutine doesn't leak if we time out
    go func() {
        time.Sleep(5 * time.Second)  // simulate a slow API call (5 seconds)
        ch <- "LLM response: vendor appears high-risk"  // send the response
    }()
    return ch  // return the channel — result will arrive later
}

func main() {
    llmResult := callLLM("Analyze this vendor's security posture")  // start the LLM call

    // select with a timeout: wait for the LLM OR give up after 3 seconds
    select {
    case response := <-llmResult:  // if the LLM responds in time...
        fmt.Println("Got response:", response)  // use the response
    case <-time.After(3 * time.Second):  // if 3 seconds pass with no response...
        fmt.Println("Timeout! LLM took too long, using cached result")  // give up
    }
    // Output: "Timeout! LLM took too long, using cached result"
    // because the LLM takes 5s but our timeout is only 3s
}

Non-Blocking Check with default

Adding a default case makes the select non-blocking. If no channel is ready, the default case runs immediately. This is useful for polling or when you want to do something else while waiting.

package main  // entry point

import (
    "fmt"   // for printing
    "time"  // for Sleep
)

func main() {
    messages := make(chan string, 5)  // buffered channel for incoming messages

    // Simulate a message arriving after 2 seconds
    go func() {
        time.Sleep(2 * time.Second)   // wait 2 seconds
        messages <- "new vendor alert"  // then send a message
    }()

    // Poll for messages in a loop — check, do other work, check again
    for i := 0; i < 5; i++ {  // try 5 times
        select {
        case msg := <-messages:  // if a message is available...
            fmt.Println("Received:", msg)  // handle it
            return  // we're done
        default:  // if no message is available RIGHT NOW...
            fmt.Println("No messages yet, doing other work...")  // do something else
            time.Sleep(500 * time.Millisecond)  // wait a bit before checking again
        }
    }
    // Output:
    // No messages yet, doing other work...
    // No messages yet, doing other work...
    // No messages yet, doing other work...
    // No messages yet, doing other work...
    // Received: new vendor alert
}

Select in a Loop: The Agent Orchestrator

In real systems, you often put select inside a for loop to continuously handle events from multiple sources. This pattern is the backbone of event-driven systems like agent orchestrators. Think of it as a restaurant manager who continuously checks on multiple tables, handles whatever needs attention, and keeps going until the restaurant closes.

package main  // entry point

import (
    "context"  // for cancellation signals
    "fmt"      // for printing
    "time"     // for timing and tickers
)

func main() {
    // Create a context that automatically cancels after 5 seconds
    // This is like setting a timer: "close the restaurant in 5 seconds"
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()  // always call cancel to release resources

    // Create channels for different types of events
    alerts := make(chan string, 10)  // channel for incoming security alerts
    heartbeat := time.NewTicker(1 * time.Second)  // sends a tick every second
    defer heartbeat.Stop()  // clean up the ticker when we're done

    // Simulate an alert arriving after 2.5 seconds
    go func() {
        time.Sleep(2500 * time.Millisecond)  // wait 2.5 seconds
        alerts <- "Critical: vendor data breach detected!"  // send an alert
    }()

    // Event loop: handle multiple event types continuously
    // This runs until the context is cancelled (after 5 seconds)
    for {
        select {
        case alert := <-alerts:  // if a security alert arrives...
            fmt.Println("🚨 ALERT:", alert)  // handle it immediately

        case t := <-heartbeat.C:  // if the heartbeat ticker fires...
            fmt.Println("💓 Heartbeat at", t.Format("15:04:05"))  // log it

        case <-ctx.Done():  // if the context is cancelled (timeout or manual cancel)...
            fmt.Println("Shutting down — context cancelled:", ctx.Err())  // explain why
            return  // exit the loop and the function
        }
    }
}
select is the heartbeat of Go concurrency
The for { select { ... } } pattern is one of the most common in Go. You'll see it in HTTP servers, message processors, agent orchestrators, and event handlers. Think of it as the "event loop" of your concurrent system — it continuously reacts to whichever event happens next, and cleanly shuts down when told to stop via context cancellation.

Sync Primitives

Channels are Go's preferred tool for communication between goroutines. But some patterns are better handled with traditional synchronization primitives from the sync package. The Go proverb says: "Share memory by communicating [channels], don't communicate by sharing memory [mutexes]." But when you genuinely need shared state — like a concurrent cache, a thread-safe counter, or a one-time initialization — the sync package is the right tool.

Think of channels as a messaging system (passing notes between people) and sync primitives as physical locks and signs (bathroom occupied sign, event checklist). Different problems call for different tools.

WaitGroup: The Task Checklist

Imagine you're a project manager with a checklist of tasks. Before you can write the final report, every task must be checked off. You add tasks to the list (Add), team members check off tasks as they finish (Done), and you wait until the entire list is complete (Wait).

That's exactly what sync.WaitGroup does. It maintains an internal counter: Add(n) adds n tasks, Done() marks one task complete (decrements by 1), and Wait() blocks until the counter reaches zero. It's the most common way to wait for a group of goroutines to finish.

There's one critical rule: always call Add before launching the goroutine, and always call Done inside the goroutine using defer. If you call Add inside the goroutine, there's a race condition: Wait might return before the goroutine has even started and incremented the counter.

package main  // entry point

import (
    "fmt"   // for printing
    "sync"  // for WaitGroup
    "time"  // for simulating work
)

func main() {
    // Vendors to assess — each gets its own goroutine
    vendors := []string{"Acme", "Globex", "Initech", "Umbrella"}

    var wg sync.WaitGroup  // create a WaitGroup — our task checklist (counter starts at 0)

    for _, name := range vendors {  // loop through each vendor
        wg.Add(1)  // BEFORE launching the goroutine, add 1 to the checklist
                    // counter is now: 1, then 2, then 3, then 4

        go func(vendorName string) {  // launch a goroutine for this vendor
            defer wg.Done()  // WHEN DONE: subtract 1 from the checklist
                              // defer ensures this runs even if the function panics

            fmt.Printf("Starting assessment: %s\n", vendorName)  // announce start
            time.Sleep(1 * time.Second)  // simulate work
            fmt.Printf("Completed assessment: %s\n", vendorName)  // announce completion
        }(name)  // pass name by value to the goroutine
    }

    fmt.Println("Waiting for all assessments to complete...")  // this prints immediately
    wg.Wait()  // BLOCK here until the counter reaches 0
                // like sitting at your desk until every task on the checklist is checked off
    fmt.Println("All assessments complete!")  // only prints after ALL goroutines finish
}

Mutex: The Bathroom Lock

A sync.Mutex provides mutual exclusion — only one goroutine can hold the lock at a time. Think of it as a bathroom lock: when someone is inside (holding the lock), everyone else has to wait in line. When they leave (unlock), the next person can enter.

You need a mutex when multiple goroutines read and write the same shared data. Without a mutex, two goroutines might try to update a map at the same time, causing a panic. The idiomatic Go pattern is: lock right before accessing shared data, and defer the unlock so it always happens.

package main  // entry point

import (
    "fmt"   // for printing
    "sync"  // for Mutex and WaitGroup
)

// SafeCounter is a counter that can be safely used from multiple goroutines
// The mutex field protects the counts map from concurrent access
type SafeCounter struct {
    mu     sync.Mutex            // the lock — like a bathroom door lock
    counts map[string]int  // the shared data we're protecting
}

// Increment safely adds 1 to the count for a given key
// Multiple goroutines can call this concurrently — the mutex keeps it safe
func (c *SafeCounter) Increment(key string) {
    c.mu.Lock()          // LOCK: "I'm going in the bathroom, nobody else enter"
    defer c.mu.Unlock()  // UNLOCK when this function returns: "I'm done, next person can go"
                          // defer guarantees unlock even if the code below panics
    c.counts[key]++      // safely modify the shared map — we have exclusive access
}

// Get safely reads the count for a given key
func (c *SafeCounter) Get(key string) int {
    c.mu.Lock()          // even reads need the lock to prevent reading mid-write
    defer c.mu.Unlock()  // release the lock when done reading
    return c.counts[key] // return the current count
}

func main() {
    counter := &SafeCounter{                          // create a new safe counter
        counts: make(map[string]int),  // initialize the map
    }

    var wg sync.WaitGroup  // to wait for all goroutines

    // Launch 1000 goroutines that all increment the same counter
    // Without the mutex, this would cause a race condition and crash
    for i := 0; i < 1000; i++ {  // launch 1000 concurrent increments
        wg.Add(1)                  // add to the checklist
        go func() {
            defer wg.Done()        // mark this goroutine as done when finished
            counter.Increment("vendor-assessments")  // safely increment
        }()
    }

    wg.Wait()  // wait for all 1000 goroutines to finish
    fmt.Println("Total:", counter.Get("vendor-assessments"))  // prints: Total: 1000
    // Without the mutex, this might print 987, 993, or even crash!
}

RWMutex: Multiple Readers, One Writer

A regular Mutex is like a single-stall bathroom — only one person at a time, regardless of what they're doing. But in many systems, reads are much more frequent than writes. A sync.RWMutex (Read-Write Mutex) is like a library reading room: many people can read books simultaneously, but when someone needs to rearrange the shelves (write), everyone else must leave until they're done.

// ScoreBoard stores vendor risk scores and is safe for concurrent access
// Uses RWMutex: multiple goroutines can read simultaneously,
// but only one can write at a time
type ScoreBoard struct {
    mu     sync.RWMutex           // read-write lock: allows concurrent reads
    scores map[string]int  // vendor name → risk score
}

// Set updates a vendor's score — needs exclusive (write) access
func (s *ScoreBoard) Set(vendor string, score int) {
    s.mu.Lock()          // WRITE LOCK: "everyone out, I'm rearranging shelves"
    defer s.mu.Unlock()  // release write lock when done
    s.scores[vendor] = score  // safely write to the map
}

// Get reads a vendor's score — multiple goroutines can read simultaneously
func (s *ScoreBoard) Get(vendor string) (int, bool) {
    s.mu.RLock()          // READ LOCK: "I'm reading, others can read too"
    defer s.mu.RUnlock()  // release read lock when done
    score, ok := s.scores[vendor]  // safely read from the map
    return score, ok       // return the score and whether it was found
}

// Usage: many readers can access Get() simultaneously
// but Set() blocks all readers until the write is done

sync.Once: Do It Exactly Once

sync.Once guarantees that a function is executed exactly once, no matter how many goroutines try to trigger it. Think of it as a grand opening ribbon — only the first person to arrive cuts it, and everyone else walks through the already-opened door.

This is perfect for lazy initialization: creating an expensive resource (like a database connection or API client) only when it's first needed, and ensuring it's only created once even if 100 goroutines try to use it simultaneously.

package main  // entry point

import (
    "fmt"   // for printing
    "sync"  // for Once
)

var (
    initOnce  sync.Once      // ensures the init function runs exactly once
    dbClient  *DatabaseClient // the shared client — nil until initialized
)

// DatabaseClient represents an expensive-to-create database connection
type DatabaseClient struct {
    connected bool  // whether we successfully connected
}

// GetDB returns the database client, initializing it on first call
// No matter how many goroutines call this simultaneously,
// the expensive initialization only happens ONCE
func GetDB() *DatabaseClient {
    initOnce.Do(func() {  // this function runs EXACTLY ONCE, ever
        fmt.Println("Initializing database connection...")  // expensive setup
        dbClient = &DatabaseClient{connected: true}        // create the client
        fmt.Println("Database ready!")                        // initialization complete
    })
    return dbClient  // return the (possibly just-created) client
}

func main() {
    var wg sync.WaitGroup  // to wait for all goroutines

    // Launch 10 goroutines that all try to get the database
    for i := 0; i < 10; i++ {  // 10 concurrent requests
        wg.Add(1)               // add to checklist
        go func(id int) {
            defer wg.Done()     // mark done when finished
            db := GetDB()       // get the database client
            fmt.Printf("Goroutine %d: connected=%v\n", id, db.connected)  // use it
        }(i)
    }

    wg.Wait()  // wait for all goroutines
    // Output: "Initializing database connection..." appears ONLY ONCE
    // All 10 goroutines share the same dbClient
}

sync.Map: The Built-in Concurrent Map

sync.Map is a thread-safe map provided by the standard library. It's optimized for two specific patterns: (1) keys that are written once and read many times (like caches), and (2) keys that are disjoint across goroutines. For most other cases, a regular map protected by sync.RWMutex performs better and gives you type safety.

package main  // entry point

import (
    "fmt"   // for printing
    "sync"  // for sync.Map
)

func main() {
    var cache sync.Map  // create a concurrent map — no initialization needed

    // Store a value — like putting something in a concurrent cache
    // Note: keys and values are type 'any' — you lose compile-time type safety
    cache.Store("acme-score", 85)  // store key="acme-score", value=85

    // Load a value — need a type assertion to get the concrete type back
    value, ok := cache.Load("acme-score")  // ok is true if the key exists
    if ok {
        score := value.(int)  // type assertion: convert from 'any' back to int
        fmt.Println("Score:", score)  // Score: 85
    }

    // LoadOrStore: return existing value, or store the new one
    // Useful for "get or create" patterns
    actual, loaded := cache.LoadOrStore("acme-score", 90)  // key exists, returns old value
    fmt.Println(actual, loaded)  // 85 true — "loaded" means the key already existed

    actual, loaded = cache.LoadOrStore("new-key", 42)  // key doesn't exist, stores 42
    fmt.Println(actual, loaded)  // 42 false — "not loaded" means we stored a new value

    // Delete a key
    cache.Delete("acme-score")  // remove the entry
}
Channels vs Mutexes: when to use which
Channels: Use when you're passing ownership of data between goroutines, building pipelines, or signaling events. Channels express the flow of data.

Mutexes: Use when multiple goroutines need to read/write the same data structure (like a shared cache or counter). Mutexes protect state.

Go proverb: "Share memory by communicating, don't communicate by sharing memory." When in doubt, start with channels.

Concurrency Patterns

Fan-Out / Fan-In: The Hiring Pattern

Imagine you run a company that needs to review 100 vendor security questionnaires. If you assign one person to review all 100, it takes forever. So you fan out: you hire 5 reviewers and distribute the questionnaires evenly (20 each). Each reviewer works independently. When they finish, their completed reviews flow back through a fan in: a single collection point where all results are gathered.

This is the fan-out/fan-in pattern, and it's the foundation of concurrent processing in Go. "Fan-out" means splitting work across multiple goroutines (workers). "Fan-in" means collecting all their results into a single channel. The channel between them acts as a natural work queue with built-in backpressure — if workers are slow, the job sender automatically waits.

This pattern is everywhere in real systems: parallel HTTP requests, concurrent database queries, batch processing, and of course, running multiple AI agent assessments simultaneously. Let's build it step by step.

package main  // entry point

import (
    "fmt"   // for printing
    "sync"  // for WaitGroup
    "time"  // for simulating work
)

// Vendor represents a company we need to assess
type Vendor struct {
    Name string  // vendor's company name
}

// Result holds the output of a single vendor assessment
type Result struct {
    VendorName string  // which vendor was assessed
    RiskScore  int     // their risk score
    WorkerID   int     // which worker handled this assessment
}

// assessVendor simulates a risk assessment (1-2 seconds of work)
func assessVendor(v Vendor) int {
    time.Sleep(500 * time.Millisecond)  // simulate processing time
    return len(v.Name) * 7              // fake score based on name length
}

// fanOutFanIn distributes vendor assessments across numWorkers goroutines
// and collects all results
func fanOutFanIn(vendors []Vendor, numWorkers int) []Result {
    // Create the JOBS channel — this is the work queue
    // Vendors go IN here, workers pull them OUT
    // Buffer size = total vendors so the sender doesn't block
    jobs := make(chan Vendor, len(vendors))  // buffered channel for incoming work

    // Create the RESULTS channel — this is where completed work flows back
    // Workers put results IN here, the collector pulls them OUT
    results := make(chan Result, len(vendors))  // buffered channel for outgoing results

    // ── FAN-OUT: Start worker goroutines ──
    // Each worker is like a reviewer who grabs questionnaires from a shared pile
    var wg sync.WaitGroup  // tracks when all workers are done
    for w := 0; w < numWorkers; w++ {  // create numWorkers workers
        wg.Add(1)  // register this worker on the checklist
        go func(workerID int) {  // launch worker goroutine
            defer wg.Done()  // check off this worker when it finishes
            for vendor := range jobs {  // pull vendors from the jobs channel until it closes
                score := assessVendor(vendor)  // do the actual assessment work
                results <- Result{              // send the result to the results channel
                    VendorName: vendor.Name,     // which vendor
                    RiskScore:  score,           // their score
                    WorkerID:   workerID,        // which worker did it
                }
            }
        }(w)  // pass worker ID to the goroutine
    }

    // ── SEND ALL JOBS ──
    // Put all vendors into the jobs channel, then close it
    // Closing signals workers that no more work is coming
    for _, v := range vendors {  // loop through all vendors
        jobs <- v  // send each vendor to the jobs channel
    }
    close(jobs)  // signal: no more vendors to process

    // ── FAN-IN: Close results when all workers finish ──
    go func() {
        wg.Wait()       // wait until all workers have called Done()
        close(results)  // then close the results channel — no more results coming
    }()

    // ── COLLECT ALL RESULTS ──
    var allResults []Result           // slice to hold all results
    for r := range results {         // receive results until the channel is closed
        allResults = append(allResults, r)  // add each result to our collection
    }
    return allResults  // return all collected results
}

func main() {
    vendors := []Vendor{
        {Name: "Acme Corp"}, {Name: "Globex Inc"}, {Name: "Initech"},
        {Name: "Umbrella Co"}, {Name: "Stark Industries"}, {Name: "Wayne Enterprises"},
    }

    fmt.Println("Assessing 6 vendors with 3 workers...")  // announce the plan
    start := time.Now()  // record start time

    results := fanOutFanIn(vendors, 3)  // assess with 3 concurrent workers

    for _, r := range results {  // print each result
        fmt.Printf("  Worker %d assessed %-20s → Score: %d\n", r.WorkerID, r.VendorName, r.RiskScore)
    }
    fmt.Printf("Total time: %v\n", time.Since(start).Round(time.Millisecond))  // total elapsed time
    // With 3 workers and 6 vendors at ~500ms each: ~1 second total (not 3 seconds)
}

Pipeline Pattern: The Assembly Line

A pipeline is a series of stages connected by channels, where each stage processes data and passes it to the next. Think of it as an assembly line in a factory: raw materials enter stage 1, get partially processed and sent to stage 2, then stage 3, and so on. The beauty is that all stages run concurrently — while stage 2 processes item A, stage 1 is already working on item B.

In an agent system, a pipeline might look like: vendor IDs arrive → fetch vendor data → run risk analysis → generate compliance report → store results. All stages run simultaneously, each processing different vendors at the same time.

Each stage follows the same pattern: accept an input channel, return an output channel, launch a goroutine that processes items, and close the output channel when done.

package main  // entry point

import (
    "fmt"      // for printing
    "strings"  // for string manipulation
)

// Stage 1: Generate vendor IDs from a list
// Takes a slice of strings, returns a channel that emits them one by one
// This is the start of our assembly line — raw materials going in
func generateIDs(ids []string) <-chan string {
    out := make(chan string)  // create the output channel for this stage
    go func() {
        defer close(out)       // close when done — signals the next stage
        for _, id := range ids {  // loop through each ID
            out <- id          // send it to the next stage
        }
    }()
    return out  // return the channel immediately — data flows through it asynchronously
}

// Stage 2: Fetch vendor data (simulate by uppercasing the ID)
// Takes an input channel of IDs, returns an output channel of vendor names
func fetchVendors(ids <-chan string) <-chan string {
    out := make(chan string)  // create output channel
    go func() {
        defer close(out)  // close when done
        for id := range ids {  // receive IDs from the previous stage
            out <- strings.ToUpper(id)  // "fetch" the vendor (simulate with uppercase)
        }
    }()
    return out  // return the output channel
}

// Stage 3: Assess risk (simulate by adding a prefix)
func assessRisk(vendors <-chan string) <-chan string {
    out := make(chan string)  // create output channel
    go func() {
        defer close(out)  // close when done
        for vendor := range vendors {  // receive vendors from the previous stage
            out <- fmt.Sprintf("[ASSESSED] %s", vendor)  // add assessment prefix
        }
    }()
    return out  // return the output channel
}

func main() {
    // Compose the pipeline — each stage's output feeds the next stage's input
    // This reads like a recipe: generate → fetch → assess
    ids := generateIDs([]string{"acme", "globex", "initech"})  // stage 1: produce IDs
    vendors := fetchVendors(ids)    // stage 2: fetch vendor data from IDs
    assessed := assessRisk(vendors)  // stage 3: assess risk from vendor data

    // Consume the final output
    for result := range assessed {  // receive final results until the channel closes
        fmt.Println(result)  // print each fully-processed result
    }
    // Output:
    // [ASSESSED] ACME
    // [ASSESSED] GLOBEX
    // [ASSESSED] INITECH
}

errgroup: WaitGroup with Error Handling

errgroup from the golang.org/x/sync package is an enhanced WaitGroup that adds two critical features: error propagation and automatic context cancellation. When any goroutine returns an error, the group's context is cancelled (signaling other goroutines to stop), and g.Wait() returns that error.

Think of it as a project team with a rule: "if anyone hits a blocker, the whole team stops and we reassess." This is much better than the basic WaitGroup approach where errors from individual goroutines are silently lost.

package main  // entry point

import (
    "context"  // for context cancellation
    "fmt"      // for printing and error formatting
    "time"     // for simulating work

    "golang.org/x/sync/errgroup"  // external package for enhanced WaitGroup
)

// Vendor represents a third-party vendor to assess
type Vendor struct {
    Name string  // vendor name
}

// Result holds an assessment outcome
type Result struct {
    VendorName string  // which vendor
    Score      int     // their risk score
}

// assess simulates assessing a vendor — returns an error for "BadCorp"
func assess(ctx context.Context, v Vendor) (Result, error) {
    time.Sleep(500 * time.Millisecond)  // simulate work

    // Check if context was cancelled (maybe another goroutine failed)
    if ctx.Err() != nil {
        return Result{}, ctx.Err()  // stop early if cancelled
    }

    if v.Name == "BadCorp" {  // simulate a failure for this vendor
        return Result{}, fmt.Errorf("failed to assess %s: connection refused", v.Name)
    }

    return Result{VendorName: v.Name, Score: len(v.Name) * 5}, nil  // success
}

func main() {
    vendors := []Vendor{
        {Name: "Acme"}, {Name: "Globex"}, {Name: "BadCorp"}, {Name: "Initech"},
    }

    // Create an errgroup with a context
    // When any goroutine in the group returns an error, ctx is automatically cancelled
    g, ctx := errgroup.WithContext(context.Background())  // create group + cancellable context

    // Pre-allocate a results slice — each goroutine writes to its own index (safe, no overlap)
    results := make([]Result, len(vendors))  // one slot per vendor

    for i, v := range vendors {  // loop through each vendor
        i, v := i, v  // capture loop variables for the closure (needed in Go <1.22)

        // g.Go launches a goroutine that returns an error
        // If ANY goroutine returns an error, the context is cancelled
        g.Go(func() error {
            r, err := assess(ctx, v)  // run the assessment, passing the group's context
            if err != nil {
                return err  // return the error — this cancels the context for everyone
            }
            results[i] = r  // store result in the pre-allocated slot (safe — no overlap)
            return nil      // success
        })
    }

    // Wait for all goroutines to complete — returns the FIRST error (if any)
    if err := g.Wait(); err != nil {
        fmt.Println("Assessment failed:", err)  // prints the error from BadCorp
    } else {
        fmt.Println("All assessments succeeded!")
        for _, r := range results {
            fmt.Printf("  %s: score %d\n", r.VendorName, r.Score)
        }
    }
}
errgroup is your go-to for parallel work with error handling
In production Go code, errgroup replaces the manual "WaitGroup + error channel + select" pattern. Use it whenever you need to: (1) run multiple goroutines in parallel, (2) collect errors, and (3) cancel remaining work when something fails. It's a standard tool in every Go developer's toolkit.

Context Package

The Walkie-Talkie Analogy

Imagine a construction site where every worker carries a walkie-talkie. The site foreman (the top-level caller) can broadcast to all workers at once: "Stop what you're doing, we're shutting down for the day." Every worker hears the message immediately and stops. If a team lead is supervising a subgroup, they have their own walkie-talkie channel that relays the foreman's messages plus their own. When the foreman says "stop," the team lead's entire subgroup hears it too, automatically.

That's what context.Context does in Go. It's a value that carries three things through your entire call chain:

  • Cancellation signals: "Stop all work" — like the foreman's shutdown broadcast
  • Deadlines/timeouts: "You have 30 seconds to finish" — like a timer that automatically triggers the stop signal
  • Request-scoped values: "Here's the request ID for logging" — like a job number written on everyone's clipboard

The killer feature is propagation. Contexts form a tree: when a parent context is cancelled, all child contexts are automatically cancelled too. Imagine an assessment that launches 4 agent goroutines, each making 3 LLM API calls. That's 12 in-flight API calls. If the user cancels the request, you need all 12 calls to stop immediately to avoid wasting compute and money. Context propagation handles this automatically — cancel the parent, and every child operation receives the signal.

This is why context.Context is the first parameter of virtually every function in Go that does I/O — HTTP handlers, database queries, API clients, file operations. It's the universal "cancellation and metadata" carrier.

context.Background(): The Starting Point

Every context tree starts with a root. context.Background() returns an empty, never-cancelled context that serves as the root. It's like the main walkie-talkie channel that's always on. You typically call it in main(), in test functions, or at the top of an HTTP handler.

package main  // entry point

import (
    "context"  // the context package
    "fmt"      // for printing
)

func main() {
    // context.Background() is the root of all contexts
    // It never expires, is never cancelled, and has no values
    // Think of it as the "main channel" that's always on
    ctx := context.Background()  // create the root context

    // Pass it to functions that need it
    doWork(ctx)  // ctx is always the FIRST parameter — Go convention
}

// doWork accepts a context as its first parameter
// This is the universal Go convention for any function that does I/O

    fmt.Println("Doing work with context")  // use the context for cancellation, etc.

WithCancel: Manual Stop Button

context.WithCancel creates a child context with a cancel function. When you call cancel(), the child context (and all its descendants) receive a cancellation signal. Goroutines watching the context's Done() channel wake up and can clean up. Think of it as giving someone a walkie-talkie with a "stop" button — press it, and everyone on that channel hears the stop signal.

package main  // entry point

import (
    "context"  // for context management
    "fmt"      // for printing
    "time"     // for Sleep
)

// worker simulates a long-running task that respects cancellation
func worker(ctx context.Context, name string) {
    for {  // infinite loop — keeps working until told to stop
        select {
        case <-ctx.Done():  // listen for the cancellation signal
            // ctx.Done() returns a channel that closes when cancel() is called
            // receiving from a closed channel returns immediately
            fmt.Printf("%s: received stop signal, shutting down. Reason: %v\n",
                name, ctx.Err())  // ctx.Err() tells us WHY we were cancelled
            return  // stop working
        default:  // no cancellation yet — keep working
            fmt.Printf("%s: working...\n", name)  // report progress
            time.Sleep(500 * time.Millisecond)      // simulate doing some work
        }
    }
}

func main() {
    // Create a cancellable context — like handing out walkie-talkies with a stop button
    ctx, cancel := context.WithCancel(context.Background())  // parent: Background, child: cancellable

    // Launch two workers that listen for cancellation
    go worker(ctx, "Risk-Analyzer")       // worker 1 — gets the cancellable context
    go worker(ctx, "Compliance-Checker")  // worker 2 — same context, same stop signal

    // Let them work for 2 seconds
    time.Sleep(2 * time.Second)  // workers are running concurrently during this time

    // Press the stop button — both workers receive the cancellation signal
    fmt.Println("Boss: Cancelling all work!")
    cancel()  // this closes ctx.Done(), which wakes up both workers' select statements

    time.Sleep(100 * time.Millisecond)  // brief pause to let workers print their shutdown messages
    fmt.Println("All workers stopped.")
}

WithTimeout: Automatic Deadline

context.WithTimeout creates a child context that automatically cancels itself after a specified duration. This is essential for any operation that might hang — especially external API calls. If an LLM takes too long to respond, you want to give up and return an error rather than blocking forever.

package main  // entry point

import (
    "context"  // for context with timeout
    "errors"   // for errors.Is comparison
    "fmt"      // for printing
    "time"     // for durations
)

// callLLM simulates an LLM API call that takes 5 seconds
func callLLM(ctx context.Context, prompt string) (string, error) {
    // Simulate a slow API call — check context periodically
    select {
    case <-time.After(5 * time.Second):  // simulate 5 seconds of processing
        return "LLM says: vendor is high risk", nil  // return the response
    case <-ctx.Done():  // if context is cancelled or times out...
        return "", ctx.Err()  // return the cancellation error
    }
}

func main() {
    // Create a context that AUTOMATICALLY cancels after 2 seconds
    // The LLM call takes 5 seconds, so it will be cut short
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel()  // ALWAYS defer cancel() — this releases resources even if the timeout doesn't fire
                     // forgetting this leaks goroutines inside the context package

    fmt.Println("Calling LLM with 2-second timeout...")
    result, err := callLLM(ctx, "Analyze vendor security")  // pass the timeout context

    if errors.Is(err, context.DeadlineExceeded) {  // check if the error was a timeout
        fmt.Println("LLM timed out! Using cached result instead.")  // handle gracefully
    } else if err != nil {
        fmt.Println("LLM error:", err)  // some other error
    } else {
        fmt.Println("LLM response:", result)  // success (won't reach this in our example)
    }
    // Output: "LLM timed out! Using cached result instead."
    // because 2-second timeout < 5-second LLM call
}

WithValue: Request-Scoped Metadata

context.WithValue attaches key-value metadata to a context. This is useful for passing request-scoped data (like a request ID or user info) through your call chain without adding parameters to every function signature. But use it sparingly — it's not type-safe, and overuse makes code harder to understand.

package main  // entry point

import (
    "context"  // for context with values
    "fmt"      // for printing
)

// Define a custom type for context keys to avoid collisions
// Using a plain string like "request_id" could collide with other packages
// A custom type makes the key unique to your package
type ctxKey string  // unexported type — only this package can create keys of this type

// Define specific keys as constants
const (
    requestIDKey ctxKey = "request_id"  // key for storing request IDs in context
    userIDKey    ctxKey = "user_id"     // key for storing user IDs in context
)

// processRequest simulates handling a request, reading metadata from context
func processRequest(ctx context.Context) {
    // Extract the request ID from context
    // ctx.Value returns 'any' (interface{}), so we need a type assertion
    reqID, ok := ctx.Value(requestIDKey).(string)  // type assertion: any → string
    if !ok {
        reqID = "unknown"  // fallback if the value isn't in the context
    }

    fmt.Printf("Processing request %s\n", reqID)  // use the request ID for logging
}

func main() {
    // Start with a background context
    ctx := context.Background()  // root context

    // Attach a request ID to the context
    // WithValue creates a NEW context that wraps the parent and adds one key-value pair
    ctx = context.WithValue(ctx, requestIDKey, "req-abc-123")  // add request ID

    // Attach a user ID too — each WithValue wraps the previous context
    ctx = context.WithValue(ctx, userIDKey, "user-42")  // add user ID

    // Pass the enriched context to the handler
    processRequest(ctx)  // Output: "Processing request req-abc-123"
}

Context Propagation: The Cancellation Tree

The most powerful aspect of context is how cancellation propagates down the tree. When you cancel a parent context, all children and grandchildren are cancelled too. This diagram shows how it works in an agent system:

Context Tree — Cancellation Propagates Down: context.Background() ← root (never cancelled) │ └── WithTimeout(30s) ← HTTP request handler │ ├── WithCancel() ← Risk Agent │ ├── LLM call (ctx) ← automatically cancelled │ └── DB query (ctx) ← automatically cancelled │ └── WithCancel() ← Compliance Agent ├── LLM call (ctx) ← automatically cancelled └── API call (ctx) ← automatically cancelled If the 30s timeout fires: ✓ Risk Agent's LLM call is cancelled ✓ Risk Agent's DB query is cancelled ✓ Compliance Agent's LLM call is cancelled ✓ Compliance Agent's API call is cancelled → All 4 operations stop, saving compute and API costs
The five rules of context
  1. First parameter: ctx context.Context is always the first parameter of any function that does I/O. This is a universal Go convention.
  2. Never store in structs: Pass context through function parameters, not struct fields. Storing context in a struct makes lifetime management confusing.
  3. Always defer cancel: When you create a context with WithCancel or WithTimeout, always defer cancel() immediately. Forgetting this leaks resources.
  4. Always propagate: Pass the context to every downstream function. If you create a new context.Background() instead of using the one you received, you break the cancellation chain.
  5. Use WithValue sparingly: Only for truly request-scoped data (request IDs, auth tokens). Never for function parameters that should be explicit.

Functional Options Pattern

Imagine you walk into a coffee shop and order a latte. But you want oat milk instead of regular, an extra espresso shot, no sugar, and you want it iced. Each of those requests modifies a base order. You don't have to specify every single detail from scratch — the barista starts with a default latte, and each of your requests tweaks it. That's exactly how the functional options pattern works in Go.

Here's the problem this pattern solves. Suppose you're building an AI agent and it has many configuration settings: which LLM model to use, what temperature to set, how many tokens to allow, which tools to equip, how long to wait before timing out. If you put all of these into a constructor function, you'd end up with something monstrous — a function with 10 or more parameters, most of which have sensible defaults that you don't want to specify every time. And every time you add a new setting, every caller needs to be updated.

Some languages solve this with a "Builder" pattern — a chain of method calls that progressively set options. Go takes a different, more elegant approach: functional options. Each option is a function that modifies a configuration struct. You pass zero or more of these option functions into the constructor, and each one tweaks the defaults. The constructor starts with sensible defaults, then applies each option in order. It's like telling the barista "start with a regular latte" and then handing them a list of modifications.

This pattern was invented by Rob Pike (one of Go's creators) and popularized by Dave Cheney. It's now considered the idiomatic Go way to handle optional configuration. You'll see it throughout Go's standard library and in virtually every well-designed Go package. The critical advantage: adding a new option is a non-breaking change. Existing callers don't need to update their code when you add a new WithSomething() function.

Let's build this step by step, from the ground up.

Step 1: Define the Configuration Struct

First, we create a struct that holds all possible configuration values. This struct is usually unexported (lowercase name) so that users of your package can only configure it through the option functions, not by creating it directly. Think of this as the "order form" at the coffee shop — it lists every possible customization.

// agentConfig holds all configurable settings for an agent.
// It is unexported — users configure it through option functions only.
type agentConfig struct {
    model       string        // which LLM model to use (e.g., "gpt-4o")
    temperature float64       // creativity level: 0.0 = deterministic, 1.0 = very creative
    maxTokens   int           // maximum number of tokens the LLM can generate
    tools       []Tool        // tools the agent can use (search, calculate, etc.)
    timeout     time.Duration // how long to wait before cancelling an LLM call
    systemPrompt string       // the system-level instructions for the agent
    maxRetries  int           // how many times to retry on failure
}

Step 2: Define the Option Type

An "option" is simply a function that takes a pointer to the config struct and modifies it. We define a named type for this function signature. This is the key insight: options are functions, not data. Each function knows how to modify exactly one aspect of the configuration.

// AgentOption is a function that modifies an agent's configuration.
// Each option function focuses on changing one specific setting.
// The type is exported so users can pass options to the constructor.
type AgentOption func(*agentConfig)

Step 3: Create Option Functions

Now we create one function per configurable setting. Each function returns an AgentOption — which is itself a function that modifies the config. This is a "function that returns a function," also known as a closure. When you call WithModel("gpt-4o"), it captures the string "gpt-4o" and returns a function that, when called later, sets c.model to that value.

Think of each With... function as a sticky note you hand to the barista: "use oat milk", "add extra shot". Each note is a simple instruction that modifies one thing about the order.

// WithModel sets which LLM model the agent uses.
// Returns a function that, when applied to a config, changes the model.
func WithModel(m string) AgentOption {
    return func(c *agentConfig) { // this inner function modifies the config
        c.model = m              // set the model to the value captured from WithModel's parameter
    }
}

// WithTemperature sets the LLM creativity level (0.0 to 1.0).
// Lower = more focused and deterministic. Higher = more creative and varied.
func WithTemperature(t float64) AgentOption {
    return func(c *agentConfig) { // closure captures the temperature value 't'
        c.temperature = t        // apply it to the config when this function is called
    }
}

// WithMaxTokens limits how many tokens the LLM can generate in one response.
// Useful for controlling cost and response length.
func WithMaxTokens(n int) AgentOption {
    return func(c *agentConfig) { // closure captures the max tokens value 'n'
        c.maxTokens = n          // set the token limit in the config
    }
}

// WithTools gives the agent a set of tools it can use (search, calculate, etc.).
// The ... syntax means you can pass any number of tools.
func WithTools(tools ...Tool) AgentOption {
    return func(c *agentConfig) { // closure captures the tools slice
        c.tools = tools          // replace the default tools with the provided ones
    }
}

// WithTimeout sets how long to wait for an LLM response before giving up.
// This prevents your agent from hanging forever if the API is slow.
func WithTimeout(d time.Duration) AgentOption {
    return func(c *agentConfig) { // closure captures the duration 'd'
        c.timeout = d            // set the timeout in the config
    }
}

// WithSystemPrompt sets the system-level instructions that guide the agent's behavior.
// This is the "personality" and "rules" the agent follows.
func WithSystemPrompt(prompt string) AgentOption {
    return func(c *agentConfig) { // closure captures the prompt string
        c.systemPrompt = prompt  // set the system prompt in the config
    }
}

// WithRetries sets how many times the agent retries on failure.
// Useful for handling transient LLM API errors gracefully.
func WithRetries(n int) AgentOption {
    return func(c *agentConfig) { // closure captures the retry count 'n'
        c.maxRetries = n         // set the retry count in the config
    }
}

Step 4: The Constructor — Defaults + Options

The constructor function is where it all comes together. It starts by creating a config with sensible defaults (the "regular latte"), then loops through every option function that was passed in and applies it (your customizations). The ...AgentOption parameter means the caller can pass zero, one, five, or twenty options. If they pass zero, they get the defaults.

// NewAgent creates a new agent with the given name and optional configuration.
// It starts with sensible defaults, then applies each option function in order.
// The ...AgentOption parameter means callers can pass any number of options.
func NewAgent(name string, opts ...AgentOption) *LLMAgent {
    // Step A: Create config with sensible defaults (the "regular latte")
    cfg := &agentConfig{
        model:       "gpt-4o",          // default to GPT-4o
        temperature: 0.7,               // default: moderate creativity
        maxTokens:   4096,              // default: allow up to 4096 tokens
        timeout:     30 * time.Second,  // default: 30-second timeout
        maxRetries:  2,                 // default: retry twice on failure
    }

    // Step B: Apply each option function to override specific defaults
    // This is the barista reading your list of modifications one by one
    for _, opt := range opts { // loop through every option the caller provided
        opt(cfg)                // call the option function, passing our config for modification
    }

    // Step C: Build the agent using the now-customized configuration
    return &LLMAgent{
        name:   name, // the agent's name (required, not optional)
        config: *cfg, // copy the finalized config into the agent
    }
}

Step 5: Using the Pattern

Now look at how clean the call site becomes. Each With... call reads like an English instruction. You only specify what you want to change; everything else uses defaults. And if someone adds WithMaxRetries() next month, no existing code breaks.

// Create an agent with just defaults — no options needed
// This is like ordering "just a regular latte, no changes"
simpleAgent := NewAgent("summarizer") // uses all default settings

// Create a customized agent — override only what you need
// This reads like: "I want a risk-assessor agent with these tweaks..."
riskAgent := NewAgent("risk-assessor",       // required: the agent's name
    WithModel("gpt-4-turbo"),                // override: use a faster model
    WithTemperature(0.2),                    // override: low creativity for consistent scoring
    WithTools(searchTool, calculatorTool),    // override: equip these specific tools
    WithTimeout(60 * time.Second),            // override: allow more time for complex analysis
    WithSystemPrompt("You are a risk assessment specialist..."), // override: custom instructions
)

// Create another agent with different options — same constructor, different config
docAgent := NewAgent("doc-analyzer",         // required: the agent's name
    WithModel("gpt-4o"),                     // override: use gpt-4o for better reasoning
    WithMaxTokens(8192),                     // override: allow longer responses for documents
    WithRetries(3),                           // override: retry 3 times (document APIs can be flaky)
)
Why not just use a config struct directly?
You could pass a config struct to the constructor: NewAgent("name", AgentConfig{Model: "gpt-4"}). But this has a problem: you can't tell the difference between "the user set temperature to 0.0" and "the user didn't set temperature at all" (because Go's zero value for float64 is 0.0). Functional options avoid this ambiguity entirely — if you don't pass WithTemperature(), the default is used. If you pass WithTemperature(0.0), that explicit value is applied.

The Middleware Pattern

A closely related pattern is middleware — wrapping behavior around something without changing it. Think of middleware like layers of an onion, or security checkpoints at an airport. Each checkpoint (middleware layer) does one specific job: check your ID, scan your bag, verify your boarding pass. Your original "thing" (the flight) remains unchanged, but each layer adds behavior before and after.

In Go, middleware for agents works like this: you take an agent, wrap it with a layer that adds logging, wrap that with a layer that adds retry logic, and wrap that with a layer that adds rate limiting. Each wrapper delegates to the inner agent but adds its own behavior. This is powerful because each layer is independent — you can add, remove, or reorder layers without touching the agent itself.

The secret is that each middleware returns the same interface it received. So from the outside, a wrapped agent looks identical to an unwrapped one. Code that uses an Agent doesn't know or care how many layers of middleware are around it.

// AgentMiddleware is a function that wraps an Agent with additional behavior.
// It takes an Agent and returns a new Agent with the same interface.
// Think of it as a security checkpoint that surrounds the original agent.
type AgentMiddleware func(Agent) Agent

// WithLogging creates middleware that logs every agent call.
// It records when the agent starts, when it finishes, and how long it took.
func WithLogging(logger *slog.Logger) AgentMiddleware {
    return func(next Agent) Agent {        // takes the inner agent, returns a wrapper
        return &loggingAgent{               // the wrapper struct that implements Agent
            next:   next,                    // store the inner agent to delegate to
            logger: logger,                  // store the logger for this middleware
        }
    }
}

// loggingAgent wraps another agent and adds logging behavior.
type loggingAgent struct {
    next   Agent        // the inner agent we delegate to
    logger *slog.Logger // the logger for recording agent activity
}

// Run implements the Agent interface — logs, then delegates, then logs again.
func (a *loggingAgent) Run(ctx context.Context, msgs []Message) (Response, error) {
    start := time.Now()                      // record when the agent started
    a.logger.Info("agent starting",           // log that we're about to call the agent
        "agent", a.next.Name(),              // include the agent's name in the log
        "messages", len(msgs),               // include how many messages are being sent
    )

    resp, err := a.next.Run(ctx, msgs)       // delegate to the actual inner agent

    a.logger.Info("agent finished",           // log that the agent has completed
        "agent", a.next.Name(),              // include the agent's name again
        "duration", time.Since(start),       // how long the agent took to run
        "error", err,                         // whether it succeeded or failed
    )
    return resp, err                         // return exactly what the inner agent returned
}

// Name implements the Agent interface — just delegates to the inner agent.
func (a *loggingAgent) Name() string {
    return a.next.Name() // the wrapper doesn't change the agent's name
}

// WithRetry creates middleware that retries the agent on failure.
// If the agent fails, it waits briefly and tries again, up to maxAttempts.
func WithRetry(maxAttempts int) AgentMiddleware {
    return func(next Agent) Agent {        // takes the inner agent, returns a wrapper
        return &retryAgent{                 // the wrapper struct with retry logic
            next:        next,               // store the inner agent
            maxAttempts: maxAttempts,         // store how many times we should retry
        }
    }
}

// Compose middleware: each layer wraps the previous one.
// The agent is like a gift being wrapped in layers of paper.
// The outermost layer (logging) runs first and last.
agent := NewRiskAssessor(llmClient)        // create the base agent
agent = WithLogging(logger)(agent)          // wrap it with logging (layer 1)
agent = WithRetry(3)(agent)                // wrap logged agent with retry (layer 2)

Strategy Pattern via Interfaces

The Strategy pattern is about swapping algorithms at runtime without changing the code that uses them. In Go, this pattern almost disappears because interfaces are implicit — any type that has the right methods automatically satisfies the interface. There's no explicit "implements" declaration.

Imagine a system where you score vendor risk. For premium vendors, you use an expensive AI-powered scorer. For basic vendors, you use a simple rules-based scorer. And you have a hybrid that tries AI first and falls back to rules if the AI fails. All three implement the same interface, so the rest of your code doesn't care which one is used.

// RiskScorer defines the strategy interface — any risk scoring approach.
// Any struct with a Score method matching this signature automatically satisfies it.
type RiskScorer interface {
    Score(ctx context.Context, vendor Vendor) (RiskScore, error) // compute a risk score
}

// LLMScorer uses AI to analyze vendor risk — expensive but thorough.
type LLMScorer struct {
    client LLMClient // the LLM client used for AI-powered scoring
}

// RuleBasedScorer uses predefined rules — fast and cheap.
type RuleBasedScorer struct {
    rules []Rule // the set of scoring rules to evaluate
}

// HybridScorer tries AI first, falls back to rules on failure.
// This is a common production pattern: use the best option, degrade gracefully.
type HybridScorer struct {
    llm   RiskScorer // the primary scorer (AI-powered)
    rules RiskScorer // the fallback scorer (rules-based)
}

// Score tries the AI scorer first. If it fails, falls back to rules.
// The caller doesn't know which strategy actually produced the score.
func (h *HybridScorer) Score(ctx context.Context, v Vendor) (RiskScore, error) {
    score, err := h.llm.Score(ctx, v)   // try the AI scorer first
    if err != nil {                      // if AI scoring failed...
        slog.Warn("LLM scorer failed, falling back to rules", // log the fallback
            "vendor", v.ID,              // include which vendor we're scoring
            "error", err,                 // include the error for debugging
        )
        return h.rules.Score(ctx, v)     // fall back to rules-based scoring
    }
    return score, nil                    // AI scoring succeeded, return the result
}
The three patterns work together
In a real agent system, you'd use all three patterns together: functional options to configure each agent when creating it, middleware to wrap agents with logging/retry/rate-limiting, and strategy to swap different implementations (like LLM providers) without changing calling code. These are the building blocks of clean Go architecture.

HTTP & APIs

Think of an HTTP server as a receptionist at a large office. People (clients) arrive at the front desk (your server), state their request ("I need the risk report for vendor 42"), and the receptionist figures out which department (handler function) should handle it. The receptionist doesn't do the work themselves — they route the request to the right person and relay the response back.

Go comes with a production-grade HTTP server built right into the standard library. This is unusual and powerful: you don't need a framework, a third-party web server, or any external dependencies. The net/http package is what many large companies use in production, handling millions of requests per day. Go 1.22 added method-based routing (you can write "POST /api/assess" directly), which eliminated the main reason people used third-party routers.

For agent systems, the HTTP server is your system's front door. Users or other services send requests (like "assess this vendor" or "check the status of an assessment"), and your HTTP handlers receive those requests, call the appropriate agent or orchestrator, and send back the results as JSON. Understanding HTTP in Go is essential for building any real-world agent system.

Unlike heavyweight web frameworks that hide complexity behind annotations and magic, Go's approach is refreshingly transparent. Every handler is a plain function. You read JSON manually, validate manually, and write responses manually. This means you can read any handler from top to bottom and understand exactly what happens when a request arrives. There's no hidden behavior, no auto-wiring, no magic.

Your First HTTP Server

Let's build a simple HTTP server step by step. We'll create an API for an agent system that accepts vendor assessment requests and returns risk scores. The server has two endpoints: a health check (to verify the server is running) and an assessment endpoint (to trigger vendor analysis).

package main // the entry point package for the executable

import (
    "encoding/json" // for converting between Go structs and JSON
    "log/slog"      // structured logging (built-in since Go 1.21)
    "net/http"      // the HTTP server and client (built-in, production-ready)
    "time"          // for timestamps in our response
)

// AssessmentRequest is what the client sends to us.
// The json tags control how these fields appear in JSON.
type AssessmentRequest struct {
    VendorID string `json:"vendor_id"` // the vendor to assess, e.g. "v-42"
    Scope    string `json:"scope"`     // assessment scope: "full", "quick", "compliance"
}

// AssessmentResponse is what we send back to the client.
type AssessmentResponse struct {
    VendorID  string    `json:"vendor_id"`  // echo back which vendor was assessed
    RiskScore int       `json:"risk_score"` // the computed risk score (0-100)
    RiskLevel string    `json:"risk_level"` // human-readable level: "low", "medium", "high"
    Summary   string    `json:"summary"`    // brief explanation of the score
    Timestamp time.Time `json:"timestamp"`  // when the assessment was completed
}

// handleAssessment processes a vendor risk assessment request.
// It receives HTTP request/response objects, just like any web handler.
// w = where we write the response, r = the incoming request.
func handleAssessment(w http.ResponseWriter, r *http.Request) {
    // Step 1: Decode the JSON request body into our Go struct.
    // json.NewDecoder reads from r.Body (the request's body).
    // Decode parses the JSON and fills in our struct's fields.
    var req AssessmentRequest                                // create an empty struct to fill
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil { // parse JSON into struct
        http.Error(w, "invalid JSON in request body", http.StatusBadRequest) // 400 error
        return                                               // stop processing this request
    }

    // Step 2: Validate the request — make sure required fields are present.
    if req.VendorID == "" {                                   // check that vendor_id was provided
        http.Error(w, "vendor_id is required", http.StatusBadRequest) // 400 error
        return                                               // stop processing
    }

    // Step 3: Call the agent/orchestrator to perform the actual assessment.
    // r.Context() carries the HTTP request's cancellation signal —
    // if the client disconnects, this context gets cancelled automatically.
    result, err := orchestrator.Assess(r.Context(), req.VendorID) // run the assessment
    if err != nil {                                           // if assessment failed...
        slog.Error("assessment failed",                        // log the error with context
            "vendor", req.VendorID,                           // which vendor failed
            "error", err,                                      // what went wrong
        )
        http.Error(w, "assessment failed", http.StatusInternalServerError) // 500 error
        return                                               // stop processing
    }

    // Step 4: Send the successful response as JSON.
    w.Header().Set("Content-Type", "application/json")       // tell client it's JSON
    w.WriteHeader(http.StatusOK)                              // send 200 OK status code
    json.NewEncoder(w).Encode(result)                         // convert struct to JSON and write it
}

// handleHealth is a simple health check endpoint.
// Monitoring tools ping this to verify the server is alive.
func handleHealth(w http.ResponseWriter, r *http.Request) {
    w.Header().Set("Content-Type", "application/json")       // tell client it's JSON
    json.NewEncoder(w).Encode(map[string]string{            // create a simple map for the response
        "status": "healthy",                                // the server is running
        "time":   time.Now().Format(time.RFC3339),          // current time in standard format
    })
}

func main() {
    // Create a new HTTP multiplexer (request router).
    // The mux is the "receptionist" — it looks at each request's
    // method and path, and routes it to the correct handler function.
    mux := http.NewServeMux() // create a new router

    // Register routes: "METHOD /path" maps to handler functions.
    // Go 1.22+ supports method-based routing with this syntax.
    mux.HandleFunc("POST /api/v1/assess", handleAssessment) // POST requests to assess vendors
    mux.HandleFunc("GET /api/v1/health", handleHealth)      // GET requests for health checks

    // Start the HTTP server.
    // This blocks forever, listening for incoming requests on port 8080.
    slog.Info("starting agent server", "port", 8080)  // log that we're starting
    err := http.ListenAndServe(":8080", mux)                 // start listening on port 8080
    slog.Error("server stopped", "error", err)                // if we get here, the server crashed
}

JSON Encoding & Decoding in Depth

JSON is the lingua franca of web APIs. In Go, the encoding/json package converts between Go structs and JSON. It uses struct tags — those backtick-quoted strings after each field — to know what the JSON field names should be. Think of struct tags as labels on a package: they tell the JSON encoder/decoder what name to use, whether to skip empty fields, and more.

There are two main approaches: json.Marshal/json.Unmarshal (convert to/from byte slices) and json.NewEncoder/json.NewDecoder (stream to/from an io.Reader/io.Writer). The streaming version is preferred for HTTP handlers because it avoids creating an intermediate byte slice.

// VendorResponse demonstrates all the useful struct tag features.
type VendorResponse struct {
    ID        string   `json:"id"`                  // JSON field name is "id"
    Name      string   `json:"name"`                // JSON field name is "name"
    RiskScore int      `json:"risk_score"`          // underscores for JSON convention
    Tags      []string `json:"tags,omitempty"`      // omit from JSON if the slice is empty
    Internal  string   `json:"-"`                   // NEVER include in JSON output (private field)
    Notes     string   `json:"notes,omitempty"`     // omit from JSON if the string is empty
}

HTTP Middleware: Security Guards for Your Server

Middleware is like having security guards stationed at the entrance of a building. Every person (request) who enters must pass through the guards. One guard checks your ID (authentication). Another scans your bag (input validation). A third writes your name in a visitor log (request logging). The actual office (your handler) doesn't need to worry about any of this — the guards handle it before anyone gets through.

In Go, HTTP middleware follows the same wrapping pattern you saw with agent middleware. Each middleware function takes an http.Handler and returns a new http.Handler that adds behavior. Middleware can run code before the request reaches your handler (like logging the request start time), after the handler runs (like logging the response time), or both.

The beautiful thing about this pattern is composability. Each middleware is independent. You can add CORS handling, request logging, authentication, rate limiting, and panic recovery — each in its own function, each testable in isolation. You stack them together by wrapping one around the other.

// loggingMiddleware records every HTTP request that comes through.
// It wraps the actual handler, recording the start time before
// the request and the duration after the response is sent.
func loggingMiddleware(next http.Handler) http.Handler {
    // http.HandlerFunc converts a plain function into an http.Handler.
    // This is an adapter that lets us use a function where an interface is expected.
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()                  // record when the request started
        next.ServeHTTP(w, r)                 // call the actual handler (or next middleware)
        slog.Info("http request",              // log after the handler finishes
            "method", r.Method,              // GET, POST, etc.
            "path", r.URL.Path,              // which endpoint was called
            "duration", time.Since(start),   // how long the request took
        )
    })
}

// corsMiddleware adds CORS headers so browser-based clients can call our API.
// Without CORS headers, browsers block requests from different domains.
func corsMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Access-Control-Allow-Origin", "*")        // allow any origin
        w.Header().Set("Access-Control-Allow-Methods", "GET, POST, OPTIONS") // allowed methods
        w.Header().Set("Access-Control-Allow-Headers", "Content-Type, Authorization") // allowed headers

        if r.Method == "OPTIONS" {             // handle preflight requests
            w.WriteHeader(http.StatusOK)       // respond 200 to preflight, don't process further
            return                              // stop here for OPTIONS requests
        }
        next.ServeHTTP(w, r)                   // for non-OPTIONS, continue to the actual handler
    })
}

// authMiddleware checks that requests have a valid API key.
// This is the "ID check" guard — no valid key, no entry.
func authMiddleware(apiKey string) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {           // return a middleware function
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            key := r.Header.Get("Authorization")            // extract the API key from the header
            if key != "Bearer "+apiKey {                      // check if the key matches
                http.Error(w, "unauthorized", http.StatusUnauthorized) // 401 — rejected
                return                                        // do NOT call the next handler
            }
            next.ServeHTTP(w, r)                             // key is valid, proceed to handler
        })
    }
}

// recoverMiddleware catches panics and returns a 500 error instead of crashing.
// Without this, a panic in any handler would kill your entire server.
func recoverMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {                         // defer runs after the handler completes (or panics)
            if err := recover(); err != nil { // recover() catches any panic
                slog.Error("panic recovered",     // log the panic for debugging
                    "error", err,                // what caused the panic
                    "path", r.URL.Path,          // which endpoint panicked
                )
                http.Error(w, "internal server error", http.StatusInternalServerError) // 500
            }
        }()
        next.ServeHTTP(w, r)                   // call the actual handler
    })
}

// Compose all middleware together to protect the server.
// The request flows through each layer: recover → logging → cors → auth → handler.
// If any middleware rejects the request, the chain stops.
func setupServer(mux *http.ServeMux, apiKey string) http.Handler {
    var handler http.Handler = mux                   // start with the raw router
    handler = authMiddleware(apiKey)(handler)          // wrap with auth (innermost layer)
    handler = corsMiddleware(handler)                  // wrap with CORS handling
    handler = loggingMiddleware(handler)               // wrap with request logging
    handler = recoverMiddleware(handler)               // wrap with panic recovery (outermost layer)
    return handler                                    // return the fully wrapped handler
}
Middleware reads inside-out, executes outside-in
When you write recoverMiddleware(loggingMiddleware(corsMiddleware(mux))), the outermost middleware (recover) runs first on the request. It's like airport security: the outermost checkpoint is the first one you encounter. If a panic happens deep inside your handler, the outermost recovery middleware catches it. Reading the code, you see the innermost wrapper first — but execution starts from the outermost layer.

Testing

Think of testing like a pre-launch checklist for a space mission. Before a rocket launches, engineers run through hundreds of checks: fuel levels, engine pressure, navigation systems, communication links. Each check verifies one specific thing. If any check fails, the launch is delayed until the problem is fixed. In software, tests serve exactly the same purpose: they verify that each part of your code works correctly before you "launch" it to production.

Go has testing built directly into the language toolchain. There's no external testing framework to install, no test runner to configure, no complicated setup. You create a file ending in _test.go, write functions that start with Test, and run them with go test. That's it. The testing package is minimal but remarkably powerful — it covers unit tests, benchmarks (performance tests), and even fuzz testing (automatically generated random inputs).

Test files live right next to the code they test, not in a separate directory tree. This is intentional: it makes it easy to see which code has tests, encourages writing tests alongside features, and keeps related code physically close together. When you see agent.go and agent_test.go side by side, you know exactly what's tested.

Go's assertion style is deliberately minimal: you write if got != want { t.Errorf(...) } instead of using assertion functions like assertEquals. This seems verbose at first, but it produces clearer error messages because you control exactly what information is displayed on failure. Many Go developers also use the popular testify library for more concise assertions — both approaches are perfectly acceptable.

Your First Test

Let's start with the basics. A test function must be in a _test.go file, must start with Test (capital T), and must take a single *testing.T parameter. The t object is your interface to the testing framework — you use it to report failures, skip tests, run subtests, and log diagnostic information.

// File: risk_test.go
// Test files always end with _test.go — Go only compiles them during testing.
package risk // same package as the code being tested

import "testing" // the built-in testing package

// TestScoreToBand tests that risk scores map to the correct risk bands.
// Every test function must start with "Test" and take *testing.T as the only parameter.
func TestScoreToBand(t *testing.T) {
    got := scoreToBand(85)    // call the function we're testing with score 85
    want := "high"            // we expect 85 to be classified as "high" risk

    if got != want {           // compare actual result to expected result
        // t.Errorf reports a test failure but continues running the test.
        // Use t.Fatalf instead if you want to stop immediately on failure.
        t.Errorf("scoreToBand(85) = %q, want %q", got, want) // report what went wrong
    }
}

Table-Driven Tests: The Go Testing Pattern

Table-driven tests are the single most important Go testing pattern. The idea is beautifully simple: instead of writing separate test functions for each case, you define a table (a slice of structs) where each row is a test case with inputs and expected outputs. Then you loop through the table, running each case as a subtest. Think of it as a spreadsheet: each row is a scenario, each column is a parameter.

This pattern has three major advantages. First, adding a new test case is trivial — just add a row to the table. Second, the test logic is written once and shared across all cases, so there's no duplication. Third, each case gets a name, so when a test fails, you know exactly which scenario failed. Table-driven tests are the heart of Go testing culture.

// TestScoreToBand_TableDriven uses a table of test cases.
// Each case has a name, input, and expected output.
func TestScoreToBand_TableDriven(t *testing.T) {
    // Define the table: a slice of anonymous structs.
    // Each struct represents one test case (one row in the table).
    tests := []struct {
        name  string // human-readable description of this test case
        score int    // the input to our function
        want  string // the expected output
    }{
        {name: "critical risk", score: 95, want: "critical"},   // 95 should be critical
        {name: "high risk",     score: 80, want: "high"},       // 80 should be high
        {name: "medium risk",   score: 50, want: "medium"},     // 50 should be medium
        {name: "low risk",      score: 20, want: "low"},        // 20 should be low
        {name: "zero score",    score: 0,  want: "low"},        // 0 should also be low
        {name: "boundary 79",   score: 79, want: "medium"},     // 79 is right below "high"
        {name: "boundary 90",   score: 90, want: "critical"},   // 90 is the critical threshold
        {name: "max score",     score: 100, want: "critical"},  // 100 is maximum risk
    }

    // Loop through each test case and run it as a named subtest.
    for _, tt := range tests {        // tt is a common Go convention for "table test"
        t.Run(tt.name, func(t *testing.T) { // t.Run creates a named subtest
            got := scoreToBand(tt.score)  // call the function with this case's input
            if got != tt.want {           // compare actual to expected
                t.Errorf("scoreToBand(%d) = %q, want %q", // detailed error message
                    tt.score, got, tt.want) // includes the input and both outputs
            }
        })
    }
}

Mocking with Interfaces

When testing agent systems, you don't want to call real LLM APIs in every test — that would be slow, expensive, and non-deterministic (the LLM might return different text each time). Instead, you create "mock" implementations that pretend to be the real thing but return predictable, hardcoded responses.

Go makes mocking incredibly easy because interfaces are implicit. You don't need a mocking framework. You just create a struct that has the same methods as the interface, fill it with whatever test data you need, and pass it in. The code under test doesn't know or care that it's talking to a mock — it just sees an interface.

// mockLLMClient pretends to be a real LLM client for testing.
// It returns whatever response and error we configure it with.
type mockLLMClient struct {
    response string  // the fake response to return
    err      error   // the fake error to return (nil for success)
    calls    int     // count how many times Complete was called
}

// Complete implements the LLMClient interface.
// Instead of calling a real API, it returns our pre-configured response.
func (m *mockLLMClient) Complete(ctx context.Context, req CompletionRequest) (CompletionResponse, error) {
    m.calls++                              // track that this method was called
    if m.err != nil {                      // if we configured an error...
        return CompletionResponse{}, m.err // return it (simulates API failure)
    }
    return CompletionResponse{             // return the pre-configured success response
        Message: Message{
            Role:    RoleAssistant,          // the response comes from the "assistant"
            Content: m.response,            // the fake content we set up
        },
    }, nil                                 // nil error means success
}

// TestRiskAssessor_HighRisk tests that the agent correctly identifies high risk.
func TestRiskAssessor_HighRisk(t *testing.T) {
    // Create a mock that returns a high-risk assessment.
    mock := &mockLLMClient{                                    // create the mock
        response: `{"score": 92, "level": "critical"}`,         // fake LLM response
    }

    agent := NewRiskAssessor(mock)                              // create agent with the mock
    resp, err := agent.Run(context.Background(), testMessages)  // run the agent

    if err != nil {                                              // the agent should succeed
        t.Fatalf("unexpected error: %v", err)                    // Fatalf stops the test immediately
    }
    if mock.calls != 1 {                                         // verify the LLM was called exactly once
        t.Errorf("expected 1 LLM call, got %d", mock.calls)     // report if it was called wrong number of times
    }
    if len(resp.Messages) == 0 {                                 // response should have messages
        t.Fatal("expected at least one response message")         // fail if no messages returned
    }
}

// TestRiskAssessor_LLMFailure tests that the agent handles LLM errors gracefully.

Testing HTTP Handlers with httptest

Go's net/http/httptest package lets you test HTTP handlers without starting a real server. You create a fake request, a response recorder, and call the handler directly. The recorder captures the status code, headers, and body so you can verify them.

import (
    "net/http"         // for HTTP types and status codes
    "net/http/httptest" // for testing HTTP handlers without a real server
    "strings"          // for creating a reader from a string
    "testing"          // the testing package
)

// TestHandleAssessment tests the HTTP handler for vendor assessment.
func TestHandleAssessment(t *testing.T) {
    // Create a fake HTTP request body (the JSON the client would send).
    body := strings.NewReader(`{"vendor_id":"v-42","scope":"full"}`) // fake JSON body

    // Create a fake HTTP request — no real network involved.
    req := httptest.NewRequest(http.MethodPost, "/api/v1/assess", body) // POST to /assess
    req.Header.Set("Content-Type", "application/json")                // set content type

    // Create a response recorder — it captures what the handler writes.
    rec := httptest.NewRecorder()  // acts like an http.ResponseWriter that records everything

    // Call the handler directly — no server needed.
    handleAssessment(rec, req)     // invoke the handler with our fake request/response

    // Verify the response status code.
    if rec.Code != http.StatusOK { // check that we got 200 OK
        t.Errorf("expected status 200, got %d", rec.Code) // report the actual status code
    }

    // Verify the response body contains expected data.
    var resp AssessmentResponse                              // struct to decode into
    json.NewDecoder(rec.Body).Decode(&resp)                  // decode the recorded response
    if resp.VendorID != "v-42" {                              // check the vendor ID was echoed back
        t.Errorf("expected vendor v-42, got %s", resp.VendorID) // report the mismatch
    }
}

The Race Detector

Go has a built-in race condition detector that is arguably one of the most valuable tools in its toolchain. A race condition happens when two goroutines access the same variable simultaneously and at least one of them writes to it. Race conditions cause some of the nastiest bugs in software — they're intermittent, hard to reproduce, and can corrupt data silently.

The race detector works by instrumenting your code at compile time to track every memory access. When it detects two goroutines accessing the same memory without proper synchronization, it reports the exact locations in your code. Always run tests with -race for concurrent code. In agent systems where multiple goroutines process requests simultaneously, the race detector is indispensable.

# Run all tests with the race detector enabled.
# This catches data races that would be invisible otherwise.
go test -race ./...                     # test all packages with race detection

# Run tests for a specific package with verbose output.
go test -v -race ./internal/agent/...   # verbose + race detection for agent package

# Run only tests matching a specific pattern.
go test -run TestRiskAssessor -race ./  # run only tests with "TestRiskAssessor" in the name

# Generate a test coverage report.
go test -cover ./...                    # show what percentage of code is tested

# Run benchmarks to measure performance.
go test -bench=. -benchmem ./...        # run all benchmarks with memory allocation info

# Skip slow integration tests during development.
go test -short ./...                    # only run fast unit tests

Skipping Slow Tests

Integration tests (like those that call real LLM APIs) are slow and expensive. The -short flag lets you skip them during development while still running them in CI. Inside the test, check testing.Short() and skip if it's set.

// TestLLMIntegration calls a real LLM API — skip it during quick test runs.
func TestLLMIntegration(t *testing.T) {
    if testing.Short() {                  // check if -short flag was passed
        t.Skip("skipping integration test in short mode") // skip with a message
    }

    // This test only runs when you explicitly allow slow tests.
    client := NewOpenAIClient(os.Getenv("OPENAI_API_KEY"))   // create a real LLM client
    resp, err := client.Complete(context.Background(), CompletionRequest{ // make a real API call
        Model:    "gpt-4o-mini",                                // use a cheap model for tests
        Messages: []Message{{Role: RoleUser, Content: "Say hello"}}, // simple prompt
    })
    if err != nil {                                              // LLM call should succeed
        t.Fatalf("LLM call failed: %v", err)                      // fail with the error details
    }
    if resp.Message.Content == "" {                                // response should not be empty
        t.Error("expected non-empty response from LLM")             // report if empty
    }
}
Always use -race in your CI pipeline
go test -race has caught countless concurrency bugs before they reached production. For agent systems with heavy goroutine usage, the race detector is your best friend. Add it to your CI configuration and never ship code that fails the race check. The performance cost of running with -race is roughly 2-10x, which is fine for tests.

Think of a Go project layout like the blueprint of a building. What Goes Where: A Quick Guide

When you're not sure where to put a new file, use this mental checklist:

cmd/ — Entry Points

Each subdirectory under cmd/ produces one executable binary. The main.go file should be small: load config, create dependencies, wire them together, start the server. Think "turn the key to start the car."

internal/ — Private Code

All your business logic, agents, LLM clients, tools, and handlers live here. Nobody outside your project can import this. This is where 90% of your code goes.

pkg/ — Public Libraries

Only put code here that you explicitly want other projects to import. Most projects keep this empty or have one or two small utility packages. When in doubt, use internal/.

config/ — Configuration

Config structs and environment variable loading. The config is loaded in main.go and passed explicitly to every component that needs it. No global config, no hidden state.

The main.go: Wiring Everything Together

In Go, dependency wiring is explicit. There's no dependency injection container that magically discovers and connects your components. Instead, main.go creates each component, connects them together, and starts the server. This is more code than auto-wiring, but it means you can read main.go and see exactly how every piece of your system is connected. Nothing is hidden.

// File: cmd/server/main.go
// This is the entry point — where the program starts running.
package main // the "main" package is special — it produces an executable

import (
    "fmt"      // for string formatting
    "log"      // for fatal logging during startup
    "log/slog" // for structured logging throughout the application
    "net/http" // for the HTTP server
)

func main() {
    // Step 1: Load configuration from environment variables.
    cfg, err := loadConfig()       // read settings from environment
    if err != nil {                 // if config loading fails...
        log.Fatalf("config: %v", err) // crash immediately — can't run without config
    }

    // Step 2: Create the LLM client (talks to the AI model).
    llmClient := NewRateLimitedLLM(               // wrap the real client with rate limiting
        NewOpenAIClient(cfg.OpenAIKey),            // create the real OpenAI client
        10,                                       // allow 10 requests per second
    )

    // Step 3: Set up tools (the agent's "hands").
    registry := setupTools(cfg)                    // create and register all available tools

    // Step 4: Build the agent orchestrator.
    orchestrator := setupOrchestrator(llmClient, registry, cfg) // wire agents together

    // Step 5: Set up the HTTP handler with middleware.
    handler := setupHTTPHandler(orchestrator)       // create HTTP routes + middleware stack

    // Step 6: Start the server.
    slog.Info("starting agent system",         // log the startup with all key settings
        "port", cfg.Port,                          // which port we're listening on
        "model", cfg.LLMModel,                     // which LLM model we're using
        "workers", cfg.MaxWorkers,                 // how many worker goroutines
    )
    addr := fmt.Sprintf(":%d", cfg.Port)          // format the listen address (e.g., ":8080")
    err = http.ListenAndServe(addr, handler)       // start listening — blocks forever
    slog.Error("server stopped", "error", err)      // only reached if server crashes
}
Explicit wiring is a feature, not a bug
Yes, manually wiring dependencies in main.go is more typing. But it means there's exactly one place in your codebase where all dependencies are created and connected. No scanning, no reflection, no runtime surprises. When something goes wrong during startup, you know exactly where to look. This transparency becomes invaluable as your system grows.

What Is an Agentic System?

Imagine a team of expert analysts at a consulting firm. When a new client needs a risk assessment, the firm doesn't assign one person to do everything. They work in parallel when possible, share findings through a shared workspace, and the manager coordinates the entire process.

That's an agentic system. Replace the human analysts with LLM-powered agents, the tools with software APIs, and the shared workspace with channels and shared state — and you have a modern AI agent system. Each agent is a program that can reason (via an LLM), act (via tools), and communicate (via messages). The key insight is that a single LLM call is like asking one question to one person. An agentic system orchestrates many such calls across multiple specialized agents to accomplish tasks that no single LLM call could handle alone.

In practice, this means agents that can automatically read documents, analyze reports, cross-reference databases, score risks based on multiple criteria, and generate comprehensive reports for stakeholders — all with minimal human intervention. A task that might take a human analyst days can be completed in minutes.

Why Go Is Perfect for Agent Systems

Go was designed at Google to build large-scale concurrent systems. It turns out that the exact features Google needed for their infrastructure are the same features you need for agentic systems. Let's look at why Go fits this problem so naturally.

Goroutines = Cheap Agents

Each agent runs as a goroutine with a starting stack of just ~2KB. You can run thousands of agents simultaneously without breaking a sweat. A machine with 4GB of RAM could theoretically run 2 million goroutines. This means you don't need to worry about "how many agents can I afford to run?" — the answer is essentially "as many as you need."

Channels = Agent Communication

Channels are Go's built-in mechanism for safe communication between goroutines. Agents can send messages to each other through channels without worrying about data races. It's like having a pneumatic tube system connecting offices — you put a message in one end, and it arrives safely at the other end.

Context = Cancellation Propagation

When an orchestrator needs to cancel all child agents (timeout, user cancellation, error), context propagation handles it automatically. Cancel the parent context, and every agent, every LLM call, and every tool execution that descends from it receives the signal. This is critical for controlling costs — you don't want to keep paying for LLM API calls when the user has already disconnected.

Static Binary = Simple Deployment

Go compiles to a single binary with no runtime dependencies. No runtime to install, no library versions to manage. Your entire agent system — orchestrator, agents, tools, and HTTP server — is one file. Deploy it to a container, a VM, or serverless function. Startup time is measured in milliseconds.

The Agent Interface

At the heart of every agentic system is the Agent interface. This is the central abstraction that every agent implements — whether it's a simple agent that calls an LLM, a complex agent that uses tools, or an orchestrator that coordinates other agents. The beauty of this design is that all agents look the same from the outside. An orchestrator doesn't need to know whether it's talking to a simple LLM agent or a complex multi-step agent — it just calls Run() and gets a response.

The interface is deliberately small: just two methods. Run() executes the agent with a set of messages and returns a response. Name() identifies the agent (for logging and orchestration). Small interfaces are a core Go principle — the smaller the interface, the more types can satisfy it, and the more flexible your system becomes.

The Message struct represents a single message in a conversation. Every message has a role (system, user, assistant, or tool) and content. Messages can also contain tool calls (when the LLM wants to use a tool) and metadata (for tracking, logging, etc.). This is the same message format used by LLM APIs like OpenAI and Anthropic — our design mirrors theirs for natural compatibility.

The Response struct includes a HandOff field, which enables a powerful pattern: an agent can signal that another agent should take over. Think of it like a receptionist saying "I can't help with that, let me transfer you to the compliance department." This handoff mechanism is the foundation of the orchestration patterns we'll explore in Chapter 20.

package agent // all agent-related types live in the agent package

import "context" // for cancellation and timeout propagation

// Role represents who authored a message in the conversation.
// LLM APIs use these roles to understand the conversation structure.
type Role string // a string type alias for clarity and type safety

const (
    RoleSystem    Role = "system"    // instructions that define agent behavior ("you are a risk analyst...")
    RoleUser      Role = "user"      // messages from the end user or calling agent
    RoleAssistant Role = "assistant" // responses generated by the LLM
    RoleTool      Role = "tool"      // results returned from tool executions
)

// Message is a single entry in a conversation.
// It mirrors the message format used by LLM APIs (OpenAI, Anthropic, etc.).
type Message struct {
    Role       Role             `json:"role"`                  // who said this: system, user, assistant, or tool
    Content    string           `json:"content"`               // the actual text content of the message
    ToolCalls  []ToolCall       `json:"tool_calls,omitempty"`  // tools the LLM wants to invoke (if any)
    ToolCallID string           `json:"tool_call_id,omitempty"` // which tool call this message responds to
    Metadata   map[string]any `json:"metadata,omitempty"`    // extra data: token counts, timing, etc.
}

// ToolCall represents the LLM's request to invoke a specific tool.
// When the LLM needs information it doesn't have, it "calls" a tool.
type ToolCall struct {
    ID        string `json:"id"`        // unique identifier for this specific tool call
    Name      string `json:"name"`      // which tool to invoke (e.g., "database_lookup")
    Arguments string `json:"arguments"` // JSON-encoded arguments for the tool
}

// Agent is THE core interface of the entire system.
// Every agent — simple or complex — implements these two methods.
// This is intentionally small: small interfaces = maximum flexibility.
type Agent interface {
    // Run executes the agent with a conversation history and returns a response.
    // The context carries cancellation signals and deadlines.
    // The messages slice is the conversation so far.
    Run(ctx context.Context, messages []Message) (Response, error)

    // Name returns the agent's identifier (used for logging and orchestration).
    Name() string
}

// Response is what an agent returns after execution.
type Response struct {
    Messages []Message // the messages the agent produced (could be one or many)
    HandOff  *HandOff  // if not nil, this agent wants to transfer control to another agent
}

// HandOff signals that the current agent wants another agent to continue.
// Like a receptionist saying "let me transfer you to the right department."
type HandOff struct {
    TargetAgent string    // name of the agent to hand off to
    Context     []Message // the conversation context to pass along
    Reason      string    // why the handoff is happening (for logging/debugging)
}

The Agentic Loop: How an Agent Processes a Request

This is the most important concept in this entire guide. The agentic loop is the heartbeat of every LLM-powered agent. Understanding this loop is understanding how modern AI agents work. Let's walk through it step by step.

When an agent receives a request, it doesn't just make a single LLM call and return the result. Instead, it enters a loop. Here's the cycle:

  1. Build the conversation: Take the user's messages and prepend the system prompt (the agent's instructions, like "you are a vendor risk analyst").
  2. Call the LLM: Send the entire conversation to the LLM and get a response.
  3. Check for tool calls: Did the LLM ask to use any tools? If the LLM says "I need to look up vendor data in the database", that's a tool call.
  4. If no tool calls: The LLM has produced a final answer. Return it. Loop ends.
  5. If there are tool calls: Execute each tool (potentially in parallel using goroutines), collect the results, add them to the conversation, and go back to step 2.

This loop continues until the LLM produces a response without any tool calls, which means it has all the information it needs to answer. Think of it like a detective investigating a case: they form a hypothesis, gather evidence (tool calls), update their understanding, gather more evidence, and eventually reach a conclusion (the final response).

Here's the visual flow:

The Concrete Agent: Implementing the Loop

Now let's implement this loop in Go. The LLMAgent is a concrete struct that satisfies the Agent interface. It holds everything the agent needs: its name, system prompt, LLM client, tools, and configuration. The Run method implements the agentic loop described above.

Pay special attention to how tool calls are executed concurrently using goroutines. When the LLM requests three tools simultaneously (e.g., "search the database, parse this PDF, and check compliance"), we run all three in parallel. This is where Go's concurrency model shines — each tool call is a goroutine, results flow back through channels, and we wait for all of them to complete before continuing the loop.

// LLMAgent is a concrete agent powered by a Large Language Model.
// It implements the Agent interface and follows the agentic loop pattern.
type LLMAgent struct {
    name         string        // the agent's unique name (e.g., "risk-assessor")
    systemPrompt string        // instructions that define how this agent behaves
    llmClient    LLMClient     // the interface to communicate with the LLM
    tools        map[string]Tool // tools available to this agent, keyed by name
    maxTurns     int           // safety limit: max loop iterations to prevent infinite loops
}

// Name returns the agent's name (satisfies the Agent interface).
func (a *LLMAgent) Name() string {
    return a.name // simply return the stored name
}

// Run executes the agentic loop: call LLM → execute tools → repeat.
// This is THE core method — the heartbeat of the agent.
func (a *LLMAgent) Run(ctx context.Context, messages []Message) (Response, error) {
    // Step 1: Build the full conversation by prepending the system prompt.
    // The system message tells the LLM who it is and how to behave.
    conversation := make([]Message, 0, len(messages)+1) // pre-allocate capacity
    conversation = append(conversation, Message{            // add system prompt first
        Role:    RoleSystem,                                // this is a system-level instruction
        Content: a.systemPrompt,                             // the agent's behavioral instructions
    })
    conversation = append(conversation, messages...)         // then add all user/conversation messages

    // Step 2: Enter the agentic loop.
    // The loop continues until: the LLM gives a final answer, we hit
    // the max turns limit, or the context is cancelled.
    for turn := 0; turn < a.maxTurns; turn++ {
        // Step 2a: Check if the context has been cancelled (timeout or manual cancel).
        if err := ctx.Err(); err != nil {                  // check for cancellation
            return Response{}, fmt.Errorf("agent %s cancelled: %w", a.name, err)
        }

        // Step 2b: Call the LLM with the current conversation.
        // This is an API call that might take seconds — context carries the timeout.
        llmResp, err := a.llmClient.Complete(ctx, CompletionRequest{ // call the LLM
            Messages: conversation,                          // send the full conversation
            Tools:    a.toolDefinitions(),                   // tell the LLM what tools are available
        })
        if err != nil {                                       // if the LLM call failed...
            return Response{}, fmt.Errorf("agent %s LLM call failed: %w", a.name, err)
        }

        // Step 2c: Add the LLM's response to the conversation history.
        conversation = append(conversation, llmResp.Message) // record what the LLM said

        // Step 2d: Check if the LLM wants to use any tools.
        if len(llmResp.Message.ToolCalls) == 0 {              // no tool calls = final answer
            // The LLM has all the information it needs — return the response.
            return Response{Messages: conversation}, nil     // done! return the full conversation
        }

        // Step 2e: The LLM wants to use tools — execute them concurrently.
        // This is where Go's goroutines shine: multiple tool calls run in parallel.
        toolResults, err := a.executeToolsConcurrently(ctx, llmResp.Message.ToolCalls)
        if err != nil {                                       // if any tool execution failed...
            return Response{}, fmt.Errorf("agent %s tool execution: %w", a.name, err)
        }

        // Step 2f: Add tool results to the conversation, then loop back to step 2b.
        // The LLM will see the tool results and decide what to do next.
        conversation = append(conversation, toolResults...)  // add all tool results to the conversation
        // Loop continues: back to calling the LLM with the updated conversation.
    }

    // If we get here, we exceeded the max turns — something is wrong.
    return Response{}, fmt.Errorf("agent %s exceeded max turns (%d)", a.name, a.maxTurns)
}

// executeToolsConcurrently runs multiple tool calls in parallel using goroutines.
// This is a key advantage of Go: when the LLM requests 3 tools, all 3 run simultaneously.
func (a *LLMAgent) executeToolsConcurrently(ctx context.Context, calls []ToolCall) ([]Message, error) {
    // toolResult collects the result from one tool execution.
    type toolResult struct {
        msg Message // the tool's response as a Message
        err error   // any error that occurred during execution
    }

    results := make(chan toolResult, len(calls)) // buffered channel for collecting results

    // Launch each tool call as a separate goroutine.
    for _, call := range calls {                    // iterate over each tool call
        call := call                                  // capture the loop variable for the goroutine
        go func() {                                   // launch a goroutine for this tool call
            tool, ok := a.tools[call.Name]            // look up the tool by name
            if !ok {                                   // if the tool doesn't exist...
                results <- toolResult{                 // send an error result
                    err: fmt.Errorf("unknown tool: %s", call.Name),
                }
                return                                 // exit this goroutine
            }

            output, err := tool.Execute(ctx, call.Arguments) // execute the tool
            if err != nil {                              // if the tool failed...
                results <- toolResult{err: err}        // report the error
                return                                  // exit this goroutine
            }

            results <- toolResult{                     // send the successful result
                msg: Message{
                    Role:       RoleTool,               // this message is a tool result
                    Content:    output,                  // the tool's output text
                    ToolCallID: call.ID,                 // links this result to the original call
                },
            }
        }()
    }

    // Collect results from all goroutines.
    msgs := make([]Message, 0, len(calls))  // pre-allocate for all results
    for i := 0; i < len(calls); i++ {           // wait for exactly len(calls) results
        result := <-results                       // receive one result from the channel
        if result.err != nil {                     // if this tool call failed...
            return nil, result.err                 // return the error (fail fast)
        }
        msgs = append(msgs, result.msg)            // add the successful result to our list
    }

    return msgs, nil                               // return all tool results
}

A Complete Walkthrough: How an Agent Processes "Assess Vendor 42"

Let's trace through exactly what happens when a user sends "Assess vendor 42" to a risk assessment agent equipped with a database lookup tool and a web search tool.

Turn 1: The conversation starts with the system prompt ("You are a risk analyst...") plus the user message ("Assess vendor 42"). The LLM receives this and responds: "I need to look up vendor 42's data. Let me search the database." Along with this text, the LLM's response includes a tool call: {name: "database_lookup", arguments: '{"vendor_id": "v-42"}'}.

Tool execution: The agent sees the tool call, finds the "database_lookup" tool in its registry, and executes it. The tool queries the database and returns: "Vendor: Acme Corp, Industry: Cloud Services, Last Audit: 2024-01, SOC 2: Yes, Incident History: 2 breaches in 3 years."

Turn 2: The tool result is added to the conversation, and the LLM is called again with the full history. Now the LLM has the vendor data and responds: "Based on the data, I want to check recent news about this vendor." Another tool call: {name: "web_search", arguments: '{"query": "Acme Corp security breach 2024"}'}.

Tool execution: The web search tool runs and returns recent articles about the vendor.

Turn 3: With both the database data and the news results in the conversation, the LLM now has enough information. It responds with its final assessment: "Vendor 42 (Acme Corp) has a risk score of 72 (Medium). While they have SOC 2 certification, two recent security incidents and delayed remediation raise concerns..." This response has no tool calls, so the loop ends. The agent returns this as the final response.

The loop is self-correcting
If a tool call returns an error (like "database connection failed"), the error message is added to the conversation. The LLM sees this and can adapt: it might retry with different parameters, try a different tool, or produce a partial answer noting that some data was unavailable. This self-correction capability is what makes agents more resilient than simple scripted workflows.

LLM Integration

What Is an LLM?

A Large Language Model (LLM) is an AI system trained on vast amounts of text that can understand and generate human-like language. Think of it as a very sophisticated autocomplete engine: given a conversation, it predicts the most likely (and most helpful) next response. Models like GPT-4, Claude, and Gemini are LLMs. In an agentic system, the LLM is the "brain" — it reasons about problems, decides which tools to use, and generates responses. But it can only think; it cannot act on its own. That's what tools are for (next chapter).

In code, communicating with an LLM means making an API call: you send a conversation (a list of messages) and receive a response. The API call is essentially an HTTP request to the provider's server (OpenAI, Anthropic, etc.) with JSON payloads. This is fundamentally an I/O operation, which makes Go an ideal choice — Go excels at managing many concurrent I/O operations efficiently.

Why Abstract Behind an Interface

Different LLM providers (OpenAI, Anthropic, Google, local models) have different APIs, different request formats, and different response structures. But from your agent's perspective, the operation is always the same: "send messages, get a response." By defining an interface that captures this common behavior, you can swap providers without changing any agent code. Today you use OpenAI; tomorrow you switch to Anthropic. The agents don't know and don't care.

This is the power of interfaces in Go: they create a boundary between "what you need" (a completion) and "how it's done" (OpenAI API, Anthropic API, a local model, or even a mock for testing). Every component in your system depends on the interface, never on a specific implementation.

// LLMClient is the interface that all LLM providers must implement.
// By coding to this interface, agents can work with ANY LLM provider.
// Today it's OpenAI; tomorrow it could be Anthropic or a local model.
type LLMClient interface {
    // Complete sends a conversation to the LLM and returns its response.
    // The context carries cancellation/timeout signals.
    Complete(ctx context.Context, req CompletionRequest) (CompletionResponse, error)
}

// CompletionRequest is what we send to the LLM.
type CompletionRequest struct {
    Model       string          `json:"model"`       // which model to use (e.g., "gpt-4o")
    Messages    []Message       `json:"messages"`    // the conversation history
    Tools       []ToolDef       `json:"tools,omitempty"` // tools the LLM can call
    Temperature float64         `json:"temperature"` // creativity level (0.0 to 1.0)
    MaxTokens   int             `json:"max_tokens"`  // max tokens in the response
}

// CompletionResponse is what the LLM sends back to us.
type CompletionResponse struct {
    Message    Message // the LLM's response message (may include tool calls)
    TokensUsed int     // how many tokens were consumed (for cost tracking)
}

// ToolDef describes a tool to the LLM so it knows what tools it can call.
// This is sent with the request so the LLM understands its available tools.
type ToolDef struct {
    Name        string `json:"name"`        // the tool's name (e.g., "database_lookup")
    Description string `json:"description"` // what the tool does (helps the LLM decide when to use it)
    Parameters  any    `json:"parameters"`  // JSON Schema describing the tool's input parameters
}

OpenAI Implementation

Here's a concrete implementation of the LLMClient interface that talks to OpenAI's API. It constructs an HTTP request with the right headers and JSON body, sends it, and parses the response. Notice how the context is threaded through every operation — if the caller cancels, the HTTP request is cancelled too.

// OpenAIClient implements LLMClient for OpenAI's API.
type OpenAIClient struct {
    apiKey     string       // your OpenAI API key for authentication
    baseURL    string       // the API endpoint (default: "https://api.openai.com/v1")
    httpClient *http.Client // the HTTP client used to make API calls
    model      string       // the default model to use (e.g., "gpt-4o")
}

// NewOpenAIClient creates a new OpenAI client with the given API key.
func NewOpenAIClient(apiKey string) *OpenAIClient {
    return &OpenAIClient{
        apiKey:  apiKey,                              // store the API key for authentication
        baseURL: "https://api.openai.com/v1",         // default API endpoint
        httpClient: &http.Client{                     // create an HTTP client with a timeout
            Timeout: 60 * time.Second,                // don't wait more than 60 seconds for a response
        },
        model: "gpt-4o",                              // default model
    }
}

// Complete sends messages to OpenAI and returns the LLM's response.
// This is where the actual API communication happens.
func (c *OpenAIClient) Complete(ctx context.Context, req CompletionRequest) (CompletionResponse, error) {
    // Step 1: Use the request's model, or fall back to the default.
    model := req.Model                                // check if the request specifies a model
    if model == "" {                                    // if no model specified in the request...
        model = c.model                                // use the client's default model
    }

    // Step 2: Build the OpenAI API request body.
    apiReq := map[string]any{                        // create the JSON request body
        "model":       model,                         // which model to use
        "messages":    req.Messages,                    // the conversation history
        "temperature": req.Temperature,                 // creativity level
        "max_tokens":  req.MaxTokens,                   // response length limit
    }
    if len(req.Tools) > 0 {                             // if tools are available...
        apiReq["tools"] = req.Tools                     // include them in the request
    }

    // Step 3: Encode the request body as JSON.
    body, err := json.Marshal(apiReq)                  // convert our map to JSON bytes
    if err != nil {                                      // if JSON encoding failed...
        return CompletionResponse{}, fmt.Errorf("marshal request: %w", err)
    }

    // Step 4: Create the HTTP request with context for cancellation.
    httpReq, err := http.NewRequestWithContext(ctx,     // create request with context
        http.MethodPost,                                // POST method
        c.baseURL+"/chat/completions",                 // the API endpoint
        bytes.NewReader(body),                          // the JSON body
    )
    if err != nil {                                      // if request creation failed...
        return CompletionResponse{}, fmt.Errorf("create request: %w", err)
    }
    httpReq.Header.Set("Content-Type", "application/json") // tell the server we're sending JSON
    httpReq.Header.Set("Authorization", "Bearer "+c.apiKey) // authenticate with our API key

    // Step 5: Send the HTTP request and get the response.
    httpResp, err := c.httpClient.Do(httpReq)           // execute the HTTP request
    if err != nil {                                       // if the HTTP call failed...
        return CompletionResponse{}, fmt.Errorf("API call: %w", err)
    }
    defer httpResp.Body.Close()                          // always close the response body when done

    // Step 6: Check for HTTP errors (rate limiting, server errors, etc.).
    if httpResp.StatusCode != http.StatusOK {              // if not 200 OK...
        body, _ := io.ReadAll(httpResp.Body)              // read the error body for details
        return CompletionResponse{}, fmt.Errorf(          // return a descriptive error
            "OpenAI API error %d: %s",                     // include status code and body
            httpResp.StatusCode, string(body),
        )
    }

    // Step 7: Decode the JSON response from OpenAI.
    var apiResp openAIResponse                             // struct matching OpenAI's JSON format
    if err := json.NewDecoder(httpResp.Body).Decode(&apiResp); err != nil {
        return CompletionResponse{}, fmt.Errorf("decode response: %w", err)
    }

    // Step 8: Convert OpenAI's response format to our internal format.
    return CompletionResponse{
        Message:    convertToMessage(apiResp),             // convert to our Message type
        TokensUsed: apiResp.Usage.TotalTokens,             // extract token usage for cost tracking
    }, nil
}

// openAIResponse matches the JSON structure returned by OpenAI's API.
type openAIResponse struct {
    Choices []struct {                                     // OpenAI returns an array of choices
        Message struct {                                    // each choice has a message
            Role       string     `json:"role"`          // who generated this message
            Content    string     `json:"content"`       // the text content
            ToolCalls  []struct {                          // any tool calls the LLM wants to make
                ID       string `json:"id"`              // unique ID for this tool call
                Function struct {                         // the function to call
                    Name      string `json:"name"`        // tool name
                    Arguments string `json:"arguments"`   // JSON-encoded arguments
                } `json:"function"`
            } `json:"tool_calls,omitempty"`
        } `json:"message"`
    } `json:"choices"`
    Usage struct {                                          // token usage statistics
        TotalTokens int `json:"total_tokens"`              // total tokens consumed
    } `json:"usage"`
}

Streaming Responses with Channels

For real-time user experiences, you often want to stream the LLM's response token by token instead of waiting for the entire response. Go channels are perfect for this — the LLM integration goroutine reads from the API stream and sends each chunk into a channel. The consumer reads from the channel and displays tokens as they arrive. It's like watching someone type a response in real time instead of waiting for them to finish the whole paragraph.

// StreamingLLMClient extends the basic client with streaming support.
type StreamingLLMClient interface {
    LLMClient                                          // embed the basic interface (still need Complete)

    // Stream sends messages to the LLM and returns a channel that receives
    // response chunks as they arrive, one token at a time.
    Stream(ctx context.Context, req CompletionRequest) (<-chan StreamChunk, error)
}

// StreamChunk is one piece of the streaming response.
type StreamChunk struct {
    Content    string     // a fragment of the response text (usually a few tokens)
    ToolCalls  []ToolCall // tool calls can also arrive in chunks
    Done       bool       // true when the stream is complete
    Err        error      // any error that occurred during streaming
}

// Stream implements real-time token-by-token response streaming.
func (c *OpenAIClient) Stream(ctx context.Context, req CompletionRequest) (<-chan StreamChunk, error) {
    chunks := make(chan StreamChunk)                    // create unbuffered channel for chunks

    // Launch a goroutine that reads from the API stream and sends chunks.
    go func() {
        defer close(chunks)                               // close the channel when we're done

        // ... (set up streaming HTTP request to OpenAI)

        // Read Server-Sent Events (SSE) from the API response.
        scanner := bufio.NewScanner(httpResp.Body)        // read the response line by line
        for scanner.Scan() {                               // for each line in the stream...
            line := scanner.Text()                         // get the line text
            if !strings.HasPrefix(line, "data: ") {         // skip non-data lines
                continue                                    // SSE format: data lines start with "data: "
            }

            data := strings.TrimPrefix(line, "data: ")   // extract the JSON data
            if data == "[DONE]" {                           // OpenAI signals end with [DONE]
                chunks <- StreamChunk{Done: true}          // tell the consumer we're done
                return                                       // exit the goroutine
            }

            var event streamEvent                          // parse the SSE event
            json.Unmarshal([]byte(data), &event)           // decode the JSON chunk

            select {
            case chunks <- StreamChunk{Content: event.Delta}: // send the chunk to the consumer
            case <-ctx.Done():                              // if the context was cancelled...
                return                                       // stop streaming immediately
            }
        }
    }()

    return chunks, nil                                    // return the channel immediately
}
Interface-based design enables testing and swapping
Because agents depend on the LLMClient interface (not on OpenAIClient directly), you can: (1) swap OpenAI for Anthropic by just changing one line in main.go, (2) use a mock client in tests that returns instant, predictable responses, and (3) add a rate-limited wrapper without changing any agent code. This is the power of programming to interfaces.

Tool Calling System

What Are Tools?

An LLM can only think — it can reason about information, analyze text, and generate responses. But it can't do anything in the real world. It can't query a database, search the web, read a PDF, or send an email. Tools are the agent's hands. They bridge the gap between the LLM's thinking ability and actual actions in the world.

Think of it like this: you're sitting in a room with a brilliant consultant (the LLM). The consultant can analyze any information you give them and provide expert advice. But they can't leave the room. If they need data that isn't in the room, they write a note saying "please look up X" and hand it to you. You go fetch the data, bring it back, and they continue their analysis. That's exactly how tool calling works: the LLM identifies what information it needs, generates a tool call (the note), the tool executes and returns the result, and the LLM continues reasoning.

In an agent system, tools give agents the ability to interact with real-world systems: querying databases, parsing PDF reports, checking compliance frameworks, searching the web for recent incidents, and calculating risk scores. Without tools, an agent can only work with the information in its training data and the current conversation. With tools, it can access live, real-time information.

The Tool Interface

Every tool implements a simple interface with three responsibilities: describe itself (so the LLM knows what it does and how to call it), execute (actually do the work), and return results as text (so the LLM can read and reason about them).

// Tool is the interface that every tool must implement.
// Tools are the agent's ability to interact with the outside world.
type Tool interface {
    // Name returns the tool's identifier (e.g., "database_lookup").
    // The LLM uses this name to request the tool.
    Name() string

    // Description returns a human-readable explanation of what the tool does.
    // The LLM reads this to decide when to use the tool.
    Description() string

    // Parameters returns the JSON Schema describing the tool's input.
    // This tells the LLM what arguments to provide when calling the tool.
    Parameters() any

    // Execute runs the tool with the given JSON arguments and returns the result.
    // The arguments come from the LLM and match the Parameters schema.
    // The result is always a string that the LLM can read and reason about.
    Execute(ctx context.Context, arguments string) (string, error)
}

The Tool Registry

A registry is a catalog of all available tools, organized by name. When the LLM says "I want to use the database_lookup tool," the agent looks it up in the registry by name, executes it, and returns the result. Think of it as a phone book for tools — you look up the name and find the right tool to call.

// ToolRegistry is a catalog of all available tools, indexed by name.
// Agents use the registry to find and execute tools requested by the LLM.
type ToolRegistry struct {
    tools map[string]Tool // map from tool name to tool implementation
}

// NewToolRegistry creates an empty registry.
func NewToolRegistry() *ToolRegistry {
    return &ToolRegistry{
        tools: make(map[string]Tool), // initialize an empty map
    }
}

// Register adds a tool to the registry.
// After registration, agents can find and use this tool.
func (r *ToolRegistry) Register(tool Tool) {
    r.tools[tool.Name()] = tool // store the tool using its name as the key
}

// Get retrieves a tool by name. Returns the tool and whether it was found.
func (r *ToolRegistry) Get(name string) (Tool, bool) {
    tool, ok := r.tools[name] // look up the tool in our map
    return tool, ok            // return the tool and whether it was found
}

// Definitions returns tool definitions for all registered tools.
// These definitions are sent to the LLM so it knows what tools are available.
func (r *ToolRegistry) Definitions() []ToolDef {
    defs := make([]ToolDef, 0, len(r.tools)) // pre-allocate for efficiency
    for _, tool := range r.tools {                // iterate over all registered tools
        defs = append(defs, ToolDef{                // create a definition for each tool
            Name:        tool.Name(),               // the tool's name
            Description: tool.Description(),        // what it does
            Parameters:  tool.Parameters(),         // what arguments it accepts
        })
    }
    return defs                                    // return all definitions
}

Multi-Agent Orchestration

What Is Orchestration?

Think of an orchestra. You have violinists, cellists, flautists, percussionists — each an expert at their instrument. But without a conductor, they'd all play at their own tempo, in their own key, creating cacophony. The conductor coordinates them: decides who plays when, ensures they're in sync, and shapes individual performances into a unified masterpiece.

Multi-agent orchestration is the same concept applied to AI agents. Each agent is a specialist (risk assessor, document analyzer, compliance checker), but without orchestration, they'd work independently with no coordination. An orchestrator decides: Which agents should run? In what order? Should some run in parallel? How do results flow between them? What happens when one fails?

There are four main orchestration patterns, each suited for different situations. Understanding when to use each pattern is as important as knowing how to implement them. Let's explore each one with real-world analogies, detailed explanations, and fully commented code.

Pattern 1: Sequential Orchestration

Sequential orchestration is the simplest pattern: agents run one after another, like an assembly line in a factory. Agent A finishes its work and passes the result to Agent B. Agent B finishes and passes to Agent C. Each agent builds on the work of the previous one.

When to use it: When each step depends on the output of the previous step. You can't analyze a document before you've parsed it. You can't generate a report before you've gathered all the findings. If there's a strict dependency chain, use sequential orchestration.

Real-world analogy: A relay race — each runner completes their leg before handing the baton to the next.

Sequential Orchestration (Assembly Line) User Request │ ▼ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Agent A │ ──→ │ Agent B │ ──→ │ Agent C │ │ (Intake) │ │ (Analysis) │ │ (Report) │ └─────────────┘ └─────────────┘ └─────────────┘ │ ▼ Final Response
// SequentialOrchestrator runs agents one after another.
// Each agent's output becomes the next agent's input.
// Like an assembly line: each station does its work and passes the result forward.
type SequentialOrchestrator struct {
    agents []Agent // the ordered list of agents to run sequentially
}

// NewSequentialOrchestrator creates a new orchestrator with the given agents.
// Agents will be executed in the order they are provided.
func NewSequentialOrchestrator(agents ...Agent) *SequentialOrchestrator {
    return &SequentialOrchestrator{agents: agents} // store the agent list
}

// Run executes each agent in sequence, passing results forward.
func (o *SequentialOrchestrator) Run(ctx context.Context, input []Message) (Response, error) {
    current := input // start with the original input messages

    for _, agent := range o.agents { // run each agent in order
        slog.Info("sequential: running agent", // log which agent is running
            "agent", agent.Name(),         // the agent's name for debugging
        )

        resp, err := agent.Run(ctx, current) // run this agent with the current messages
        if err != nil {                       // if this agent failed...
            return Response{}, fmt.Errorf(    // wrap the error with context
                "agent %s failed: %w",          // include which agent failed
                agent.Name(), err,
            )
        }

        current = resp.Messages // this agent's output becomes the next agent's input
    }

    return Response{Messages: current}, nil // return the final agent's output
}

// Name identifies this orchestrator for logging and nesting.
func (o *SequentialOrchestrator) Name() string {
    return "sequential-orchestrator" // a human-readable name
}

Pattern 2: Parallel Orchestration

Parallel orchestration runs multiple agents simultaneously and combines their results. This is like a team of researchers all investigating different aspects of the same question at the same time. Instead of one researcher doing everything sequentially, five researchers each tackle one area in parallel, then their findings are merged.

When to use it: When multiple agents can work independently on the same input. If risk assessment, document analysis, and compliance checking don't depend on each other, run them all at once. This can cut your total processing time dramatically — instead of waiting for each agent sequentially (30s + 30s + 30s = 90s), they all run concurrently (max 30s total).

Real-world analogy: A group of food critics each reviewing the same restaurant independently, then combining their reviews into a comprehensive assessment.

Parallel Orchestration (Fan-out / Fan-in) User Request │ ┌─────────┼─────────┐ │ │ │ ▼ ▼ ▼ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ Agent A │ │ Agent B │ │ Agent C │ │ (Risk) │ │ (Docs) │ │(Comply) │ └────┬─────┘ └────┬─────┘ └────┬─────┘ │ │ │ └─────────┼─────────┘ │ ▼ ┌────────────┐ │ Merger │ ← Combines all results └─────┬──────┘ │ ▼ Final Response
// agentResult holds one agent's output from a parallel run.
type agentResult struct {
    agentName string   // which agent produced this result
    response  Response // the agent's output
    err       error    // any error that occurred
}

// ResultMerger combines results from multiple agents into a single response.
// Different merging strategies produce different final outputs.
type ResultMerger interface {
    Merge(ctx context.Context, results []agentResult) (Response, error)
}

// ParallelOrchestrator runs all agents simultaneously and merges results.
type ParallelOrchestrator struct {
    agents []Agent        // agents to run in parallel
    merger ResultMerger   // strategy for combining results
}

// NewParallelOrchestrator creates a parallel orchestrator.
func NewParallelOrchestrator(merger ResultMerger, agents ...Agent) *ParallelOrchestrator {
    return &ParallelOrchestrator{agents: agents, merger: merger}
}

// Run executes all agents concurrently using goroutines.
// This is where Go's concurrency model truly shines.
func (o *ParallelOrchestrator) Run(ctx context.Context, input []Message) (Response, error) {
    // Create a buffered channel to collect results from all agents.
    // Buffer size = number of agents, so no goroutine blocks on send.
    results := make(chan agentResult, len(o.agents))

    // Fan-out: launch each agent in its own goroutine.
    // All agents start executing simultaneously.
    for _, agent := range o.agents {       // iterate over all agents
        agent := agent                       // capture the loop variable for the goroutine
        go func() {                          // launch a goroutine for this agent
            slog.Info("parallel: starting agent", // log which agent is starting
                "agent", agent.Name(),
            )
            resp, err := agent.Run(ctx, input) // run the agent (all share the same input)
            results <- agentResult{             // send the result to the channel
                agentName: agent.Name(),        // which agent produced this
                response:  resp,                // the agent's output
                err:       err,                 // any error
            }
        }()
    }

    // Fan-in: collect results from all agents.
    // We wait for exactly len(agents) results.
    collected := make([]agentResult, 0, len(o.agents)) // pre-allocate
    for i := 0; i < len(o.agents); i++ {  // wait for each agent to finish
        select {
        case r := <-results:                 // receive one result
            collected = append(collected, r)  // add it to our collection
        case <-ctx.Done():                   // if the context was cancelled...
            return Response{}, ctx.Err()       // stop waiting, return the cancellation error
        }
    }

    // Merge all results into a single response using the merger strategy.
    return o.merger.Merge(ctx, collected) // combine results from all agents
}

// Name identifies this orchestrator.
func (o *ParallelOrchestrator) Name() string {
    return "parallel-orchestrator"
}

Pattern 3: Supervisor Orchestration

In supervisor orchestration, one "boss" agent (the supervisor) decides which worker agent should handle each task. The supervisor receives a request, analyzes it, and delegates to the appropriate specialist. After the specialist finishes, the supervisor reviews the result and decides what to do next: assign another task, request more work, or return the final result.

When to use it: When the workflow isn't fixed — the choice of which agent to run depends on the input. A triage system might need to route some requests to a quick assessment and others to a full deep-dive, based on the vendor tier. The supervisor agent makes this decision dynamically.

Real-world analogy: A hospital emergency room triage nurse who examines each patient and decides which specialist they need to see.

Supervisor Orchestration (Boss → Workers) User Request │ ▼ ┌──────────────┐ │ Supervisor │ ← Decides who should handle the task │ Agent │ └──────┬───────┘ │ HandOff decision ┌────────┼────────┐ │ │ │ ▼ ▼ ▼ ┌─────────┐ ┌──────┐ ┌──────────┐ │ Risk │ │ Docs │ │Compliance│ │ Agent │ │Agent │ │ Agent │ └────┬────┘ └──┬───┘ └────┬─────┘ │ │ │ └─────────┼──────────┘ │ ▼ ┌──────────────┐ │ Supervisor │ ← Reviews result, may delegate more │ (again) │ └──────┬───────┘ │ ▼ Final Response
// SupervisorOrchestrator uses a supervisor agent to delegate work.
// The supervisor decides which agent handles each task.
type SupervisorOrchestrator struct {
    supervisor  Agent              // the boss agent that makes delegation decisions
    agents      map[string]Agent // available worker agents, keyed by name
    maxHandoffs int                // safety limit to prevent infinite delegation loops
}

// Run lets the supervisor delegate work to specialist agents.
func (o *SupervisorOrchestrator) Run(ctx context.Context, input []Message) (Response, error) {
    current := input // the current conversation state

    for i := 0; i < o.maxHandoffs; i++ { // loop until done or safety limit reached
        // Step 1: Ask the supervisor what to do with the current messages.
        resp, err := o.supervisor.Run(ctx, current) // supervisor analyzes and decides
        if err != nil {                               // if supervisor failed...
            return Response{}, fmt.Errorf("supervisor: %w", err)
        }

        // Step 2: If no handoff, the supervisor has the final answer.
        if resp.HandOff == nil {                      // no delegation needed
            return resp, nil                            // return the supervisor's response as final
        }

        // Step 3: Supervisor wants to delegate — find the target agent.
        slog.Info("supervisor delegating",               // log the delegation
            "to", resp.HandOff.TargetAgent,              // which agent was chosen
            "reason", resp.HandOff.Reason,               // why this agent was chosen
        )
        target, ok := o.agents[resp.HandOff.TargetAgent] // look up the target agent
        if !ok {                                         // if the agent doesn't exist...
            return Response{}, fmt.Errorf(                 // return an error
                "supervisor requested unknown agent: %s",
                resp.HandOff.TargetAgent,
            )
        }

        // Step 4: Run the delegated agent with the context from the handoff.
        agentResp, err := target.Run(ctx, resp.HandOff.Context) // execute the specialist
        if err != nil {                                           // if the specialist failed...
            return Response{}, fmt.Errorf("agent %s: %w", resp.HandOff.TargetAgent, err)
        }

        // Step 5: Feed the specialist's results back to the supervisor.
        // The supervisor will review and decide if more work is needed.
        current = agentResp.Messages // update the conversation with specialist's output
    }

    // Safety: if we reach here, too many delegations happened.
    return Response{}, fmt.Errorf("max handoffs (%d) exceeded", o.maxHandoffs)
}

// Name identifies this orchestrator.
func (o *SupervisorOrchestrator) Name() string {
    return "supervisor-orchestrator"
}

Pattern 4: Handoff Orchestration

In handoff orchestration, there is no central boss. Instead, agents pass work to each other directly. When Agent A finishes its part, it decides which agent should continue and hands off the conversation. It's like a group of colleagues working on a problem: "I've done the financial analysis, you take over for the technical review."

When to use it: When agents are peers (no hierarchy) and the natural flow of work passes between them. This pattern is very flexible but can be harder to debug because there's no central coordinator tracking the overall state.

Real-world analogy: A group of doctors in a multi-specialty clinic. The general practitioner examines you first, then hands you off to a cardiologist, who might hand you off to a surgeon. Each doctor decides the next step based on their findings.

Handoff Orchestration (Peer-to-Peer) ┌─────────┐ HandOff ┌─────────┐ HandOff ┌─────────┐ │ Agent A │ ──────────→ │ Agent B │ ──────────→ │ Agent C │ │ (Triage) │ │ (Review) │ │ (Report) │ └─────────┘ └─────────┘ └────┬────┘ │ ▼ No HandOff = Done Final Response
// HandoffOrchestrator lets agents pass work to each other directly.
// There is no central boss — each agent decides who should go next.
type HandoffOrchestrator struct {
    agents   map[string]Agent // all available agents, keyed by name
    maxTurns int                // safety limit: max total handoffs allowed
}

// NewHandoffOrchestrator creates an orchestrator with the given agents.
func NewHandoffOrchestrator(maxTurns int, agents ...Agent) *HandoffOrchestrator {
    agentMap := make(map[string]Agent)  // create a map to look up agents by name
    for _, a := range agents {             // populate the map from the provided list
        agentMap[a.Name()] = a              // use the agent's name as the key
    }
    return &HandoffOrchestrator{
        agents:   agentMap,                  // store the agent map
        maxTurns: maxTurns,                  // store the safety limit
    }
}

// Run starts with the specified agent and follows handoffs until completion.
func (o *HandoffOrchestrator) Run(ctx context.Context, startAgent string, input []Message) (Response, error) {
    currentAgent := startAgent  // which agent is currently running
    conversation := input       // the current conversation state

    for turn := 0; turn < o.maxTurns; turn++ { // loop through turns until done
        // Step 1: Find the current agent by name.
        agent, ok := o.agents[currentAgent]     // look up the agent
        if !ok {                                 // if the agent doesn't exist...
            return Response{}, fmt.Errorf("agent not found: %s", currentAgent)
        }

        // Step 2: Run the current agent.
        slog.Info("handoff: running agent",        // log which agent is running
            "agent", currentAgent,                // the agent's name
            "turn", turn,                          // which turn number we're on
        )
        resp, err := agent.Run(ctx, conversation) // execute the agent
        if err != nil {                             // if the agent failed...
            return Response{}, fmt.Errorf("agent %s (turn %d): %w", currentAgent, turn, err)
        }

        // Step 3: Check if the agent wants to hand off to another agent.
        if resp.HandOff == nil {                   // no handoff = this agent is done
            return resp, nil                         // return the final response
        }

        // Step 4: Hand off to the next agent.
        slog.Info("handoff: transferring",           // log the handoff for debugging
            "from", currentAgent,                   // which agent is handing off
            "to", resp.HandOff.TargetAgent,         // which agent receives the work
            "reason", resp.HandOff.Reason,          // why the handoff is happening
        )
        currentAgent = resp.HandOff.TargetAgent   // switch to the new agent
        conversation = resp.HandOff.Context        // use the handoff context as the new conversation
    }

    // Safety: too many handoffs — something is likely wrong.
    return Response{}, fmt.Errorf("max turns (%d) exceeded", o.maxTurns)
}

// Name identifies this orchestrator.
func (o *HandoffOrchestrator) Name() string {
    return "handoff-orchestrator"
}

When to Use Which Pattern

Choosing the right orchestration pattern depends on your specific use case. Here's a decision guide:

Sequential

Use when: Steps have strict dependencies. Output of step N is input to step N+1.
Example: Parse document → Extract data → Score risk → Generate report.
Pros: Simple, predictable, easy to debug.
Cons: Slowest (no parallelism).

Parallel

Use when: Multiple independent analyses on the same input.
Example: Run risk, compliance, and document analysis simultaneously.
Pros: Fast (wall-clock time = slowest single agent).
Cons: Need a merge strategy; harder to handle partial failures.

Supervisor

Use when: The workflow depends on the input and a central coordinator should decide.
Example: Triage incoming requests to the appropriate specialist agent.
Pros: Flexible, centralized control, easy to add new workers.
Cons: Supervisor is a single point of failure; supervisor LLM calls add latency.

Handoff

Use when: Agents are peers that naturally pass work between each other.
Example: Triage agent → Risk agent → Compliance agent → Report agent, each deciding the next step.
Pros: Very flexible, decentralized, agents are autonomous.
Cons: Harder to debug; risk of infinite loops without turn limits.

Combining Patterns: Full Assessment Orchestrator

In practice, you often combine multiple patterns. An assessment might use sequential orchestration at the top level (intake → analysis → report), with parallel orchestration inside the analysis phase (risk + docs + compliance run simultaneously). This gives you the best of both worlds: clear overall flow with maximum concurrency where it matters.

// AssessmentOrchestrator combines sequential and parallel patterns for a full assessment.
// Phase 1 (sequential): Intake agent gathers initial information.
// Phase 2 (parallel): Risk, document, and compliance agents run simultaneously.
// Phase 3 (sequential): Report agent compiles everything into a final report.
type AssessmentOrchestrator struct {
    intake     Agent // Phase 1: gathers vendor information
    riskAgent  Agent // Phase 2a: analyzes risk factors
    docAgent   Agent // Phase 2b: analyzes vendor documents
    compliance Agent // Phase 2c: checks compliance frameworks
    reporter   Agent // Phase 3: generates the final report
}

// RunAssessment orchestrates a full vendor risk assessment.
func (o *AssessmentOrchestrator) RunAssessment(ctx context.Context, vendorID string) (*Report, error) {
    // Phase 1: Intake (sequential — must complete before analysis begins).
    slog.Info("starting intake", "vendor", vendorID) // log the start of phase 1
    intakeResult, err := o.intake.Run(ctx, []Message{ // run the intake agent
        {Role: RoleUser, Content: fmt.Sprintf("Process vendor intake for: %s", vendorID)},
    })
    if err != nil {                                     // if intake failed...
        return nil, fmt.Errorf("intake failed: %w", err)  // report the failure
    }

    // Phase 2: Parallel analysis (three agents run simultaneously).
    // errgroup manages concurrent goroutines with error propagation.
    // If any agent fails, the group's context is cancelled (stopping the others).
    slog.Info("starting parallel analysis", "vendor", vendorID) // log phase 2
    g, gCtx := errgroup.WithContext(ctx)                     // create an error group with derived context
    var riskResult, docResult, complianceResult Response      // variables to hold each agent's output

    g.Go(func() error {                                      // launch risk agent in a goroutine
        var err error                                         // declare err in this scope
        riskResult, err = o.riskAgent.Run(gCtx, intakeResult.Messages) // run risk analysis
        return err                                            // return error (or nil for success)
    })
    g.Go(func() error {                                      // launch document agent in a goroutine
        var err error
        docResult, err = o.docAgent.Run(gCtx, intakeResult.Messages) // run document analysis
        return err
    })
    g.Go(func() error {                                      // launch compliance agent in a goroutine
        var err error
        complianceResult, err = o.compliance.Run(gCtx, intakeResult.Messages) // run compliance check
        return err
    })

    if err := g.Wait(); err != nil {                          // wait for all three to finish
        return nil, fmt.Errorf("parallel analysis failed: %w", err) // report first error
    }

    // Phase 3: Report generation (sequential — needs all results).
    slog.Info("generating report", "vendor", vendorID)     // log phase 3
    combined := mergeResults(riskResult, docResult, complianceResult) // combine all findings
    reportResult, err := o.reporter.Run(ctx, combined)        // generate the report
    if err != nil {                                             // if report generation failed...
        return nil, fmt.Errorf("report generation failed: %w", err)
    }

    return parseReport(reportResult)                           // parse the final report and return
}

// Name identifies this orchestrator.
func (o *AssessmentOrchestrator) Name() string {
    return "assessment-orchestrator"
}
Architecture principles for production agent systems
  1. Single Responsibility Agents — Each agent does one thing well. No "god agents" that try to do everything.
  2. Idempotent Tool Execution — Tools should be safe to retry. LLM calls are non-deterministic, so tools might be called more than once.
  3. Timeout Everything — Use context.WithTimeout for every LLM call, tool execution, and agent run. Never let anything run forever.
  4. Graceful Degradation — If the risk agent fails, still produce a partial report with the data you have.
  5. Cost Tracking — Track token usage per agent per request. LLM APIs are expensive, and runaway loops can drain budgets fast.

Agent Memory & State

What Is Agent Memory?

Imagine you walk into a doctor's office. The doctor asks about your symptoms, reviews your medical history, and then prescribes a treatment. Now imagine a different doctor — one with amnesia. Every time you speak a sentence, they forget everything you said before. They cannot connect your headache to the medication you mentioned 30 seconds ago. They cannot look up your chart from last year's visit. This doctor is useless.

An agent without memory is that amnesic doctor. Every time you call it, it starts completely blank — no conversation history, no knowledge of what other agents discovered, no recollection of past assessments. Memory is what transforms a stateless function call into an intelligent, context-aware system that improves over time.

Agent memory comes in three flavors, each serving a different purpose:

  • Short-term memory (conversation context) — Like your working memory when having a conversation. It holds the current exchange: what the user asked, what the agent replied, what tools were called. This lives for the duration of one interaction and is discarded afterward.
  • Working memory (shared state) — Like a shared whiteboard in a team meeting. When multiple agents collaborate on a single task, they need a place to write findings that other agents can read. The risk agent discovers a critical vulnerability and writes it to the whiteboard; the compliance agent reads it when mapping controls.
  • Long-term memory (persistent knowledge) — Like notes in a filing cabinet. Past vendor assessments, historical risk scores, previous findings — all stored permanently and retrievable later. When assessing a vendor for the second time, the agent can look up what it found last year.
The filing cabinet analogy
Think of short-term memory as the sticky notes on your desk (temporary, small, discarded after use). Working memory is the whiteboard in a conference room (shared, visible to everyone in the meeting, erased when the meeting ends). Long-term memory is the filing cabinet in the back office (permanent, organized, searchable when you need historical context).

The MemoryStore Interface

In Go, we define a MemoryStore interface that describes what memory operations are available, without dictating how they work. This lets us swap between an in-memory implementation (fast, great for tests), a Redis implementation (fast, shared across servers), or a PostgreSQL implementation (durable, searchable) without changing any agent code.

The interface has three core operations: save messages (write conversation history), load messages (retrieve conversation history), and search (find relevant past memories by semantic similarity). Every agent that needs memory depends on this interface, never on a specific database.

package memory                          // memory package groups all memory-related code

import (
    "context"                            // context carries deadlines and cancellation signals
    "time"                               // time provides duration and timestamp types
)

// MemoryStore defines the contract for any memory backend.
// Any type with these three methods automatically satisfies this interface.
type MemoryStore interface {
    // SaveMessages persists conversation messages for a given session.
    // sessionID groups messages that belong to the same conversation.
    SaveMessages(ctx context.Context, sessionID string, msgs []Message) error

    // LoadMessages retrieves all messages for a session.
    // Returns an empty slice (not an error) if the session doesn't exist.
    LoadMessages(ctx context.Context, sessionID string) ([]Message, error)

    // Search finds past memories relevant to a query string.
    // limit caps how many results to return (e.g., top 5 most relevant).
    Search(ctx context.Context, query string, limit int) ([]Memory, error)
}

// Memory represents a single retrieved memory entry.
// When you search for "encryption policies", you get back Memory items
// ranked by relevance score.
type Memory struct {
    Content   string             // the actual text content of this memory
    Score     float64            // relevance score (0.0 to 1.0, higher = more relevant)
    Metadata  map[string]any    // flexible key-value data (vendor ID, agent name, etc.)
    Timestamp time.Time          // when this memory was originally created
}

// Message represents a single message in a conversation.
// Agents communicate by exchanging sequences of these messages.
type Message struct {
    Role    string             // who sent this: "system", "user", or "assistant"
    Content string             // the actual text of the message
}

In-Memory Implementation (For Testing)

The simplest memory backend stores everything in a Go map. This is perfect for unit tests and local development because it requires no external services — no Redis, no database, nothing to install. It is like using a whiteboard that gets erased when the program stops. The data lives only as long as the process runs.

We use a sync.RWMutex to make this safe for concurrent access. Think of a mutex like a single-person bathroom lock: only one goroutine can write at a time (exclusive lock), but multiple goroutines can read simultaneously (shared lock). Without this lock, two agents saving messages at the same time could corrupt the map.

import (
    "context"                            // for cancellation and deadline propagation
    "strings"                            // for basic text search in the Search method
    "sync"                               // provides the RWMutex for thread-safe access
)

// InMemoryStore keeps all data in a Go map, protected by a read-write mutex.
// Data is lost when the program exits. Use this for tests only.
type InMemoryStore struct {
    mu       sync.RWMutex               // protects concurrent access to the sessions map
    sessions map[string][]Message       // maps session IDs to their message history
}

// NewInMemoryStore creates a ready-to-use in-memory store.
// The map must be initialized with make() — a nil map panics on write.
func NewInMemoryStore() *InMemoryStore {
    return &InMemoryStore{
        sessions: make(map[string][]Message),  // initialize empty map
    }
}

// SaveMessages stores a copy of messages under the given session ID.
// We acquire an exclusive (write) lock because we're modifying the map.
func (s *InMemoryStore) SaveMessages(ctx context.Context, sessionID string, msgs []Message) error {
    s.mu.Lock()                          // acquire exclusive lock — no other reads or writes
    defer s.mu.Unlock()                  // release lock when function returns, even on error

    // Make a copy of the slice so the caller can't accidentally mutate our data.
    // This is defensive programming — the caller might modify their slice later.
    copied := make([]Message, len(msgs)) // allocate a new slice with the same length
    copy(copied, msgs)                   // copy all elements from msgs into our new slice
    s.sessions[sessionID] = copied       // store the copy in our map

    return nil                           // in-memory storage never fails, so error is always nil
}

// LoadMessages retrieves messages for a session.
// We use a read lock (RLock) because multiple goroutines can read simultaneously.
func (s *InMemoryStore) LoadMessages(ctx context.Context, sessionID string) ([]Message, error) {
    s.mu.RLock()                         // acquire shared lock — other reads are still allowed
    defer s.mu.RUnlock()                 // release when done

    msgs, exists := s.sessions[sessionID] // look up the session; exists is false if not found
    if !exists {
        return nil, nil                 // session not found — return empty, not an error
    }

    // Return a copy so the caller can't corrupt our internal data
    copied := make([]Message, len(msgs)) // allocate fresh slice
    copy(copied, msgs)                   // copy contents
    return copied, nil                  // return the copy
}

// Search performs a basic substring search across all stored messages.
// In production, you'd use vector embeddings for semantic search instead.
func (s *InMemoryStore) Search(ctx context.Context, query string, limit int) ([]Memory, error) {
    s.mu.RLock()                         // shared lock for reading
    defer s.mu.RUnlock()                 // release when done

    var results []Memory                  // accumulate matching memories here
    lowerQuery := strings.ToLower(query) // normalize to lowercase for case-insensitive matching

    // Iterate over every session and every message looking for matches
    for _, msgs := range s.sessions {     // loop through all sessions (ignore session ID)
        for _, msg := range msgs {        // loop through each message in this session
            // Check if the message content contains our search query
            if strings.Contains(strings.ToLower(msg.Content), lowerQuery) {
                results = append(results, Memory{
                    Content: msg.Content,    // the matching message text
                    Score:   1.0,            // simple match — always score 1.0
                })
            }

            // Stop searching once we have enough results
            if len(results) >= limit {
                return results, nil      // early return — no need to scan more
            }
        }
    }

    return results, nil                  // return whatever we found (may be fewer than limit)
}

// If any required method is missing, this line causes a compile error.

Redis Implementation (For Production)

Redis is an in-memory data store that runs as a separate server. Think of it as a giant shared dictionary that all your application servers can read and write to. Unlike our InMemoryStore, Redis survives application restarts, and multiple server instances can share the same Redis — making it ideal for production deployments where you run several copies of your application behind a load balancer.

Redis also supports automatic expiration (TTL — time to live). We set a TTL on session data so old conversations are automatically cleaned up. Without TTL, your Redis memory would grow forever as conversations accumulate.

)

// RedisMemory stores conversation data in Redis with automatic expiration.
// Redis runs as a separate server, so data persists across app restarts
// and is shared across multiple instances of your application.
type RedisMemory struct {

}

// NewRedisMemory creates a Redis-backed memory store.
// redisURL is the connection address (e.g., "localhost:6379").
// ttl sets how long conversation data lives before Redis deletes it.

        client: redis.NewClient(&redis.Options{  // configure the Redis connection
            Addr: redisURL,                       // server address like "localhost:6379"

    }

// SaveMessages converts the message slice to JSON and stores it in Redis.
// The key format "session:abc123" keeps session data organized.
func (m *RedisMemory) SaveMessages(ctx context.Context, sessionID string, msgs []Message) error {
    data, err := json.Marshal(msgs)      // convert the Go slice to a JSON byte array
    if err != nil {
        return fmt.Errorf("marshal messages: %w", err)  // wrap error with context
    }

    // Set stores the JSON data in Redis with the specified TTL.
    // After ttl expires, Redis automatically deletes this key.
    key := "session:" + sessionID        // prefix helps organize keys in Redis
    return m.client.Set(ctx, key, data, m.ttl).Err()  // .Err() extracts the error
}

// LoadMessages retrieves and decodes messages from Redis.
// If the session key doesn't exist (or has expired), returns empty — not an error.
func (m *RedisMemory) LoadMessages(ctx context.Context, sessionID string) ([]Message, error) {
    key := "session:" + sessionID        // construct the same key format we used to save
    data, err := m.client.Get(ctx, key).Bytes()  // fetch raw bytes from Redis

    // redis.Nil means the key doesn't exist — this is expected, not an error.
    // It's like looking for a file that doesn't exist: not a crash, just empty.
    if errors.Is(err, redis.Nil) {
        return nil, nil                 // no session found — return empty slice
    }
    if err != nil {
        return nil, fmt.Errorf("redis get: %w", err)  // real error (connection issue, etc.)
    }

    var msgs []Message                    // declare the slice that JSON will decode into
    err = json.Unmarshal(data, &msgs)    // parse JSON bytes back into Go structs
    return msgs, err                     // return messages (and any parse error)
}

Shared State Between Agents

When multiple agents work together on a single assessment, they need a shared workspace — like a team whiteboard where the risk agent writes "critical vulnerability found in encryption" and the compliance agent reads it when checking controls. This is working memory: it exists for the duration of one assessment and is discarded afterward.

The SharedState struct below is a thread-safe key-value store. The generic Get function uses Go's type parameters to retrieve values with compile-time type safety — no need for manual type casting.

import "sync"                            // provides the RWMutex for thread-safe access

// SharedState is a thread-safe key-value store for passing data between agents.
// Think of it as a whiteboard in a meeting room: any agent can write to it,
// any agent can read from it, and a mutex prevents two agents from
// writing over each other at the same time.
type SharedState struct {
    mu   sync.RWMutex                    // protects concurrent access to the data map
    data map[string]any                 // stores any type of value under a string key
}

// NewSharedState creates an empty shared state, ready for use.
func NewSharedState() *SharedState {
    return &SharedState{
        data: make(map[string]any),      // initialize the map so it's ready for writes
    }
}

// Set writes a value to the shared state under the given key.
// Uses an exclusive lock because writing must block all other access.
func (s *SharedState) Set(key string, value any) {
    s.mu.Lock()                          // exclusive lock — blocks all readers and writers
    defer s.mu.Unlock()                  // automatically release lock when function exits
    s.data[key] = value                  // store the value
}

// Get retrieves a value from the shared state with compile-time type safety.
// The [T any] makes this a generic function — you specify the expected type
// when calling it, and Go handles the type assertion for you.
// Returns the value and true if found and type matches; zero value and false otherwise.
func Get[T any](s *SharedState, key string) (T, bool) {
    s.mu.RLock()                         // shared lock — multiple readers can enter
    defer s.mu.RUnlock()                 // release when done

    val, ok := s.data[key]               // look up the key; ok is false if not found
    if !ok {
        var zero T                       // declare a zero-value of type T
        return zero, false              // key not found — return zero value
    }

    typed, ok := val.(T)                 // type assertion: try to convert 'any' to type T
    return typed, ok                     // if types match, ok is true; otherwise false
}

// Risk agent writes its findings:

// state.Set("critical_findings", []string{"weak encryption", "no MFA"})
//
// Compliance agent reads them:

When you pass a slice to a function in Go, you pass a header that points to the underlying array. If the caller later modifies their slice, your stored data changes too — like two people sharing the same notebook. By copying, each side gets its own notebook. This defensive pattern prevents subtle, hard-to-debug data corruption in concurrent systems.

Guardrails & Observability

What Are Guardrails?

Imagine driving on a mountain road at night. The road has no guardrails, no lane markings, and no speed signs. One wrong turn and you plunge off the cliff. Now imagine the same road with steel guardrails, reflective lane markers, and speed limit signs. The guardrails do not drive the car for you — they do not steer or accelerate. But they prevent catastrophic outcomes when something goes wrong.

Agent guardrails work the same way. An LLM is powerful but unpredictable. It might hallucinate a vendor risk score, leak personally identifiable information (PII) from its training data, generate output that violates compliance rules, or produce responses that are wildly off-topic. Guardrails are automated checks that run before input reaches the agent (input guardrails) and after the agent produces output (output guardrails). They do not replace the agent's intelligence — they prevent it from going off the cliff.

There are two types of guardrails:

  • Input guardrails — Check the incoming request before the agent sees it. Block requests that contain PII, exceed length limits, or attempt prompt injection attacks.
  • Output guardrails — Check the agent's response before it reaches the user. Ensure the output matches expected formats, does not contain harmful content, and includes required fields.

The Guardrail Interface

We define a Guardrail interface with a single Check method. Each guardrail examines the messages and returns a result: passed (safe to proceed), or failed (with a reason and recommended action). The action can be "block" (stop immediately), "warn" (log a warning but continue), or "log" (record for audit but take no action).

import (
    "context"                            // for cancellation and timeouts
    "regexp"                             // for regular expression pattern matching
)

// Guardrail defines the contract for any safety check.
// Each guardrail inspects messages and decides: safe or not safe?
type Guardrail interface {
    // Check examines messages and returns whether they pass the safety check.
    Check(ctx context.Context, messages []Message) (GuardrailResult, error)
}

// GuardrailResult tells us whether the check passed and what to do if it failed.
type GuardrailResult struct {
    Passed bool                          // true = safe to proceed, false = problem found
    Reason string                        // human-readable explanation of why it failed
    Action string                        // what to do: "block", "warn", or "log"
}

PII Detection Guardrail

One critical guardrail for any system handling vendor data: detecting personally identifiable information. Social Security numbers, credit card numbers, email addresses — these should never be sent to an LLM or included in agent output. This guardrail uses regular expressions to scan every message for common PII patterns.

// PIIGuardrail scans messages for personally identifiable information.
// It uses regex patterns to detect things like SSNs, credit card numbers, etc.
type PIIGuardrail struct {
    patterns []*regexp.Regexp            // compiled regex patterns for PII detection
}

// NewPIIGuardrail creates a guardrail pre-loaded with common PII patterns.
// Each pattern targets a different type of sensitive data.
func NewPIIGuardrail() *PIIGuardrail {
    return &PIIGuardrail{
        patterns: []*regexp.Regexp{
            // SSN: three digits, dash, two digits, dash, four digits
            regexp.MustCompile(`\b\d{3}-\d{2}-\d{4}\b`),

            // Credit card: 13-19 digits (covers Visa, Mastercard, Amex, etc.)
            regexp.MustCompile(`\b\d{13,19}\b`),

            // Email address: basic pattern for catching most emails
            regexp.MustCompile(`\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b`),
        },
    }
}

// Check scans every message for PII patterns.
// If any pattern matches anywhere in any message, the check fails.
func (g *PIIGuardrail) Check(ctx context.Context, msgs []Message) (GuardrailResult, error) {
    for _, msg := range msgs {           // iterate through each message in the conversation
        for _, p := range g.patterns {    // check each PII pattern against this message
            if p.MatchString(msg.Content) { // does this message contain PII?
                return GuardrailResult{
                    Passed: false,        // check failed — PII was found
                    Reason: "PII detected in message content",
                    Action: "block",       // stop processing immediately
                }, nil                   // no error — the guardrail worked correctly
            }
        }
    }

    // No PII found in any message — safe to proceed
    return GuardrailResult{Passed: true}, nil
}

Applying Guardrails as Middleware

Guardrails wrap around agents like layers of an onion. The WithGuardrails function takes an agent and returns a new agent that runs input checks before calling the original agent, then runs output checks on the result. This is the middleware pattern — adding behavior without modifying the agent itself. The agent does not even know it is being guarded.

// guardedAgent wraps an existing agent with input and output guardrails.
// It satisfies the Agent interface itself, so from the outside,
// it looks and behaves like any other agent.
type guardedAgent struct {
    next             Agent                // the real agent we're protecting
    inputGuardrails  []Guardrail          // checks to run BEFORE the agent
    outputGuardrails []Guardrail          // checks to run AFTER the agent
}

// Run first checks all input guardrails, then runs the real agent,
// then checks all output guardrails before returning the response.
func (g *guardedAgent) Run(ctx context.Context, msgs []Message) (Response, error) {
    // Phase 1: Run all input guardrails on the incoming messages
    for _, guard := range g.inputGuardrails {
        result, err := guard.Check(ctx, msgs)  // run this guardrail's check
        if err != nil {
            return Response{}, fmt.Errorf("input guardrail error: %w", err)
        }
        if !result.Passed && result.Action == "block" {
            // Guardrail says "block" — do NOT call the agent at all
            return Response{}, fmt.Errorf("input blocked: %s", result.Reason)
        }
    }

    // Phase 2: Input is safe — run the actual agent
    resp, err := g.next.Run(ctx, msgs)   // call the wrapped agent
    if err != nil {
        return Response{}, err            // agent failed — propagate the error
    }

    // Phase 3: Run all output guardrails on the agent's response
    for _, guard := range g.outputGuardrails {
        result, err := guard.Check(ctx, resp.Messages)
        if err != nil {
            return Response{}, fmt.Errorf("output guardrail error: %w", err)
        }
        if !result.Passed && result.Action == "block" {
            // Agent produced unsafe output — block it
            return Response{}, fmt.Errorf("output blocked: %s", result.Reason)
        }
    }

    return resp, nil                     // all checks passed — return the response
}

// Name delegates to the wrapped agent — the guardrail is invisible.
func (g *guardedAgent) Name() string {
    return g.next.Name()                 // use the real agent's name
}

// WithGuardrails creates a middleware function that wraps any agent.
// Usage: guardedAgent := WithGuardrails(inputChecks, outputChecks)(myAgent)
func WithGuardrails(input, output []Guardrail) AgentMiddleware {
    return func(next Agent) Agent {      // returns a function that wraps any agent
        return &guardedAgent{             // create the guarded wrapper
            next:             next,       // the original agent to protect
            inputGuardrails:  input,      // checks to run on input
            outputGuardrails: output,     // checks to run on output
        }
    }
}

Rate Limiting

LLM APIs charge per token and enforce rate limits (e.g., OpenAI allows a certain number of requests per minute). Without rate limiting, a batch of 500 vendor assessments could fire 500 simultaneous API calls, hit the limit, and get all requests rejected with HTTP 429 errors. A rate limiter acts like a turnstile at a subway station: it lets people through at a controlled pace, preventing overcrowding.

import (
    "context"                            // carries cancellation into the Wait call
    "fmt"                                // for error message formatting

    "golang.org/x/time/rate"              // Go's official rate limiter package
)

// RateLimitedLLM wraps any LLM client and throttles how fast we call it.
// Like a turnstile: lets requests through at a steady pace.
type RateLimitedLLM struct {
    client  LLMClient                    // the actual LLM client to call
    limiter *rate.Limiter                // controls the request rate
}

// NewRateLimitedLLM creates a rate-limited wrapper around any LLM client.
// requestsPerSecond sets the maximum steady-state rate (e.g., 10 = 10 req/sec).
func NewRateLimitedLLM(client LLMClient, requestsPerSecond int) *RateLimitedLLM {
    return &RateLimitedLLM{
        client: client,                  // store reference to the real client
        limiter: rate.NewLimiter(         // create the rate limiter
            rate.Limit(requestsPerSecond),  // steady-state rate: N requests per second
            requestsPerSecond,            // burst size: allow brief bursts up to this amount
        ),
    }
}

// Complete waits for rate limit clearance, then forwards to the real client.
// If the context is cancelled while waiting, it returns immediately.
func (r *RateLimitedLLM) Complete(ctx context.Context, req CompletionRequest) (CompletionResponse, error) {
    // Wait blocks until the limiter allows one more request.
    // If we're under the limit, it returns immediately.
    // If we've exceeded the limit, it pauses until a slot opens up.
    if err := r.limiter.Wait(ctx); err != nil {
        return CompletionResponse{}, fmt.Errorf("rate limit wait: %w", err)
    }

    return r.client.Complete(ctx, req)   // now call the real LLM
}

Cost Tracking

LLM API calls cost money. A single GPT-4 Turbo call with a few thousand tokens costs about $0.04. Multiply by 50 vendors, each needing 5 agent interactions, and a single batch assessment costs $10+. Without tracking, costs spiral invisibly. Cost tracking wraps the LLM client and records token usage and dollar cost for every call.

We use atomic.Int64 for the counters because multiple goroutines may be tracking costs simultaneously. Atomic operations are like a self-service checkout counter with an electronic tally — multiple customers can add items to the total without interfering with each other, and no explicit locking is needed.

import (
    "context"                            // for request lifecycle management
    "log/slog"                           // for structured logging
    "sync/atomic"                        // for thread-safe integer operations
    "time"                               // for measuring call duration
)

// MeteredLLM wraps an LLM client and tracks token usage and cost.
// Think of it as an electricity meter on your house — it doesn't
// change how electricity works, it just records how much you use.
type MeteredLLM struct {
    client       LLMClient               // the real LLM client to meter
    totalTokens  atomic.Int64            // running count of all tokens used
    totalCostUSD atomic.Int64            // total cost in microdollars (1 USD = 1,000,000)
}

// Complete calls the real LLM and records usage metrics.
func (m *MeteredLLM) Complete(ctx context.Context, req CompletionRequest) (CompletionResponse, error) {
    start := time.Now()                  // record start time to measure duration
    resp, err := m.client.Complete(ctx, req)  // make the actual LLM call
    duration := time.Since(start)        // calculate how long the call took

    if err == nil {                      // only track metrics on successful calls
        // Add token count to our running total (thread-safe)
        m.totalTokens.Add(int64(resp.Usage.TotalTokens))

        // Calculate cost in microdollars and add to running total
        costMicro := calculateCostMicrodollars(req.Model, resp.Usage)
        m.totalCostUSD.Add(costMicro)

        // Log every call with structured key-value pairs
        slog.Info("llm call completed",
            "model", req.Model,                        // which model was used
            "prompt_tokens", resp.Usage.PromptTokens,    // tokens in the input
            "completion_tokens", resp.Usage.CompletionTokens,  // tokens in the output
            "cost_usd", float64(costMicro)/1_000_000,  // convert microdollars to dollars
            "duration", duration,                      // how long the call took
        )
    }

    return resp, err                     // pass through the response unchanged
}

// calculateCostMicrodollars converts token usage to cost in microdollars.
// We use microdollars (1 USD = 1,000,000 micro) to avoid floating-point
// precision issues when summing many tiny amounts.
func calculateCostMicrodollars(model string, usage Usage) int64 {
    switch model {                       // pricing depends on the model
    case "gpt-4-turbo":                  // GPT-4 Turbo: $10/M input, $30/M output
        return int64(usage.PromptTokens)*10 + int64(usage.CompletionTokens)*30
    case "gpt-4o-mini":                  // GPT-4o Mini: much cheaper
        return int64(usage.PromptTokens)*0 + int64(usage.CompletionTokens)*1
    default:                             // unknown model — log zero cost
        return 0
    }
}

Structured Logging with slog

Logging is how you understand what your system is doing at runtime. Traditional logging writes human-readable strings: "Processed vendor ACME, score=85". But when you have thousands of log lines, searching for all assessments with a score above 80 means writing fragile text-parsing scripts.

Structured logging solves this by outputting key-value pairs in a machine-readable format (usually JSON). Each log entry has named fields that you can filter, aggregate, and alert on in tools like Grafana, Datadog, or Splunk. Go's standard library includes log/slog (since Go 1.21), which provides structured logging out of the box.

import (
    "log/slog"                           // Go's built-in structured logging (since 1.21)
    "os"                                 // for os.Stdout (where logs go)
    "time"                               // for time.Since() in duration measurement
)

func setupLogger() *slog.Logger {
    // Create a JSON handler that writes to stdout.
    // JSON format makes logs easy to parse by log aggregation tools.
    handler := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
        Level: slog.LevelInfo,           // minimum log level (Debug, Info, Warn, Error)
    })

    // Create a logger instance with the JSON handler
    logger := slog.New(handler)          // this logger outputs structured JSON
    return logger                        // return for use throughout the application
}

// Example: logging an agent completion with structured key-value pairs
func logAgentCompletion(agentName string, vendorID string, start time.Time, tokens int, score int) {
    slog.Info("agent completed",          // the log message — short, descriptive
        "agent", agentName,               // which agent ran — easy to filter by agent
        "vendor_id", vendorID,            // which vendor — easy to trace a specific assessment
        "duration", time.Since(start),    // how long it took — spot slow assessments
        "tokens", tokens,                 // token usage — track costs
        "risk_score", score,              // the result — audit trail for decisions
    )
    // Output: {"level":"INFO","msg":"agent completed","agent":"risk-assessor",
    //          "vendor_id":"v-123","duration":"2.4s","tokens":1850,"risk_score":85}
}

// Logging levels: use the right level for the right situation
func loggingExamples() {
    slog.Debug("parsing vendor document", "file", "soc2.pdf")     // detailed trace info
    slog.Info("assessment started", "vendor", "Acme Corp")       // normal operations
    slog.Warn("rate limit approaching", "usage", "85%")         // something needs attention
    slog.Error("LLM call failed", "error", "timeout after 30s") // something broke
}

OpenTelemetry Tracing (Brief Intro)

When a single vendor assessment involves five agents, three LLM calls, and ten tool executions, structured logs alone are not enough. You need to see the entire execution as a tree: the orchestrator started, then the risk agent ran (which made two tool calls), then the compliance agent ran in parallel (which made one tool call), and finally the report agent assembled everything. This tree view is called a trace.

OpenTelemetry is the industry standard for distributed tracing. It is vendor-neutral — you write instrumentation code once and send traces to any backend (Jaeger, Grafana Tempo, Datadog, etc.). Each unit of work is a span with a name, attributes, and a parent span. Spans nest to form the execution tree.

Think of tracing like a GPS tracker on a delivery truck. Logs tell you "the truck arrived at warehouse B." Tracing tells you the entire route: "the truck started at warehouse A, drove to warehouse B (took 20 minutes), picked up a package, drove to the customer (took 15 minutes), delivered at 3:42 PM." You can see the full journey, identify bottlenecks, and debug problems.

import (
    "context"                            // spans are propagated through context

    "go.opentelemetry.io/otel"           // OpenTelemetry core API
    "go.opentelemetry.io/otel/attribute" // for attaching key-value metadata to spans
    "go.opentelemetry.io/otel/codes"     // for marking spans as OK or Error
    "go.opentelemetry.io/otel/trace"     // for creating and managing spans
)

// Create a tracer for our application. The string identifies our service
// in the tracing backend so you can filter traces by service name.
var tracer = otel.Tracer("my-agents")

// Run wraps the agent's execution in a tracing span.
// The span records the agent's name, model, input size, output size,
// and any errors — all visible in your tracing dashboard.
func (a *LLMAgent) Run(ctx context.Context, msgs []Message) (Response, error) {
    // Start a new span named "agent.run". This becomes a node in the trace tree.
    // The new ctx carries the span, so any child spans will nest inside this one.
    ctx, span := tracer.Start(ctx, "agent.run",
        trace.WithAttributes(            // attach metadata to the span
            attribute.String("agent.name", a.name),   // which agent is running
            attribute.String("agent.model", a.model), // which LLM model it uses
            attribute.Int("input.messages", len(msgs)), // how many input messages
        ),
    )
    defer span.End()                     // end the span when this function returns

    // Run the actual agent logic
    resp, err := a.runInternal(ctx, msgs) // the real work happens here

    if err != nil {
        span.RecordError(err)            // attach the error to the span for visibility
        span.SetStatus(codes.Error, err.Error())  // mark the span as failed
    } else {
        // Record output size on success
        span.SetAttributes(attribute.Int("output.messages", len(resp.Messages)))
    }

    return resp, err                     // return the response to the caller
}
Observability is non-negotiable for agent systems
Trace every LLM call, tool execution, and agent handoff. Without tracing, debugging a multi-agent pipeline is impossible — you will see an error but have no idea which of five agents caused it or what happened in the 15 steps before it. Use OpenTelemetry — it integrates with every major observability platform and costs nothing to add to your code.

Go Quick Reference

A concise cheat sheet of the most common Go operations you will reach for daily. Each example is self-contained with a brief explanation.

Print to Console

fmt.Println prints values followed by a newline. fmt.Printf uses format verbs (%s for strings, %d for integers, %v for any value).

import "fmt"                            // formatting and printing package

fmt.Println("Hello, Go!")              // prints text followed by a newline
fmt.Printf("Name: %s, Age: %d\n", name, age)  // formatted output with verbs

Format a String

fmt.Sprintf returns a formatted string instead of printing it. Use it to build strings with dynamic values.

msg := fmt.Sprintf("Vendor %s scored %d", name, score)  // returns formatted string

Read a File

os.ReadFile reads an entire file into a byte slice. Always check the error before using the data.

import "os"                              // operating system functions including file I/O

data, err := os.ReadFile("report.txt")  // read entire file into memory as []byte
if err != nil {                        // always check for errors (file might not exist)
    log.Fatal(err)                       // print error and exit if file can't be read
}
text := string(data)                   // convert bytes to a string

HTTP GET Request

http.Get makes a GET request and returns a response. Always close the response body when done.

import (
    "io"                                 // for reading the response body
    "net/http"                           // HTTP client and server
)

resp, err := http.Get("https://api.example.com/vendors")  // make GET request
if err != nil {
    log.Fatal(err)                       // network error
}
defer resp.Body.Close()                 // always close the body when done

body, err := io.ReadAll(resp.Body)       // read the entire response body

Parse JSON

json.Unmarshal decodes JSON bytes into a Go struct. Fields are mapped by struct tags (`json:"name"`).

import "encoding/json"                  // JSON encoding and decoding

type Vendor struct {
    Name  string `json:"name"`          // maps to "name" in JSON
    Score int    `json:"score"`         // maps to "score" in JSON
}

var vendor Vendor                        // declare a variable to decode into
err := json.Unmarshal(data, &vendor)     // decode JSON bytes into the struct

// Encoding: convert a struct to JSON bytes
output, err := json.Marshal(vendor)      // encode struct to JSON bytes

Sleep / Pause Execution

time.Sleep pauses the current goroutine. Use time.Duration constants for clarity.

import "time"                            // time operations and durations

time.Sleep(2 * time.Second)              // pause for 2 seconds
time.Sleep(500 * time.Millisecond)       // pause for 500 milliseconds

Get Current Time

time.Now() returns the current time. time.Since() calculates elapsed duration — useful for timing operations.

start := time.Now()                      // capture the current time
// ... do some work ...
elapsed := time.Since(start)             // how long since start (e.g., "2.34s")
fmt.Println("Took:", elapsed)           // prints human-readable duration

Start a Goroutine

The go keyword launches a function in a new goroutine (lightweight thread). It runs concurrently with the rest of your code.

go processVendor(vendor)                 // launch processVendor in a new goroutine

// With an anonymous function (inline goroutine):
go func() {
    result := expensiveCalculation()     // runs concurrently
    fmt.Println(result)                  // prints when calculation finishes
}()                                      // () immediately invokes the anonymous function

Lock a Mutex

A mutex prevents multiple goroutines from modifying shared data at the same time. Always pair Lock with defer Unlock.

import "sync"                            // synchronization primitives

var mu sync.Mutex                        // declare a mutex (zero value is unlocked)

mu.Lock()                                // acquire the lock (blocks if another goroutine holds it)
defer mu.Unlock()                        // release the lock when the function returns
counter++                                // safe to modify shared data while holding the lock

Sort a Slice

slices.Sort sorts a slice in place. For custom sorting, use slices.SortFunc with a comparison function.

import "slices"                          // slice utilities (since Go 1.21)

numbers := []int{5, 2, 8, 1}           // unsorted slice
slices.Sort(numbers)                     // sorts in place: [1, 2, 5, 8]

// Custom sort: sort vendors by risk score (highest first)
slices.SortFunc(vendors, func(a, b Vendor) int {
    return b.RiskScore - a.RiskScore     // negative = a first, positive = b first
})

Structured Logging

slog provides structured key-value logging. Each log entry has a level, message, and any number of key-value pairs.

import "log/slog"                        // structured logging (since Go 1.21)

slog.Info("assessment complete",         // log message
    "vendor", "Acme Corp",              // key-value pair: vendor name
    "score", 85,                         // key-value pair: risk score
    "duration", elapsed,                // key-value pair: how long it took
)

Error Handling Pattern

Go returns errors as values. Check err != nil after every operation that can fail. Wrap errors with fmt.Errorf and %w to add context.

result, err := doSomething()             // call a function that may fail
if err != nil {                        // check if it failed
    return fmt.Errorf("doing something: %w", err)  // wrap with context and return
}

Declare and Use an Interface

Interfaces define behavior, not identity. Any type with the right methods automatically satisfies the interface — no implements keyword needed.

// Define the interface: what methods must a type have?
type Agent interface {
    Run(ctx context.Context, msgs []Message) (Response, error)  // execute the agent
    Name() string                        // return the agent's name
}

// Any struct with Run() and Name() methods satisfies Agent automatically.
// No "implements" keyword needed. Just have the right methods.

Create a Struct and Constructor

Structs group related data. Convention: write a NewTypeName function that returns a pointer to a new instance.

type Vendor struct {                     // define a struct type
    ID   string                          // exported field (uppercase = public)
    Name string                          // exported field
    tier string                          // unexported field (lowercase = private to package)
}

func NewVendor(id, name string) *Vendor {  // constructor function returns a pointer
    return &Vendor{ID: id, Name: name}   // create and return a new Vendor
}

Channels for Communication

Channels are typed conduits for passing data between goroutines. They are how concurrent code communicates safely.

ch := make(chan string)                 // create an unbuffered channel of strings

go func() {
    ch <- "hello"                       // send a value into the channel
}()

msg := <-ch                             // receive a value from the channel (blocks until sent)

Context for Cancellation and Timeouts

Context carries deadlines and cancellation signals through your call chain. Pass it as the first parameter to every function that does I/O.

import "context"                         // request-scoped values, deadlines, and cancellation

// Create a context that automatically cancels after 5 seconds
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()                           // always call cancel to release resources

result, err := agent.Run(ctx, msgs)      // the agent will stop if 5 seconds elapse

Defer for Cleanup

defer schedules a function call to run when the enclosing function returns. Use it for cleanup: closing files, releasing locks, ending spans.

file, err := os.Open("data.csv")        // open a file
if err != nil {
    return err                           // handle error
}
defer file.Close()                       // guaranteed to close when function returns
